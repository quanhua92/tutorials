SECTION A: multi-codebook tokenization -- 12 Hz, 4 parallel codebooks
======================================================================
Every neural TTS must first turn audio into a compact target the model
can predict. Qwen3-TTS uses the Qwen3-TTS-Tokenizer-12Hz: it frames
audio at 12 Hz and quantises each frame into 4
parallel codebook ids. So one second of audio becomes a 12x4 grid of
integers -- 48 tokens. That grid is what the LM
autoregressively predicts. Because it is short and discrete, an LLM-style
transformer can generate it directly.

  tokenizer config:
    frame rate           = 12 Hz  (12 frames / second)
    codebooks per frame  = 4
    vocab per codebook   = 1024 ids (10 bits/id)
    samples / frame      = 1333  (16kHz / 12Hz)
    audio per frame      = 83.33 ms

  encode 1.0 s of audio -> 12 frames x 4 codebooks:
    frame | cb0 | cb1 | cb2 | cb3
    -----------------------------------------
    0     |   327 |   254 |   181 |   108
    1     |    47 |   998 |   925 |   852
    2     |  1022 |   949 |   876 |   803
    3     |   806 |   733 |   660 |   587
    4     |   993 |   920 |   847 |   774
    5     |  1015 |   942 |   869 |   796
    ... (6 more frames)

  token rate   = 12 frames x 4 codebooks = 48 tokens/s
  bitrate      = 48 x 10 bits     = 480 bps
  raw PCM      = 16000 x 16         = 256,000 bps  (16kHz mono int16)
  compression  = 256,000 / 480 = 533x
[check] token rate = 12 * 4 = 48 tokens/s: OK
[check] bitrate = 48 * 10 = 480 bps: OK
[check] compression vs PCM > 500x: OK

  vs traditional neural codecs (encodec/soundstream at 50-75 Hz):
    codec                       Hz      frames/s  relative seq len
    Qwen3-TTS-Tokenizer         12      12        1.0x
    EnCodec 24kHz               75      75        6.2x
    SoundStream                 50      50        4.2x

  -> A 75 Hz codec makes the LM predict ~6x more frames for the
     same audio. Fewer frames at 12 Hz = shorter autoregressive run =
     faster generation and a sooner first packet.

  --> The tokenizer's job is to make the LM's job EASY. 48 tokens per
      second is a short, discrete target -- short enough for a transformer
      to predict frame-by-frame and stream out as it goes.

SECTION B: non-DiT architecture -- multi-codebook LM vs cascade vs DiT
======================================================================
Qwen3-TTS deliberately AVOIDS both legacy designs:
  - CASCADE TTS (Tacotron->HiFi-GAN): text -> acoustic features -> vocoder.
    Two separately-trained stages; stage-2 vocoder cannot fix stage-1's
    errors, so they COMPOUND. End-to-end quality is capped by the worst
    stage.
  - DiT TTS (NaturalSpeech-DiT, E2-TTS): a Diffusion Transformer denoises
    the full audio latent over N iterative steps. Nothing is usable until
    the LAST step, because each step refines the whole -> BLOCKS streaming.
Qwen3-TTS is a single multi-codebook LM: it predicts the token grid
autoregressively, one frame per step, and each frame decodes immediately.

  generating 1.0s of speech (12.0 frames at 12 Hz):

    architecture              model passes    usable before done?   streams?
    ----------------------------------------------------------------------
    cascade (acoustic+vocoder)2               no (stages)           no
    DiT (25 denoise steps)    25              no (whole audio)      no
    Qwen3 multi-codebook LM   12 (1/frame)      yes (per frame)       YES
[check] cascade = 2 stages: OK
[check] DiT needs N>1 denoising passes (25): OK
[check] Qwen3 LM = 1 pass per frame (12 for 1s): OK

  why DiT cannot stream but the LM can:
    DiT : step k refines the WHOLE latent. Audio at frame 0 is NOT final
          until step 25 finishes. Must wait for all 25 steps.
    LM  : frame 0's tokens are FINAL the instant they are emitted. Decode
          frame 0 -> 83.33 ms audio -> ship it. Frame 1 next.

  -> No iterative denoising = nothing forces you to wait for the whole
      utterance. That is the architectural reason the 97 ms first packet

  cascade error compounding (modelled):
    stage-1 (acoustic) accuracy = 0.97
    stage-2 (vocoder) accuracy = 0.95
    cascade end-to-end         = 0.97 x 0.95 = 0.9215
    Qwen3 single end-to-end    = 0.99  (one model, no handoff)
[check] cascade compounds: 0.97*0.95 = 0.9215: OK
[check] single end-to-end > cascade compound: OK

  --> One model, trained end to end, predicting a short token grid. No
      cascade to compound errors, no denoising loop to block streaming.

SECTION C: dual-track streaming -- first audio packet at 97 ms
======================================================================
The SAME model serves two tracks:
  - NON-STREAMING (track 1): feed the FULL text, attend over all of it,
    generate every frame, THEN play. Highest quality (full context), but
    latency scales with total audio length.
  - STREAMING (track 2): process text incrementally; as soon as ONE frame
    is generated, decode and ship it. End-to-end first packet = 97 ms.
    Latency is FIXED regardless of how long the utterance is.

  streaming first-packet latency breakdown (sums to the gold value):
    text chunk + tokenize              12 ms  ######
    LM prefill (prompt + 1st char)     35 ms  #################
    generate 1st token frame (4 ids)   20 ms  ##########
    tokenizer decode frame -> audio    30 ms  ###############
    TOTAL                              97 ms

SECTION D: voice control -- clone, design, control, timbre reuse
======================================================================
Four voice capabilities, all from the one model:
  1. CLONE   : 3 s of reference audio -> speaker embedding ->
               synthesise ANY text in that voice.
  2. DESIGN  : a natural-language description -> a brand-new voice.
  3. CONTROL : an instruction overlaid on an existing voice for one line.
  4. REUSE   : a designed/cloned voice is persisted and reused across
               conversations and for multi-character dialogues.
