Distributed Data Parallel (DDP) — interactive DDP.md ddp.py

For newcomers: DDP copies the whole model onto every GPU; each GPU eats a different mini-batch, computes a local gradient, then AllReduces (averages) them so every GPU ends up with the identical gradient → identical optimizer step → replicas never drift. Drag the phase slider below to walk one optimizer step. Every number is recomputed live in JS from the identical formula in ddp.py (a faithful single-process K=2 simulation).  |  → NCCL collectives   → Tensor Parallel

① One DDP step: 2 replicas → local grads → AllReduce → identical update

How to use: drag the phase slider through the 4 stages. Phase 0 = identical init (broadcast). Phase 1 = each rank runs fwd+bwd on its own batch → different local grads (note g0 ≠ g1). Phase 2 = AllReduce averages them → both ranks now hold the identical grad. Phase 3 = both take the same SGD step → params stay bit-identical (no drift).
AllReduce: both ranks now hold identical averaged grad
rank 0 (GPU 0, batch X0) rank 1 (GPU 1, batch X1) AllReduce-averaged (identical on both) init weights W
What to notice: at phase 1 the two gradient bar-groups clearly differ (left=blue, right=orange). At phase 2 they collapse to the same green bars on both sides — that is AllReduce making the replicas agree. At phase 3 the weight bars update identically. This is why DDP replicas can never drift. (DDP.md §A–B)

② Gradient accumulation: many micro-batches, ONE AllReduce at the end

How to use: drag accum steps. The timeline shows micro-batches per rank; red = sync OFF (local accumulate only), green = the LAST micro-step where require_backward_grad_sync=True and the single AllReduce fires. Effective batch = bs × accum × world_size.
world_size = 2, bs = 2 → effective batch = 8
Why: without this trick you'd AllReduce accum times per optimizer step instead of once. nanoGPT sets model.require_backward_grad_sync = (micro_step == accum-1) to skip AllReduce on all but the last micro-step. (DDP.md §C)

③ Cosine LR + linear warmup (nanoGPT get_lr, Goyal 2017)

How to use: drag step. The curve warms up linearly 0→peak, then cosine-decays peak→min_lr, then holds. Bigger batch ⇒ scale peak LR up (linear scaling rule) and warm up longer to avoid early instability.
peak=1.0, min_lr=0.1, warmup=10, decay_iters=50 (readable scale)

④ The 20-bytes/param memory bill (mixed-precision Adam, per GPU)

How to read: each bar is one component DDP replicates on every GPU. The two fp16 bars (params+grads) are what 🔗 QUANTIZATION shrinks; the four fp32 bars (master+grad+momentum+variance) are the 16N redundancy 🔗 ZeRO will shard away.
Total = 20 bytes/param. An 8B model costs ~149 GB per GPU in DDP. ZeRO-3 partitions all of it down to 16N/K; QUANTIZATION attacks the weight bytes. Orthogonal axes.
