Heap vs Clustered / IOT table — interactive HEAP_VS_CLUSTERED.md heap_vs_clustered.py

The one question: where does the row physically live? On the left, a heap (PostgreSQL) — an unordered file; indexes store a physical TID, so every lookup ends with one extra heap read. On the right, a clustered / index-organized table (MySQL InnoDB) — rows live inside the PK B-tree leaf; secondary indexes store the PK, costing a second tree walk. Pick an operation and watch the access path light up on both sides; the I/O counters recompute live with the same model as heap_vs_clustered.py.

No separate heap — the PK leaf is the table. Secondary entries store the PK (a logical handle, stable across splits), so a secondary lookup re-walks the PK tree to fetch the row.
⚠ updating the PK physically MOVES the row here (maybe a split) and rewrites every secondary index — keep InnoDB PKs immutable.

② I/O comparison — same workload, different spending

With index height h = 2: a heap always adds +1 heap read to any index lookup; a clustered secondary adds a +h PK-tree walk. For one PK + one secondary lookup the totals are equal (6 = 6) — but clustered spends less on PK lookups and range scans, the heap spends less on secondary lookups.
