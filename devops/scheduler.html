📖 Read the full guide — diagrams, decision tables, common pitfalls, and a cheat sheet. This page is the interactive companion.
The scheduler assigns each Pending pod to ONE node in two phases: Filter (drop nodes that can't host it) then Score (rank the survivors). Toggle the scoring policy and watch the winner flip: LeastAllocated spreads (emptiest node), MostAllocated bin-packs (fullest node). All numbers recompute live from the same model as scheduler.py.

① Filter phase — can the pod run here? (hard constraints)

Pod 'web' requests 2 CPU + 4 GB, no tolerations. Each node must pass ALL of: enough free CPU, enough free memory, and all NoSchedule taints tolerated.

② Score phase — rank feasible nodes (the winner flips with policy)

What this proves: the JS recomputes LeastAllocated / MostAllocated / Balanced scores from the same feasible set as scheduler.py Section B. The green check: OK confirms winners match the .py gold (Least→node-3, Most→node-1).

③ podAntiAffinity — 3 replicas spread across 3 nodes (HA)

What this shows: schedule 3 replicas one at a time. antiAffinity steers each later replica AWAY from nodes already hosting a 'web' pod, so a node fault kills at most one replica.
