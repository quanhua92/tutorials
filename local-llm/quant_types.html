① Dequant playground — nibbles + scale → floats

What this shows: you provide the on-disk representation (8 nibble values 0–15 + a scale d), the playground runs the selected type's dequant formula and prints the recovered floats. Same formulas as quant_types.py Sections A/B/D. Switch type to see how the same nibbles dequantize differently.
Q4_0: w = d·(q−8) symmetric. Q4_K_M: w = m + d·q asymmetric (the sub-block affine inside a super-block). IQ3_S: same affine but the high-importance channel gets a finer effective step (importance matrix benefit, conceptually).

② Side-by-side — same nibbles, three dequants

What this shows: the current nibbles dequantized under all three families at once, plus bits/weight, relative quality (Q8_0 = 100% reference), and decode-speed tier. Decode is bandwidth-bound so lower bpw ≈ faster.
Relative quality is the inverse of the quantization-noise ceiling: noise ≈ range / (2·levels). More levels (higher bpw) → lower noise → higher quality. Q8_0 (256 levels) anchors 100%.

③ Verified bits/weight — struct sizes from ggml-common.h

What this shows: every byte count below is a static_assert(sizeof(block_*)) in ggml/src/ggml-common.h, recomputed identically in JS. Legacy = 32-weight blocks; K/I = 256-weight super-blocks.
Pure-format bpw comes from the C struct. The _M/_S/_L model-level bpw (e.g. Q4_K_M ≈ 4.84) is a weighted average across mixed layer types — see QUANT_TYPES.md §3.
