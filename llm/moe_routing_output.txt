========================================================================
  SECTION D: auxiliary losses -- load-balance + router z-loss
========================================================================
Router collapse: without a push, the receptionist sends everything to
1-2 favorite experts and the rest never train. Two losses fix this.

For our N=4 tokens, per-expert statistics:
| expert i | f_i (frac of tokens routed) | P_i (avg gate prob) | f_i*P_i |
|----------|------------------------------|---------------------|---------|
| 0        | 0.7500                       | 0.3461              | +0.2596  |
| 1        | 0.5000                       | 0.2566              | +0.1283  |
| 2        | 0.2500                       | 0.1340              | +0.0335  |
| 3        | 0.5000                       | 0.2632              | +0.1316  |

L_bal = E * sum_i f_i * P_i = 4 * 0.553034 = 2.212137
  (minimum when balanced is k=2; larger => more imbalanced)
  sum(f) = 2.0000  (== k, since each token picks k)
  sum(P) = 1.0000  (== 1, gate weights renormalize)

L_z = (1/N) sum_j (log sum_i e^H_i)^2 = 2.164589
  (penalizes LARGE logits -> keeps router numerically stable)

========================================================================
  SECTION E: token dropping + expert capacity
========================================================================
When experts live on separate GPUs (Expert Parallelism, §G), each GPU
needs a fixed-size buffer. Expert Capacity C caps how many tokens one
expert may process per step:

  C = capacity_factor * N*k/E = 1.0 * 4*2/4 = 2.0
  (each expert may process at most 2 tokens; overflow is DROPPED)

Routing results vs capacity (the dropping decision):
| expert i | tokens received | capacity C | action                  |
|----------|-----------------|------------|-------------------------|
| 0        | 3               | 2          | DROP 1 (only 2 run, rest -> residual) |
| 1        | 2               | 2          | exact fit               |
| 2        | 1               | 2          | PAD 1 zero(s)           |
| 3        | 2               | 2          | exact fit               |

[check] total tokens dropped this step = 1

Dropping degrades quality at inference (a dropped token skips its MLP
update entirely -> just the residual). Production servers (ZeroServe,
vLLM) use no-drop / dynamic-capacity routing instead. DeepSeek-V3's
aux-loss-free balancing (§F) keeps load even so C is rarely hit.

========================================================================
  SECTION F: DeepSeek-V3 -- shared + fine-grained + aux-loss-free bias
========================================================================
DeepSeek-V3 (arXiv:2412.19437) adds three things to classic top-k MoE:

(1) SHARED expert -- always on, captures common features so routed
    experts can specialize.   y = y_shared + y_routed

(2) FINE-GRAINED -- split one fat expert (F=16) into 2 thin ones
    (F=8 each). top-k now mixes more specialized knowledge per
    token without raising total params. (demonstrated conceptually)

(3) AUX-LOSS-FREE bias -- instead of a gradient-pushing L_bal, add a
    learnable bias b_i to the router logits:
        G(x)_i = softmax( H(x)_i + b_i )
    and nudge b_i up/down by real load (no gradient to the model).

Token m=0, bias b = [0.0, 0.0, 0.3, 0.0]:
  plain   H[0,0]      top-2 experts = [1, 0]
  biased  H[0,0]+b    top-2 experts = [1, 2]  (expert 2 nudged by +0.30 -> flips out expert 0)

[check] bias changes routing decision?  True  (OK)

DeepSeek-V3 uses 256 routed + 1 shared expert, k=8: 671B total / 37B
active. The aux-loss-free bias keeps all 256 experts evenly loaded with
ZERO gradient interference into the main loss (unlike L_bal in §D).

========================================================================
  SECTION G: expert parallelism (EP) + grouped GEMM  [sketch, 1 device]
========================================================================
In Expert Parallelism, each expert lives on a DIFFERENT GPU. Routing
then needs an All-to-All communication (everyone sends everyone):

  step 1: each GPU computes router logits for its local tokens.
  step 2: All-to-All -- each token is sent to the GPU owning its top-k.
  step 3: each GPU runs ONLY its expert(s) on the tokens it received.
  step 4: All-to-All (reverse) -- send outputs back to the origin GPU.
  step 5: each origin GPU weights the returned outputs by G and sums.

(This is a SKETCH -- no real multi-GPU here. 🔗 TENSOR_PARALLEL covers
the collective primitives like All-to-All used in step 2/4.)

GROUPED GEMM -- the kernel that makes step 3 fast:
  After routing, expert i gets M_i tokens, and the M_i DIFFER per expert.
  torch.bmm needs equal batch sizes -> you'd have to PAD every expert's
  buffer to max(M_i) (wasteful). Grouped GEMM (CUTLASS/Triton) runs all
  E independent (M_i x N x K) matmuls in ONE kernel, no padding.

Our tiny model's token->expert assignment (the ragged input to grouped GEMM):
| expert i | tokens assigned (M_i)     | M_i |
|----------|---------------------------|-----|
| 0        | t0, t2, t3                | 3   |
| 1        | t0, t1                    | 2   |
| 2        | t1                        | 1   |
| 3        | t2, t3                    | 2   |

M_i is NOT uniform -> exactly the ragged case grouped GEMM solves.
DeepSeek-V3 / Mixtral training/inference engines all use grouped GEMM
