RAG-Slim (Edge Retrieval-Augmented Generation) — interactive 📖 RAG_SLIM.md 📄 rag_slim.py

📖 Read the full guide — architecture diagrams, decision tables, killer gotchas, cheat sheet. This page is the interactive companion.
Same pipeline, lighter parts. Server RAG (1024-dim embedder + HNSW + reranker) won't fit a phone. RAG-Slim swaps in a tiny 384-dim embedder, a flat cosine matrix, and a tight top-1..3 context. Pick a query below and watch the cosine bars rank the toy corpus; drag d & N and watch the index-memory bar react to the device RAM cap. Every number is recomputed in JS from the identical formula in rag_slim.py.  |  compare with grounding assertion →

① Query the flat index — cosine sim bars + assembled context prompt

How to use: pick a preset query (each is a hardcoded 8-dim vector, same as rag_slim.py Section B). The teal bars are cosine similarity to each chunk; the top-k (k from the slider) are highlighted and injected into the prompt readout. Or drag the three custom-tilt sliders to build your own query vector (pet-ness / code-ness / weather-ness axes).
(any non-zero tilt overrides the preset query)

② Does it fit the phone? — flat index memory vs device RAM (drag d & N)

How to read: the flat index is just the [N, d] float32 matrix — N·d·4 bytes, zero build cost. Drag d (embedder dim) and N (corpus size); the teal bar is index memory against the 512 MB phone-RAM cap. Even the largest realistic edge cell (d=384, N=100k = 154 MB) fits comfortably — that's why HNSW's graph memory is pure overhead here.
flat index = N·d·4 bytes; cap = 512 MB
