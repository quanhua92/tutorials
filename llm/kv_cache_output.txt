  SECTION B: dense cache shape evolution (prefill 3 -> decode 1)
========================================================================
Pre-allocated cache shape: (1, 2, 8, 8)  = [B=1, H_kv=2, max_seq_len=8, D=8]

DECODE 1 token:
  cache.k[:,:,:offset,:].shape = (1, 2, 4, 8)  (offset now 4)

========================================================================
  SECTION C: fragmentation math (why dense wastes, why paged doesn't)
========================================================================
All numbers below are COMPUTED here, not hand-waved. (fp16/bf16 = 2 B)

THE 93% EXAMPLE (max_seq_len vs actual usage, one request):
  max_seq_len reserved = 8192, actual tokens used = 512
  waste fraction = 1 - used/max = 1 - 512/8192 = 0.9375 = 93.75%
  -> 93.75% of this request's reserved slab is dead memory.

PER-REQUEST dense KV bytes (LLaMA-7B shape: 32 layers, 32 KV heads, d=128):
  2(KV) x 32 x 32 x 8192 x 128 x 2 B = 4,294,967,296 bytes = 4.000 GiB
  vLLM reports up to ~1.7 GiB for LLaMA-13B (more layers/heads) per seq.

100 CONCURRENT requests, reserved vs actually used (used=512 each):
  reserved = 400.0 GiB
  used     = 25.0 GiB
  wasted   = 375.0 GiB (93.8% of reserved)
  (This dead slab caps your batch size -> caps throughput.)

PAGED cache waste bound (only the LAST page of each request is partial):
| page_size | worst-case internal waste |
|-----------|---------------------------|
| 4         | 25.00% |
| 16        | 6.25%  <- vLLM default block size |
| 128       | 0.78%  <- tiny-llm page_size (real) |

vLLM measured: PagedAttention wastes <4% in practice vs 60%-80% for the
dense/over-reserving systems it replaced. (Sources: vLLM blog + paper.)

  SECTION D: paged pool + block_table (logical -> non-contiguous physical)

page_size = 4. Interleaved two requests so storage is scattered.

BLOCK TABLE (logical_page -> physical_page_id):
| request | logical pages (in order)        | physical pages      | contiguous? |

========================================================================
  SECTION F: rewind(n) on paged cache (speculative decoding, no off-by-one)
========================================================================
Start: prefill 5 tokens, page_size=4.
  page_ids  = [0, 1]   (logical -> physical)
  page_lens = [4, 1]   (page 0 full, page 1 has 1 token)
  offset    = 5
  free_list before rewind = [2, 3, 4, 5, 6, 7]

rewind(1): speculative draft rejected 1 token.
  page_ids  = [0]   <- physical page 1 RETURNED to free list
  page_lens = [4]   <- page 0 back to full (4 tokens)
  offset    = 4
  free_list after rewind  = [1, 2, 3, 4, 5, 6, 7]
  pool.used = [0]

THE OFF-BY-ONE TRAP (why rewind uses CEIL, not floor):
| offset | rewind(n) | new_offset | ceil(new/PS) pages kept | floor would keep | correct? |
|--------|-----------|------------|--------------------------|------------------|----------|
| 4      | 1         | 3          | 1                        | 0                | yes (ceil) |
| 5      | 1         | 4          | 1                        | 1                | yes (ceil) |
| 8      | 1         | 7          | 2                        | 1                | yes (ceil) |
| 5      | 5         | 0          | 0                        | 0                | yes (ceil) |
| 4      | 4         | 0          | 0                        | 0                | yes (ceil) |
| 9      | 1         | 8          | 2                        | 2                | yes (ceil) |

At offset=8 rewind(1): new=7. ceil(7/4)=2 pages kept (correct: 4+3).
  floor(7/4)=1 would DROP a still-needed page -> silent data loss.
At offset=5 rewind(1): new=4. ceil(4/4)=1 page kept (correct: 4 fill it).
  A naive 'new_offset//page_size' is WRONG at exact multiples.
