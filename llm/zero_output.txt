zero.py - reference impl (faithful single-process K-rank simulation).
Numbers below feed ZERO.md.  torch = 2.12.1

NOTE: no torch.distributed / NCCL is used. K ranks are a list of torch
tensors; ReduceScatter/AllGather are explicit Python loops. The
memory-bytes and communication-volume arithmetic is REAL and EXACT;
only the multi-rank execution is simulated.

========================================================================
  SECTION A: the DDP 20N baseline -- the redundancy ZeRO kills (🔗 DDP)
========================================================================
Per parameter, DDP mixed-precision Adam stores (on EVERY GPU):

| component           | bytes/param | why                                   |
|---------------------|-------------|---------------------------------------|
| fp16 parameters     |           2 | model weights used in the fp16 forward |
| fp16 gradients      |           2 | computed in the fp16 backward         |
| fp32 master params  |           4 | optimizer master copy (Micikevicius 2017) |
| fp32 grad copy      |           4 | upcast fp16 grad for the optimizer    |
| fp32 Adam momentum  |           4 | first moment (beta1 running avg)      |
| fp32 Adam variance  |           4 | second moment (beta2 running avg)     |
| TOTAL               |          20 | replicated xK = pure redundancy       |

  of the 20N:  4N is the live model (params+grads, needed full),
               16N is optimizer/grad bookkeeping -- IDENTICAL on all
               K GPUs. That 16N x K redundancy is exactly what ZeRO
               partitions away, stage by stage.

  [check] total == 20 and optimizer bookkeeping == 16? True

|---------------------|-------------|---------------------------------------|
| fp16 parameters     |           2 | FULL                                  |
| fp16 gradients      |           2 | FULL                                  |

========================================================================
  SECTION F: communication cost -- the per-device volume, in units of Psi
========================================================================
Psi = number of params. Ring ReduceScatter and AllGather each move
(K-1)/K * Psi ~= Psi per device; an AllReduce = RS + AG ~= 2*Psi.

| strategy | collectives / step                  | volume/device | vs DDP |
|----------|--------------------------------------|---------------|--------|
| DDP      | AllReduce(grad)  [= RS(grad)+AG(grad)] |           2.0*Psi |   1.00x |
| ZeRO-1   | ReduceScatter(grad) + AllGather(param) |           2.0*Psi |   1.00x |
| ZeRO-2   | ReduceScatter(grad) + AllGather(param) |           2.0*Psi |   1.00x |
| ZeRO-3   | AllGather(param fwd)+AllGather(param bwd)+ReduceScatter(grad) |           3.0*Psi |   1.50x |

  * ZeRO-1/2 REPLACE DDP's single gradient AllReduce with
    ReduceScatter(grad)+AllGather(param) -> SAME 2*Psi total volume,
    but it lets the optimizer state be partitioned in between.
    (DeepSpeed team: 'same communication volume as data parallelism'.)
  * ZeRO-3 adds one more AllGather (params gathered for BOTH fwd and bwd)
    -> 3*Psi = 1.5x DDP. (DeepSpeed team: 'modest 50% increase'.)

  NOTE on the loose '3x': some notes count ZeRO-3's THREE collectives
  (2 AllGather + 1 ReduceScatter) vs DDP's ONE fused AllReduce and call
  it '3x'. By actual BYTE VOLUME it is 3*Psi vs 2*Psi = 1.5x, per the
  ZeRO authors. The memory win is the point; the comm tax is small.

Concrete: 7B params, fp16 grads/params (2 bytes each), per-device GB:
| strategy | volume/device (GB) |
|----------|--------------------|
| DDP      |              28.00 |
| ZeRO-1   |              28.00 |
| ZeRO-2   |              28.00 |
| ZeRO-3   |              42.00 |

  [check] ZeRO-3/DPP volume ratio == 3.0/2.0 == 1.50? True

========================================================================
  SECTION G: why ZeRO comms work -- AllReduce == ReduceScatter ; AllGather (🔗)
========================================================================
K=4 ranks, N=8. Each rank holds a full local gradient.

DDP path : AllReduce-average(grads) ->
          every rank = [-0.0073, +0.1590, -0.0058, +0.0746, -0.2211, -0.2068, +0.1244, +0.0692]
ZeRO path: ReduceScatter-avg -> each rank owns ONE averaged chunk:
            rank 0 shard = [-0.0073, +0.1590]
            rank 1 shard = [-0.0058, +0.0746]
            rank 2 shard = [-0.2211, -0.2068]
            rank 3 shard = [+0.1244, +0.0692]
          AllGather -> every rank = [-0.0073, +0.1590, -0.0058, +0.0746, -0.2211, -0.2068, +0.1244, +0.0692]

  max|AllReduce - (ReduceScatter+AllGather)| = 0.000e+00
  [check] identity holds? True  (so ZeRO's RS+AG == DDP's AllReduce)

  ==> ZeRO-1/2 use the SAME total bytes as DDP: they just split the
      AllReduce at the seam (RS) so each rank can own 1/K of the state.
      ZeRO-3 does the AllGather of PARAMETERS twice (fwd + bwd) -> +50%.

========================================================================
  SECTION H: lineage recap + the per-stage memory ladder (K=8)
========================================================================
DDP replicates everything; ZeRO partitions ONE MORE category each stage:

| stage  | what it shards              | bytes/param | comms                    |
|--------|------------------------------|-------------|--------------------------|
| DDP    | replicate ALL                |          20 | AllReduce (2*Psi)        |
| ZeRO-1 | + shard optimizer states     |           6 | ReduceScatter+AllGather (2*Psi) |
| ZeRO-2 | + shard gradients            |        3.75 | ReduceScatter+AllGather (2*Psi) |
| ZeRO-3 | + shard parameters           |           2 | AG(fwd)+AG(bwd)+RS (3*Psi, 1.5x) |

Read each row left-to-right: at every step we shard ONE more category,
memory drops, and the comms cost stays ~the same until ZeRO-3 (+50%).
