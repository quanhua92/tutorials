TOAST — The Oversized-Attribute Storage Technique OVERFLOW_PAGES.md overflow_pages.py

A database page is an 8 KB suitcase. Slide the value's size past the ~2 KB threshold and watch PostgreSQL compress it, then chunk the remainder into a hidden TOAST table, leaving an 18-byte pointer in the main tuple. The compressor and payload generators are ported verbatim from overflow_pages.py and re-checked against its pinned gold values.

① Size & pattern — does this value fit inline, or must it be toasted?

What to do: drag size across the 2000-byte threshold and switch pattern. Watch the pglz step try to shrink it; if it still will not fit with the base tuple, it is toasted (red marker = threshold).
|
Rule: stored = pglz(raw) if smaller else raw; if stored + base_tuple > 2000 → out-of-line. Repetitive data collapses to almost nothing; random data does not compress at all (pglz abandons compression when the output is not strictly smaller).

② Main tuple → 18-byte pointer → TOAST chunk rows

Reading it: the main tuple keeps the value's small sibling columns plus either the bytes themselves (PLAIN/MAIN/inline) or an 18-byte varlena pointer (va_toastrelid, va_valueid, va_size, va_rawsize). The pointer's value id is the chunk_id shared by every chunk in pg_toast_<oid>, ordered by chunk_seq for reassembly.

③ Compression ratio by data pattern — why TOAST compresses FIRST

At the size from panel ①, pglz (modeled as greedy LZ77) hits each pattern very differently. Bars are raw → stored bytes; longer green = better compression. Random bytes barely move, which is why they fan out into many chunks under EXTERNAL.
