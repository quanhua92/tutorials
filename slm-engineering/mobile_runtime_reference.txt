[1] https://github.com/ggml-org/llama.cpp/blob/master/tools/quantize/README.md
    llama.cpp quantize tool README (ggml-org, official)
    Verifies: the bytes-per-weight figures. Q8_0 block = 8.5 bpw (=> 1.0625
    bytes/param); the Q4_K block = 4.5 bpw (=> 0.5625 bytes/param). So
    size_GB = params_billions * bpw / 8 = params * bytes_per_param / 1e9.
    Hence 1B@Q4 = 1e9 * 0.5625 / 1e9 = 0.5625 GB; 7B@Q4 = 3.9375 GB.

[2] /Volumes/data/workspace/tutorials/local-llm/VRAM_ESTIMATOR.md
    In-repo VRAM/memory-budget reference (the lineage source for this bundle)
    Verifies: the three-term budget VRAM = WEIGHTS + KV_CACHE + OVERHEAD; the
    KV-cache formula bytes = n_layers * 2 * n_ctx * n_kv_heads * head_dim *
    bytes_per_element (the factor 2 = K and V, both stored per layer); and the
    same bpw table (FP16=16.0, Q4_K_M=4.5, Q8_0=8.5). This bundle adapts that
    desktop-GPU math to mobile ACTIVE RAM.

[3] https://www.notebookcheck.net/Apple-A17-Pro-Processor-Benchmarks-and-Specs.756287.0.html
    Notebookcheck — Apple A17 Pro specs page (aggregator, cross-checked)
    Verifies: Apple A17 Pro Neural Engine = 16-core, 35 TOPS (INT8). Same 35
    TOPS INT8 figure recurs on the A18 / A18 Pro pages.

[4] https://en.wikipedia.org/wiki/Apple_A18
    Wikipedia — Apple A18 (cross-check for the Neural Engine TOPS)
    Verifies: A18 family ships a 16-core Neural Engine; the advertised 35 TOPS
    is the INT8 figure. Apple does not advertise native INT4 on the ANE
    (the NPU op set is INT8/FP16 oriented).

[5] https://www.qualcomm.com/content/dam/qcomm-martech/dm-assets/documents/Snapdragon-8-Elite-Gen-5-product-brief.pdf
    Qualcomm — Snapdragon 8 Elite product brief (official)
    Verifies: Qualcomm Hexagon NPU supports INT2, INT4, INT8, INT16, FP8, FP16
    (the broadest precision matrix of the mobile NPUs); the flagship Hexagon is
    quoted at ~45 TOPS class. INT4 support first landed in Snapdragon 8 Gen 2.

[6] https://www.qualcomm.com/content/dam/qcomm-martech/dm-assets/documents/Unlocking-on-device-generative-AI-with-an-NPU-and-heterogeneous-computing.pdf
    Qualcomm — "Unlocking on-device generative AI with an NPU" whitepaper (official)
    Verifies: the Hexagon NPU in Snapdragon 8 Gen 3 was redesigned for
    generative AI (transformer ops); supports mixed-precision INT4/INT8. This is
    the NPU backend ExecuTorch/LiteRT/QNN target for on-device LLM decode.

[7] https://docs.pytorch.org/executorch/stable/backends/coreml/coreml-overview.html
    ExecuTorch docs — Core ML / backends overview (official, PyTorch)
    Verifies: ExecuTorch is PyTorch's unified on-device runtime; it delegates to
    12+ hardware backends including CoreML (Apple ANE/GPU/CPU), Metal, XNNPACK
    (portable CPU), Vulkan, Qualcomm QNN, and MediaTek. One exported graph,
    pluggable execution providers.

[8] https://github.com/pytorch/executorch
    ExecuTorch GitHub (official repo, pytorch/executorch)
    Verifies: "On-device AI across mobile, embedded and edge for PyTorch" —
    the runtime targets smartphones to microcontrollers from a single PyTorch
    export. Confirms the unified-runtime / pluggable-backend design.

[9] https://onnxruntime.ai/docs/execution-providers/NNAPI-ExecutionProvider.html
    ONNX Runtime — NNAPI Execution Provider docs (official, Microsoft)
    Verifies: ONNX Runtime Mobile runs ORT-format models on Android via the
    NNAPI Execution Provider (a unified interface to CPU/GPU/NPU); sibling EPs
    exist for CoreML (Apple), QNN (Qualcomm), and (via the Web build) WebGPU.
    Graph-level quantized kernels + pluggable EPs from one model.

[10] https://onnxruntime.ai/docs/execution-providers/
     ONNX Runtime — Execution Providers overview (official)
     Verifies: the EP framework is the modular acceleration layer (CoreML, NNAPI,
     QNN, CUDA, DirectML, WebGPU) — the "one graph, many backends" property this
     bundle contrasts against fixed-function mobile NPUs and sandboxed browsers.

[11] https://developers.google.com/edge/litert
     Google — LiteRT (formerly TensorFlow Lite) docs (official)
     Verifies: LiteRT is the successor to TFLite, Google's on-device runtime with
     GPU/NPU acceleration; LiteRT-LM is the C++ library for running LLM pipelines
     across edge platforms. The Qualcomm AI Engine Direct accelerator yields up
     to ~100x over CPU on supported NPU graphs.

[12] https://huggingface.co/blog/transformersjs-v3
     Hugging Face — Transformers.js v3 announcement (official)
     Verifies: Transformers.js runs models in the browser via the ONNX Runtime
     Web backend, with WebGPU acceleration ("device: 'webgpu'") and a WASM
     fallback. This is the browser-runtime path (sandboxed, JIT'd, GC'd).

[13] https://www.sitepoint.com/webgpu-vs-webasm-transformers-js/
     SitePoint — "WebGPU vs WebASM: Browser Inference Benchmarks" (secondary)
     Verifies: WASM backend = broad compatibility but slower; WebGPU backend =
     GPU-accelerated but bears shader-compile first-run cost and is subject to
     browser GC pauses. The browser path trades portability for these overheads.

[14] https://arxiv.org/html/2509.23324v1
     Liu et al 2025 — "Scaling LLM Test-Time Compute with Mobile NPU on
     Smartphones" (arXiv:2509.23324)
     Verifies: confirms the Qualcomm Hexagon HMT/HMX unit supports INT4, INT8,
     INT16, and FP16; documents the real on-device decode throughput and the
     op-coverage gaps that force some transformer layers back onto the CPU even
     when the NPU is engaged (the "limited op coverage" caveat).

[15] https://www.mediatek.com/press-room/mediateks-new-all-big-core-design-for-flagship-dimensity-9300-chipset-maximizes-smartphone-performance-and-efficiency
     MediaTek — Dimensity 9300 launch / APU 790 (official press release)
     Verifies: MediaTek APU 790 (Dimensity 9300) supports integer and
     floating-point types for transformer/generative-AI workloads — the third
     major mobile-NPU family alongside Apple Neural Engine and Qualcomm Hexagon.
