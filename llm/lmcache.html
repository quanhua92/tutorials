LMCache — Hierarchical, Global KV-Cache Pooling — interactive LMCACHE.md lmcache.py

The global-library idea: λ single-node prefix cache (🔗 PREFIX_CACHE) shares KV on ONE node. In a cluster a request may land on a cold node → full prefill recompute. LMCache turns the datacenter's memory into ONE global, hierarchical pool (VRAM → DRAM → NVMe → S3); on a local miss a chunk-hash global lookup finds the KV on a sibling node and an RDMA pull streams it into the local block table (🔗 PAGED_ATTENTION), skipping prefill. Every number is recomputed in JS from the identical formulas in lmcache.py — bandwidths are real; the transport is a faithful simulation.

① The memory hierarchy — capacity grows down, bandwidth shrinks down

Reading: each rung holds more KV but pulls slower. Effective cross-node pull BW = min(network, dst PCIe). At 400 Gbps RoCE (~50 GB/s) vs PCIe Gen5 (~64 GB/s) the network is the bottleneck. See LMCACHE.md §3.

② Cross-node pull — local miss → global lookup → RDMA → block table

prompt [100,101,102,103,104,105] · chunk_size=2 · node1 DRAM has it, node0 is cold
What to do: click run. node0 looks up each chunk locally (MISS), asks the global index (HIT on node1 DRAM), and pulls the KV page into its block table. The gathered KV is byte-equal to the original — prefill skipped. See LMCACHE.md §5.

③ The budget: is it cheaper to MOVE the KV or RECOMPUTE it?

What to do: drag S and pick a tier. Size_KV grows with S; Latency_transfer = Size_KV / BW. The red line is the prefill roofline FLOOR (real prefill is higher). When the green bar stays under the red bar, transfer wins — pull, don't recompute. At S=512, RoCE 400G: transfer 0.252 ms vs prefill floor 3.407 ms (~13.5×). See LMCACHE.md §6.
