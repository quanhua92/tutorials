========================================================================
  SECTION B: WHY AllReduce keeps replicas identical (the proof)
========================================================================
Claim: DDP on K=2 GPUs with batches X0, X1 is MATHEMATICALLY EQUAL to
single-GPU training on the CONCATENATED batch [X0; X1]. Because the
loss is a MEAN, the gradient of the mean-of-two-halves equals the
average of the two half-gradients. So AllReduce-AVERAGE reconstructs
exactly the gradient one big GPU would have computed.

Single-GPU grad on cat([X0;X1]) (batch 4) = [+0.124783, +0.317437, -0.378959, +0.082346, -0.065859, -0.167272, -0.326616, -0.574194, +0.667979, +0.090280, -0.071918, -0.204379]
DDP averaged grad (AllReduce of g0,g1)        = [+0.124783, +0.317437, -0.378959, +0.082346, -0.065859, -0.167272, -0.326616, -0.574194, +0.667979, +0.090280, -0.071918, -0.204379]

========================================================================
  SECTION D: AMP + GradScaler -- fp16 forward, fp32 master, loss scaling
========================================================================
Mixed precision (Micikevicius et al. 2017, arXiv:1710.03740):
  - forward + backward in fp16 (fast, half the memory)
  - keep an fp32 MASTER copy of weights (accumulates tiny updates)
  - fp16 has a TINY range [6e-5, 65504]: small gradients UNDERFLOW to 0
  - GradScaler multiplies the LOSS by a big constant (e.g. 2^16) so the
    backward gradients are pushed up into fp16's representable range,
    then divides them back before the optimizer step.
  - bfloat16 shares fp32's EXPONENT range -> no underflow -> no scaler.

| grad (fp32)    | fp16(grad) | survived? | bf16(grad) |
|----------------|------------|-----------|------------|
| 1e-03          | 1.000e-03  | YES       | 9.995e-04  |
| 1e-05          | 1.001e-05  | YES       | 1.001e-05  |
| 1e-07          | 1.192e-07  | YES       | 1.001e-07  |
| 1e-08          | 0.000e+00  | NO (underflow!) | 1.001e-08  |

  Notice: 1e-8 rounds to 0.0 in fp16 but survives in bf16. fp16's
  smallest subnormal is ~6e-8, so anything smaller vanishes silently.

--- How GradScaler rescues an underflowing gradient ---
  true grad      = 1.000e-08  (fp32)
  fp16(grad)     = 0.000e+00  <-- LOST (underflow to 0)
  scale = 2^16   = 65536
  fp16(grad*scale)= 6.552e-04  <-- SURVIVES
  unscale back   = 9.997e-09  <-- recovered ~true grad

  [check] scaled->fp16->unscaled recovers the true grad?  True

--- The full fp16 step flow (nanoGPT/train.py) ---
  loss_scaled = loss * S                          # scale up
  scaler.scale(loss_scaled).backward()            # grads ~Sx bigger
  scaler.unscale_(optimizer)                      # divide grads back
  clip_grad_norm_(...)                            # clip on true grads
  scaler.step(optimizer)                          # skips if inf/nan
  scaler.update()                                 # adapt S for next step

  bf16 path: drop the scaler entirely (torch.amp.autocast(dtype=bf16)).

========================================================================
  SECTION E: cosine LR + linear warmup (nanoGPT get_lr)
========================================================================
Linear scaling rule (Goyal et al. 2017, arXiv:1706.02677): when you
multiply the batch size by K, scale the peak LR by K, and WARM UP
linearly from 0 to avoid early-training instability. Then cosine-decay
peak -> min_lr for a smooth landing (Chinchilla: decay_iters ~= max_iters).

|----|--------------|--------|
|  0 | warmup (linear) | 0.0909 |
|  2 | warmup (linear) | 0.2727 |
|  4 | warmup (linear) | 0.4545 |
|  6 | warmup (linear) | 0.6364 |
|  8 | warmup (linear) | 0.8182 |
| 10 | cosine decay | 1.0000 |
| 15 | cosine decay | 0.9657 |
| 25 | cosine decay | 0.7222 |
| 35 | cosine decay | 0.3778 |
| 45 | cosine decay | 0.1343 |
| 50 | cosine decay | 0.1000 |
| 55 | hold @ min_lr | 0.1000 |

  peak=1.0, min_lr=0.1, warmup=10, decay_iters=50

========================================================================
  SECTION F: the 20-bytes/param memory breakdown (mixed-precision Adam)
========================================================================
Per parameter, DDP mixed-precision training stores:

| component           | bytes/param | why                                   |
|---------------------|-------------|---------------------------------------|
| fp16 parameters     |           2 | the model weights used in the fp16 forward |
| fp16 gradients      |           2 | computed in the fp16 backward         |
| fp32 master params  |           4 | optimizer's master copy (Micikevicius 2017) |
| fp32 grad copy      |           4 | upcast fp16 grad for the optimizer    |
| fp32 Adam momentum  |           4 | first moment (beta1 running avg)      |
| fp32 Adam variance  |           4 | second moment (beta2 running avg)     |
| TOTAL               |          20 | ~16N effective (20N raw)              |

Concrete sizes (DDP = this is REPLICATED on every GPU):
| model        | params    | bytes/param | per-GPU (DDP) |
|--------------|-----------|-------------|---------------|
| tiny demo W  |        12 |          20 |        0.00 GB |
| GPT-2 small  | 124,000,000 |          20 |        2.31 GB |
| LLaMA-3 8B   | 8,000,000,000 |          20 |      149.01 GB |
| GPT-3 175B   | 175,000,000,000 |          20 |     3259.63 GB |

  [check] total = 2+2+4+4+4+4 = 20 bytes/param
  --> DDP replicates ALL 20N bytes on every GPU. That redundancy is
      exactly what ZeRO-1/2/3 eliminate by sharding these states. 🔗
      (🔗 QUANTIZATION attacks the 2-byte fp16 weight; ZeRO attacks the
       16-byte optimizer/grad/master redundancy. Orthogonal axes.)
