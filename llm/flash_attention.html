Naive attention materializes an [N,N] matrix in HBM; FlashAttention tiles K,V through SRAM carrying a running max m, sum l, output o — and is EXACT. Every number is recomputed in JS from the identical formula in flash_attention.py.  |  position half (RoPE) →
How to read this page (no code needed): attention builds a giant N×N score table. Naive code writes it to slow HBM and reads it back — that shuffling, not the math, is what's slow. FlashAttention does the work in tiny tiles in fast SRAM instead, so the big table is never written to slow memory. The clever bit is online softmax: keep a running HIGH SCORE (m), SUM (l), and OUTPUT (o); whenever a tile reveals a higher score, multiply the past by a correction factor to fix its scale. Divide o/l at the end. Step through the panels below — the green badge at the bottom proves the answer is identical to naive.

① Naive attention materializes an [N,N] score matrix in HBM — drag N

d = 8, scale = 1/√d. Q,K regenerated deterministically per N (the N=8 case is the pinned gold below).
What you're looking at: the N×N table of "how much each token cares about each other token". The gold cell in each row is that row's highest score — the one softmax must know first. Drag N up: the table grows as N², and at N=8192 it becomes 256 MiB — written to slow HBM and read right back at every layer of every head. That round-trip is the bottleneck.
scores S = Q@KT·scale   [8×8] — row max in gold
HBM traffic for the score matrix (naive, ~4N² floats)
At N=8192, d=64 (4-byte float): N²·4 = 256 MiB per head per layer — read+written at every layer.

② Step through the tiled online-softmax — watch m, l, o and the rescaling factor

N=8, Bc=4 → 2 K/V tiles. Step 0=before, 1=after tile j=0, 2=after tile j=1 (final ÷ l).
How to use this: pick a query row (start with q=0), then slide tiles done → 1 → 2. Watch the running HIGH SCORE m, running SUM l, and running OUTPUT o update each chunk. The correction = exp(m_old − m_new) lights up orange whenever a tile reveals a new higher score — that factor shrinks the past to the new scale. Try q=0: at tile 1 the correction jumps to 0.7305 (a new max appeared). That single factor is the whole trick.
The correction = exp(m_old − m_new) rescales ALL prior accumulators when the running max rises. On the first tile it is 0 (no past to fix); if the max does not change it is 1 (nothing to fix). Step q=0 to tile 1: correction jumps to 0.7305. That single factor is the whole trick (FLASH_ATTENTION.md §3).

③ Tiled FlashAttention vs naive — identical output (EXACT, not approximate)

The payoff: the left table (tiled, never builds the big matrix) and the right table (naive, builds it) are mathematically identical — the biggest difference across all 64 entries is pure float noise (~3e-8). Same multiplications, far fewer trips to slow memory. This is why FlashAttention is called exact, not an approximation.
