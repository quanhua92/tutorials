kube-proxy — iptables & IPVS data plane — interactive KUBE_PROXY.md kube_proxy.py

kube-proxy runs on every node and programs kernel rules so a packet to the ClusterIP (10.96.0.10:80) is DNAT'd in flight to a pod. Two data planes: iptables (a linear -m statistic chain) or IPVS (a kernel L4 LB with O(1) lookup). Slide the request count and switch weights — the page re-runs the same seeded xorshift32 + the deterministic IPVS wrr scheduler as kube_proxy.py Section GOLD.

① The chain: client dials ClusterIP → kube-proxy DNAT → one pod

What to look at: a client dials the stable 10.96.0.10:80. kube-proxy on the originating node rewrites the destination to a pod IP (DNAT) BEFORE the packet leaves. Toggle the data plane to see the different kernel mechanisms — iptables walks a rule chain, IPVS does one hash lookup.

③ conntrack — ESTABLISHED flows stick to the first-chosen backend

What this shows: a connection's first packet runs the statistic chain and picks a pod. Every later packet of the same connection is fast-pathed via conntrack to the same pod — the chain is skipped. sessionAffinity: ClientIP extends stickiness to NEW connections from the same IP.
