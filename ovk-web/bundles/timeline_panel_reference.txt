# timeline_panel — web provenance log
# Every URL has a "Verifies:" line stating exactly what it confirms.
# STEP 2 of HOW_TO_RESEARCH.md is non-skippable; ≥2 sources per concept.

https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Array/reduce
  Verifies: the cumulative-start / prefix-sum math the timeline uses. "reduce() ... executes a user-supplied 'reducer' callback function on each element of the array, in order, passing in the return value from the calculation on the preceding element. The final result ... is a single value." With an initialValue, "callbackFn starts executing with the first value in the array as currentValue" and the accumulator carries the running sum forward — exactly how start[i] = sum(dur[0..i-1]) is computed in Section B (cumulativeStarts implemented with reduce). Confirms the prefix-sum algorithm is the standard, correct way to turn per-clip durations into sequential start offsets.

https://en.wikipedia.org/wiki/Gantt_chart
  Verifies: the bar-chart timeline visual model. "A Gantt chart is a bar chart that illustrates a project schedule ... This chart lists the tasks to be performed on the vertical axis, and time intervals on the horizontal axis. The width of the horizontal bars in the graph shows the duration of each activity ... Gantt charts illustrate the start and finish ... of the terminal elements." This is precisely the Timeline panel's model: each slide is a task/clip laid on a horizontal time axis, bar width = the slide's measured duration, and each bar's start is the cumulative end of the previous one. Independent second source confirming that a sequential timeline = (ordered clips, each sized by duration, positioned by cumulative start).

https://en.wikipedia.org/wiki/Non-linear_editing
  Verifies: what a full multi-track NLE is (and thus what this timeline deliberately is NOT). A non-linear editing system arranges clips on multiple parallel tracks with overlaps, J/L cuts, and keyframed effects — the "Premiere/Resolve class" the RFC excludes. Confirms RFC §3 Non-Goal ("Full multi-track NLE ... Scene-based only"): OpenVideoKit's timeline has ONE sequential scene lane (no parallel tracks, no overlapping clips), which Section F asserts.

(in-repo) docs/rfc-0001.md §7 "Editor Surfaces"
  Verifies: the Timeline row of the Editor Surfaces table, quoted verbatim — "Timeline (bottom) | binds to root index.json (slide order) + each slide duration | reorder slides, see per-slide durations, audio lane". Primary in-repo source for what the panel binds to and what its job is.

(in-repo) docs/rfc-0001.md §5.5 "Why JSON holds only data (not the timeline structure)"
  Verifies: the central idea — "The editor's 'timeline' panel is a *view* over (slide order + measured durations) — not a rich structure it persists. This keeps the data layer small, stable, and ideal for small-model AI edits (RFC 0002)." This is the Section A + Section E claim that the timeline persists NOTHING of its own.

(in-repo) docs/rfc-0001.md §3 "Non-Goals"
  Verifies: verbatim — "Full multi-track NLE (Premiere/Resolve class). Scene-based only." Confirms the Section F non-goal assertion (single sequential scene lane, no overlapping clips).

(in-repo) docs/rfc-0001.md §5.2 "Root index.json" + §5.3 "Slide index.json"
  Verifies: §5.2 — root carries the `slides` ordering array (the spine the timeline reorders) and the `audio` model {music {loop:true}, voiceover}. §5.3 — each slide carries its own measured `duration` ("duration: 5.0, // measured (from voiceover timings)"), which is the per-clip lane width the timeline reads.

(in-repo) docs/AGENTS.md "Preview audio" + "Voiceover pipeline"
  Verifies: the audio lane contract — "ext-music: music-bed, looped, volume 0.08" and "ext-voiceover: voiceover track (if assets/voiceover.mp3 exists), volume 1.0, not looped". Also "Smart timing: ... start times are computed as prev_end + gap_between_slides (default 0.8s)" — confirms the inter-slide gap formula start[i] = sum(dur[0..i-1]) + i*gap.
