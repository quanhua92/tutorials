shared_embeddings_reference.txt
================================
Web provenance log for the SHARED_EMBEDDINGS bundle.

Every formula, signature, year, model-config value, and behavioral claim is
traced below to >=2 sources where possible. Distinct URLs: >= 2.
The reader-facing summary lives in SHARED_EMBEDDINGS.md "## Sources".

--------------------------------------------------------------------------------

[1] https://arxiv.org/abs/1608.05859
    Press & Wolf 2017 "Using the Output Embedding to Improve Language Models"
    (EACL 2017, arXiv:1608.05859; official).
    Verifies: the central claim of the bundle -- "when training language models,
    we recommend tying the input embedding and this [output] embedding" (verbatim
    from the paper's recommendation). The paper shows the topmost weight matrix
    of an LM "constitutes a valid word embedding" and that tying reduces
    perplexity (not merely saves parameters). This is the canonical reference for
    WHY small/medium LMs tie input + output embeddings.
    Author code: https://github.com/ofirpress/UsingTheOutputEmbedding

[2] https://arxiv.org/abs/1611.01462
    Inan, Khosravi, Socher 2017 "Tying Word Vectors and Word Classifiers: A Loss
    Framework for Language Modeling" (ICLR 2017, arXiv:1611.01462; official).
    Verifies: the independent derivation of weight tying from a shared loss
    framework (the input word-vector and the output classifier share a matrix).
    Published ICLR 2017. PDF mirror: https://web.stanford.edu/~inanh/papers/tying-wv.pdf
    Together with [1], this is the two-paper foundation of the tying technique.

[3] https://huggingface.co/HuggingFaceTB/SmolLM2-1.7B-Instruct-16k/blob/main/config.json
    HuggingFaceTB/SmolLM2-1.7B config.json (official model card).
    Verifies: the GOLD anchor of this bundle -- SmolLM2-1.7B has
    "vocab_size": 49152 and "tie_word_embeddings": true. So tying saves
    V*H = 49152 * 2048 = 100,663,296 parameters (the [check] and the .html
    gold badge both assert exactly this integer). hidden_size = 2048 per the
    SmolLM2 paper / config.

[4] https://huggingface.co/HuggingFaceTB/SmolLM2-1.7B
    HuggingFaceTB/SmolLM2-1.7B model card (official).
    Verifies: SmolLM2-1.7B is a 1.7B-parameter SLM; family of 135M / 360M / 1.7B.
    Confirms the SLM regime (<5B) in which tying is the default.

[5] https://www.heywhale.com/mw/dataset/662b644709492666bdd18aea
    Meta-Llama-3-8B-Instruct config.json mirror (reproduces the gated official
    config.json verbatim).
    Verifies: the KEY CONTRAST of the bundle -- Meta-Llama-3-8B has
    "tie_word_embeddings": false (i.e. it is UNTIED). vocab_size = 128256,
    hidden_size = 4096. This is the empirical evidence for "large models untie":
    at 8B scale the parameter budget permits two independent [V, H] matrices and
    the cleaner per-path gradient wins on perplexity. Section E + Section F.

[6] https://huggingface.co/docs/transformers/en/model_doc/gemma2
    Hugging Face Transformers -- Gemma2 model documentation (official).
    Verifies: Gemma-2-2B hidden_size defaults to 2304. Gemma models ship with
    tie_word_embeddings enabled (the vLLM Gemma source asserts
    `config.tie_word_embeddings` is True by design; see also
    https://github.com/vllm-project/vllm gemma model). vocab_size = 256000 (Gemma
    tokenizer default). So tying saves V*H = 256000 * 2304 = 589,824,000.

[7] https://discuss.huggingface.co/t/what-is-the-tie-word-embeddings-option-exactly-doing/8483
    Hugging Face Forums -- "What is the tie_word_embeddings option exactly doing?"
    Verifies: the IMPLEMENTATION MECHANISM the bundle implements in torch --
    "the embedding weights (vocab_size by hidden_size) are tied with the decoder
    (hidden_size by vocab_size) so the model only learns one [V, H] matrix."
    Confirms the `tie_word_embeddings` config flag semantics and that tying is a
    parameter-sharing rebind, not a copy.

[8] https://github.com/karpathy/nanoGPT
    Karpathy, nanoGPT (canonical minimal GPT-2 reference implementation).
    Verifies: GPT-2 ties by construction via the literal line
    `self.lm_head.weight = self.wte.weight` (in nanoGPT's GPT class). GPT-2 uses
    V = 50257, H = 768, tie_word_embeddings = true. This is the canonical "tying
    in practice" code pattern the bundle reproduces in Section B.

[9] https://arxiv.org/html/2603.26663v1
    "Weight Tying Biases Token Embeddings Towards the Output Space" (recent
    analysis, 2025).
    Verifies: the "when untying helps at scale" claim in Sections E/F and the
    pitfalls table. Finding (verbatim): "these results help explain why weight
    tying can harm performance at scale and have implications for training
    smaller LLMs, where the embedding [is shared]." The tied weight receives
    gradient from BOTH the input path and the output path (the bundle's Section C
    proves this numerically), which biases the tied embeddings toward the output
    space; at large scale an untied pair can win on perplexity.
    Supporting discussion: https://www.reddit.com/r/MachineLearning/comments/1eqm0lr/

[10] https://mbrenndoerfer.com/writing/weight-tying-shared-embeddings-transformers
    Brenndoerfer, "Weight Tying: Sharing Embeddings Between Input and Output
    Layers" (technical write-up).
    Verifies: the shape-matching fact that makes tying trivial -- nn.Linear(H, V)
    stores its weight as [V, H] (out_features, in_features), which is EXACTLY the
    shape of nn.Embedding(V, H).weight, so a plain attribute rebind
    `lm_head.weight = embed.weight` is the whole mechanism (Section B). Also
    confirms tying "helps rather than hurts performance" on small models.

[11] https://github.com/meta-llama/llama/issues/138
    meta-llama/llama issue #138 -- "Are the weights of the lm head of the model
    tied with the word embeddings?"
    Verifies: the tying-vs-untying decision is a real, discussed engineering
    trade-off at Llama scale ("does it have an effect on training stability?").
    Community confirms Llama-3 keeps lm_head and embed untied.

UNVERIFIED / APPROXIMATE FACTS (flagged for honesty):
- The SmolLM2-1.7B hidden_size of 2048 is taken from the SmolLM2 config/paper;
  the gold integer 100,663,296 depends only on vocab_size (49152, verified
  directly in config.json) and H=2048. If H differed the gold would differ; the
  .py prints V*H explicitly so it is re-derivable.
- The exact crossover scale where untying beats tying is empirical and
  model/data dependent; the bundle states this explicitly ("not a hard
  threshold"). The Llama-3-8B = untied data point is the concrete evidence.
- Gemma-2 vocab_size 256000 is the Gemma tokenizer default; the config pins it.
  The savings integer 589,824,000 is computed from V*H and asserted in the .py.
