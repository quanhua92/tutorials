Delta Encoding — store the change, not the number DELTA_ENCODING.md delta_encoding.py

[100,102,105,106,110] → [100, +2, +3, +1, +4]. Delta encoding stores differences between consecutive values; combined with varint it crushes smooth (monotonic / slowly-varying) data and does nothing on random data. It powers git packfile deltas, time-series databases, and video P-frames. Every number below is recomputed live in JS from the identical input as delta_encoding.py Section E.

① The pipeline — values → delta → zigzag → varint → bytes

What to do: delta alone doesn't shrink anything — it only changes WHICH values are big. Varint rewards small magnitudes, so the two compose: delta makes magnitudes small (on smooth data), varint packs them.

Zigzag interleaves the number line so small magnitudes (both signs) map to small unsigned values: 0→0, -1→1, 1→2, -2→3, 2→4. That is why tiny negative deltas still cost only 1 byte.

③ Compression ratio — smooth data crushes, random data doesn't

What this proves: delta+varint is a BET that consecutive values are close. Monotonic timestamps (all deltas +1) win big; uniform random loses (deltas are as big as the values). Bars recomputed live; the green check: OK confirms the canonical example matches delta_encoding.py Section E.
Note: JS uses 32-bit integer ops for zigzag/varint (matches the Python output for these in-range examples; Python's n>>63 generalizes to 64-bit). Real systems like git/Parquet apply delta then feed the result to a general compressor on top.
