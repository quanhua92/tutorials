📖 Read the full guide — architecture diagrams, decision tables, killer gotchas, cheat sheet. This page is the interactive companion.
Writes append to a WAL then land in an in-memory memtable; on overflow the memtable flushes to an immutable SSTable; background compaction merges them. Reads walk memtable → L0 → L1 → … short-circuited by per-SSTable Bloom filters. Re-run live in JS, identical to key_value_store.py.

① Step the write path — WAL → memtable → flush → SSTable → compaction

What to do: step through. Each PUT/DEL is appended to the WAL then the memtable (dashed gold). When the memtable hits the limit (4), it flushes to an immutable L0 SSTable (green). When L0 has ≥ 4 SSTables, size-tiered compaction merges them into one (still at L0 for STCS). Updated keys show green; tombstones show red strikethrough.

② Read path — watch a GET walk the levels + bloom filters

Type a key and run a GET. The walker visits the memtable, then every L0 SSTable newest-first (L0 ranges overlap, so all must be checked), then L1. Each SSTable is tested with a range guard, then a Bloom filter; only if both pass do we pay a disk read. The first hit wins; tombstones return NOT FOUND.
