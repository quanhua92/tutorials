Slotted Page (PostgreSQL heap storage) — interactive 📖 SLOTTED_PAGE.md 📄 slotted_page.py

📖 Read the full guide — architecture diagrams, decision tables, killer gotchas, cheat sheet. This page is the interactive companion.
A database page is a fixed-size block (PostgreSQL = 8 KB; this toy = 128 B with the identical structure). A line-pointer array grows from the left while tuples fill from the right, squeezing the free space in the middle. Drag the slider to insert rows, click a tuple to delete it (it turns red = a dead hole), and try a HOT update. The layout is recomputed live in JS with the same logic as slotted_page.py.  |  self

① The page, filling from both ends

What to do: drag # tuples to watch line pointers (purple) grow from the left and tuples (green) fill from the right. Click a green tuple to delete it (red = dead hole). The contiguous free gap only reclaims a hole after Compact (VACUUM).
header (PageHeaderData, 24 B) line pointer (ItemIdData, 4 B) live tuple (click to delete) dead hole (reclaimable) REDIRECT (HOT) free space
In words: free = pd_upper − pd_lower. Inserts carve tuples out of pd_upper (pushing it down) and line pointers out of pd_lower (pushing it up). Delete marks a line pointer DEAD but does NOT grow the contiguous gap unless the tuple sat right next to it — holes need a VACUUM compaction (section C in the .py).

② HOT update — the old line pointer redirects; indexes never move

What this shows: an UPDATE that keeps the new row on the same page and touches no indexed column is a Heap-Only Tuples update. The new version is appended and the OLD line pointer becomes a REDIRECT — so every index, which points at the old (page, item), stays valid with zero index writes.
If the new version didn't fit on the page it would NOT be HOT: the row would move to another page and every index would need a new entry. HOT exists to dodge that cost. (Pick an item in panel ① and press HOT update → to populate this.)

③ Max tuples per page = floor( (page − header) / (ip + tuple) )

What to do: pick a real page size and a row payload width. Each row costs one 4-B line pointer plus its tuple (24-B header + payload on the 8 KB page).
This is an upper bound: real capacity is a touch lower due to MAXALIGN padding on tuple headers and free-space fragmentation from deletes.
