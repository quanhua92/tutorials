① Config tuner — slide RAM, get your postgresql.conf

rules: shared_buffers=25%RAM · effective_cache_size=75%RAM · wal_buffers=min(sb/32,16MB)
shared_buffers is a real allocation (the page cache). effective_cache_size is a planner hint — it does not allocate memory. Set it high so the planner trusts the OS cache and picks indexes.

② EXPLAIN analyzer — seq vs index vs bitmap cost by selectivity

1.1 = SSD · 4.0 = HDD default
cost vs selectivity (log X). Pick the lowest line.
winner by selectivity band
Cost model (identical to postgresql.py Section C): seq = pages·1.0 + tuples·0.01 · index = idxPages·rpc + sel·tuples·(rpc+0.01) · bitmap = idxPages·rpc·0.25 + sel·pages·rpc + sel·tuples·0.01

③ Index picker — answer four questions, get the access method

Verdict recomputes the decision tree from postgresql.py Section E. The default B-tree wins most scalar work; BRIN is the secret weapon on huge ordered append-only columns (~1000× smaller).

④ Partition simulator — how much a range query saves with monthly partitions

pages scanned: full vs pruned
RANGE/LIST/HASH all supported (PG10+). HASH only prunes on equality. DROP TABLE orders_2023_01 expires data instantly — no VACUUM.

⑤ Replication lag calculator — when the replica falls behind

If apply < write, backlog grows forever. Fixes: tune autovacuum on the replica, raise max_standby_streaming_delay, add replicas, or shard. Streaming = raw WAL bytes (same major version); logical = decoded row changes (cross-version).

⑥ Hosting cost — self-hosted vs RDS vs Aurora

Self-hosted bill is low but you pay in engineering time (patching, backups, replication, PITR drills). RDS/Aurora trade money for not being paged at 3am. Reserved Instances cut RDS/Aurora ~30-40% for a 1- or 3-year commit.

⑦ VACUUM visualizer — dead tuples accumulate, then get reclaimed

MVCC: UPDATE = INSERT new + mark old DEAD. VACUUM reclaims dead tuples (marks pages reusable) but does not shrink the file. Only VACUUM FULL (table lock) returns space to the OS — partition + DROP old data instead.
