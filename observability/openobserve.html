OpenObserve — S3-native observability (Rust + Parquet) OPENOBSERVE.md openobserve.py

A single Rust binary that stores logs, metrics & traces as Parquet on S3 and queries them directly off object storage. Five stateless-friendly roles, ~15× log compression, and a ~104× storage-cost gap vs Elasticsearch. Recompute the cost model, compaction math, and query round-trips live below — all mirroring openobserve.py.

① S3 cost calculator — O2 vs Loki vs Elasticsearch (storage)

What this shows: monthly storage cost as you scale ingest volume + retention. O2 stores compressed Parquet (1/15 raw) on S3; Loki gzip chunks (1/8) on S3; ES primary+replica (~2× raw) on EBS gp3. ES also needs compute that scales with data (not shown). Identical model to openobserve.py Section E.

② Architecture explorer — the five roles

What this shows: click a node to see its job. In single-node mode all five collapse into ONE process; in HA mode they scale horizontally (Router/Querier/Ingester/Compactor/AlertManager) over NATS + PostgreSQL + S3.
apps → Router → Ingester (WAL→Memtable 256MB→Parquet) → S3 → Querier (stateless, gRPC fan-out) → UI  ·  Compactor merges files  ·  AlertManager fires destinations

③ Query playground — SQL + full-text on logs

What this shows: the same 5-record deterministic dataset as openobserve.py Section C. Pick a query template — SQL filter, full-text (match_all), or aggregation — and see the simulated hits + which index it uses.
match_all uses the Tantivy full-text index (fields: log/message/msg/content/data); WHERE col= uses Parquet predicate + column pushdown.

④ Compaction visualizer — small Parquet files → big ones

What this shows: the Compactor merges small S3 Parquet files up to a 2 GB ceiling (ZO_COMPACT_MAX_FILE_SIZE). Fewer, larger files = far less planning + fewer S3 GETs per query. Drag the small-file count/size.
O2 default ceiling = 2048 MB (2 GB). At 1200×2 MB the output is exactly [2048, 352] — matching openobserve.py Section D.
