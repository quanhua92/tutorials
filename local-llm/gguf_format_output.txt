gguf_format.py - GGUF (GPT-Generated Unified Format) binary layout.
Pure Python stdlib (struct only). Numbers below feed GGUF_FORMAT.md.
Sources: gguf-py/gguf/gguf.py + ggml.h + HuggingFace GGUF docs.

Four sections: header -> KV metadata -> tensor info -> aligned tensor_data.

==========================================================================
  SECTION E: LINEAGE - GGML (flat) -> GGUF (why each step)
==========================================================================
WHY GGUF exists: the old GGML format was a flat dump of tensors with
NO self-describing metadata. Adding a new model architecture meant
editing the C loader code and bumping a breaking format version.

GGUF fixes this with a versioned schema + a typed KV section that can
describe ANY architecture / tokenizer / chat template from inside the
file. One loader, many models, backward compatible.

stage            what changed                                   why
------------------------------------------------------------------------------
GGML (old flat)  tensors only, no metadata, breaking change per architecture every new model = a code change + a new loader
GGUF v1/v2       + KV metadata + version field + single-file layout architecture/tokenizer described IN the file
GGUF v3 (now)    + general.alignment KV + tokenizer.ggml.* + chat template KV, mmap-friendly aligned tensor_data one loader runs every architecture; mmap loads weights lazily

The mmap win: tensor_data is aligned to a multiple of 32, so the OS can mmap it directly. On a 4 GB
model that is the difference between a ~12 s eager load and a ~0.1 s
lazy startup (pages are faulted in on first access).
See MMAP_WEIGHTS.md for the page-fault / copy-on-write mechanics.
