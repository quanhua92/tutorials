mmap Weights — lazy-loaded model files, page-fault by page-fault 📖 MMAP_WEIGHTS.md 📄 mmap_weights.py

"Map, don't read." llama.cpp loads a GGUF with mmap(2): the file becomes a virtual byte array, and a 4 KiB page is read from disk only when inference first touches it (a page fault). So startup is ~page-table setup (~0.1 s) and RSS climbs to only the pages actually read. MAP_PRIVATE makes writes copy-on-write (dequant in place without touching the file); the OS page cache lets N processes share one physical copy. Recompute the whole model live below.

① Startup time & RSS — full read() load vs mmap()

What this shows: a full read() load must pull the entire file before the first token (startup ~ file_size / disk_bw, RSS = file_size at once). mmap() only sets up page tables (~0.1 s) and RSS climbs to just the pages touched. Adjust model size, disk speed, and access fraction.
read() load = model / disk_bw. mmap() = page-table setup (~constant 0.1 s). RSS under mmap = model x touched%. Same formulas as mmap_weights.py Section F.

② Page-fault simulator — first touch faults, repeats are free

What this shows: after mmap(), every page is "not present". The first touch of a page is a major fault (read from disk); every later touch is a free hit. Fault count = distinct pages, not access count. Edit the access plan and watch the table.
major = first-ever touch of a page (disk read, via the shared page cache). hit = page already mapped in this process (no fault). Mirrors mmap_weights.py Section B.

③ Copy-on-write & multi-process sharing — one physical page per (file, page)

What this shows: MAP_PRIVATE writes get a private page copy (the GGUF on disk is never touched); the page cache lets N processes share one physical frame. Set the write fraction (COW copies) and the number of instances (sharing).
COW duplicates only written pages (+1 page/write). Sharing: N instances reference the same cached frame, so total RAM ~ one copy, not N. Mirrors mmap_weights.py Sections C & D.

④ mmap vs pread — same lazy loading, different mechanism

What this shows: when mmap() is unavailable (networked / sandboxed filesystems), llama.cpp falls back to pread() — explicit on-demand reads. Same lazy effect, but no transparent faults and no free cross-process sharing.
