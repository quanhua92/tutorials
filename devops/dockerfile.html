Dockerfile build — layers, cache cascade & multi-stage DOCKERFILE.md dockerfile.py

Think of each instruction as a workstation on an assembly line: after each one a photographer snaps a photo (= a layer). On the next build, the photographer reuses a photo if everything up to that workstation is identical (HIT); the first thing that changed has no matching photo (MISS) — and so does everything after it (the cascade). ① Each instruction commits a layer — the canonical build (cold cache)

What this shows: a 4-instruction Dockerfile. The 3 FROM/RUN/COPY instructions each commit a filesystem layer; CMD only edits the image config (no layer). The layer id encodes its whole ancestry, so an ancestor change rewrites every descendant id.
Layer ids pinned from dockerfile.py Section A (sha256(parent_key + kind + content_tag)).

② The build cache — pick which line you edited, watch the cascade

What to do: choose the instruction whose INPUT changed (an edited file under COPY counts even though the line text is identical — its checksum is part of the key). From that line onward every instruction is a MISS and must re-run.
Editing COPY (the common case: you changed app source) reuses the expensive ubuntu + apt layers and re-runs only COPY + CMD. That is why you order least-changing inputs first (Section B of the .py).

③ Multi-stage — build fat (800 MB), ship thin (20 MB)

What this shows: stage 1 (builder) has the Go compiler + source and produces a binary; stage 2 (runtime) is alpine + just that binary. The final image is the last stage only — the compiler never ships.
