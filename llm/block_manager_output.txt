========================================================================
  SECTION A: the Block struct + ref_count semantics
========================================================================
Block(7) just created:
  block_id  = 7
  ref_count = 0   (<- 0 => on the free list, not yet used)
  hash      = -1   (<- -1 => never hashed yet)
  token_ids = []

ref_count is the SHARING dial:
| ref_count | state          | what it means                         |
|-----------|----------------|---------------------------------------|
| 0         | FREE           | on the free list; KV+hash STAY if any|
| 1         | LIVE (private) | exactly one sequence reads this block |
| >=2       | LIVE (shared)  | N sequences read the SAME bytes       |

Lifecycle of one block as readers come and go:
  _allocate_block()       -> block 0, ref_count=1, hash=0xfece5a30e12404d4
  2nd seq shares it       -> ref_count=2 (SHARED)
  1 seq leaves            -> ref_count=1
  last reader leaves      -> ref_count=0 -> FREE list
  BUT block.hash still    = 0xfece5a30e12404d4  and token_ids=[10, 11]
      => a future [t0,t1] prefix can RECLAIM it (no KV recompute)

[check] freed block is in free_block_ids: True -> OK

| req | block | tokens     | prefix (prev hash) | chained hash (this block)        | same as other req? |
|-----|-------|------------|---------------------|----------------------------------|--------------------|
| A   | 0     | [10, 11]   | -1 (root)           | 0xfece5a30e12404d4               | share w/ B blk0    |
| B   | 0     | [10, 11]   | -1 (root)           | 0xfece5a30e12404d4               | share w/ A blk0    |
| A   | 1     | [12, 13]   | 0xfece5a30e12404d4  | 0x2cb48a641a5b1565               | DIVERGES from B    |
| B   | 1     | [14, 15]   | 0xfece5a30e12404d4  | 0x2ab4c06583ce3085               | DIVERGES from A    |

[check] hash(A.block0) == hash(B.block0)  (shared prefix) : True -> OK
[check] hash(A.block1) != hash(B.block1)  (divergence)    : True -> OK

WHY the chain matters: block1's hash folds in block0's hash. So even
if [t2,t3] happened to equal [t4,t5] (they don't here), block1 would
still differ because the prefix hash fed in differs. The chain makes a
block's fingerprint a function of its ENTIRE token prefix, not just its
own tokens — exactly what content-addressed prefix dedup needs.

========================================================================
  SECTION E: hash_blocks -> register FULL blocks only (the #1 pitfall)
========================================================================
Request C = [10, 11, 12], block_size=2 => 2 logical blocks:
  block0 = [10, 11]  (FULL)
  block1 = [12]  (PARTIAL — only 1 of 2 tokens)

Execute 3 tokens. hash_blocks boundary math:
  start = num_cached_tokens//bs = 0//2 = 0
  end   = (cached+scheduled)//bs = 3//2 = 1
  -> register blocks [0..1) = block 0 ONLY. block 1 (partial) is SKIPPED.

hash_to_block_id after hash_blocks(C):
    0xfece5a30e12404d4 -> block 0 (tokens [10, 11])
  (block1's partial [12] is NOT registered.)

WHY partials are skipped — the false-cache-hit it prevents:
  Suppose D = [t0,t1,t2,99] arrives later. block1 of D = [t2,99].
  If we had hashed C's partial block1=[t2], a naive lookup of D's
  block1 might collide on the token t2 prefix and reuse WRONG KV.
  By hashing ONLY full blocks, D's block1 is a guaranteed MISS until
  it actually fills with [t2,99] — stable and correct.

[check] only 1 block registered (the full one) : True -> OK
[check] a 4-token request registers 2 full blocks : True -> OK

========================================================================
  SECTION F: can_append / may_append -> the page-boundary rule
========================================================================
A = [10, 11, 12, 13] (len=4, block_size=2). A.block_table = [0, 1] (2 pages, both full).

Decode tokens one at a time and watch when a new page appears:
| step | append | len(seq) | len%bs | new page needed? | may_append allocates? | A.block_table | free blocks |
|------|--------|----------|--------|------------------|------------------------|---------------|-------------|
| 1    | 20     | 5        | 1      | YES (boundary)   | YES -> new page        | [0, 1, 2]     | 3           |
| 2    | 21     | 6        | 0      | no               | no                     | [0, 1, 2]     | 3           |
| 3    | 22     | 7        | 1      | YES (boundary)   | YES -> new page        | [0, 1, 2, 3]  | 2           |
| 4    | 23     | 8        | 0      | no               | no                     | [0, 1, 2, 3]  | 2           |

Rule: a new physical page is allocated EXACTLY when len(seq) % block_size == 1.
At len=5 (5%2=1): page allocated. At len=6 (6%2=0): token drops into
the existing partial page — no allocation. At len=7 (7%2=1): page again.

[check] can_append at len=2 (no page needed) with empty pool: True (True, 0 pages needed) -> OK
[check] can_append at len=3 (boundary, page needed) with empty pool: False (False => OOM => preempt) -> OK
