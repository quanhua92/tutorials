==========================================================================
  SECTION B: OLLAMA CALL STACK - Go server -> llama.cpp -> Metal/CUDA
==========================================================================
Ollama: MIT (open source), written in Go.
CLI-first. No built-in GUI (pairs with Open WebUI for a web UI).
OpenAI-compatible API at localhost:11434/v1/chat/completions.

What `ollama run llama3.2 "Hello"` actually does, layer by layer:

  ollama run llama3.2 "Hello"
      [CLI client (Go)]
    |-resolve model -> registry tag -> local blob path
        [Ollama server (Go)]
    |-if missing: ollama pull -> download GGUF from registry
        [Ollama server (Go)]
    |-spawn llama.cpp runner subprocess (cgo / shared C lib)
        [Ollama server (Go)]
      |-load GGUF: header + KV + tensor info + mmap tensor_data
          [llama.cpp (C)]
      |-detect GPU backend: Metal | CUDA | ROCm (auto-offload)
          [llama.cpp (C)]
      |-build GGML compute graph for this architecture
          [llama.cpp (C)]
        |-tokenize prompt  ->  token ids
            [llama.cpp (C)]
        |-forward pass: matmul + RoPE + attention + sampling
            [llama.cpp (C)]
        |-detokenize -> text, stream tokens back over stdout
            [llama.cpp (C)]
    |-relay token stream to the terminal
        [Ollama server (Go)]

Key insight: the Go layer is a *thin orchestrator*. All math runs in
the llama.cpp subprocess. Ollama adds: model registry, the Modelfile,
the OpenAI-compatible HTTP API, and auto GPU detection -- nothing more.

Ollama 0.19+ adds an MLX backend on Apple Silicon (up to 93% faster
decode for supported models) -- but the orchestration shape is unchanged.

==========================================================================
  SECTION C: LM STUDIO CALL STACK - Electron -> llama.cpp/MLX -> GUI
==========================================================================
LM Studio: Proprietary (free), written in TypeScript / Electron.
GUI-first. Built-in HuggingFace model browser + side-by-side compare.
Local OpenAI-compatible API server at localhost:1234/v1/chat/completions.

What clicking 'Load model' then 'Send' actually does:

  LM Studio app window (chat panel)
      [Electron renderer (React UI)]
    |-Model browser tab -> HuggingFace API (search/filter/download)
        [Electron renderer]
    |-user picks model -> one-click GGUF/MLX download to ~/.cache
        [Electron renderer]
    |-click 'Start Server' -> local API on port 1234
        [Electron main (Node.js)]
    |-click 'Send' in chat -> request to in-process llama.cpp
        [Electron main (Node.js)]
      |-load GGUF or MLX weights (mmap)
          [llama.cpp / MLX (native binding)]
      |-detect GPU (Metal/CUDA) or MLX backend; multi-GPU supported
          [llama.cpp / MLX]
      |-build compute graph
          [llama.cpp / MLX]
        |-forward pass -> tokens
            [llama.cpp / MLX]
    |-stream tokens back to renderer, render in chat bubble
        [Electron renderer]
    |-OPTIONAL: 2nd model column -> same prompt, both responses shown
        [Electron renderer]

Key insight: the Electron layer adds *discovery + UX*. Same engine
underneath, but the value is the in-app HuggingFace browser, the chat
GUI, and side-by-side model comparison -- none of which Ollama has.

Trade-off: no CLI, no headless mode. You cannot script it in CI.

==========================================================================
  SECTION D: THE SHARED OPENAI-COMPATIBLE API
==========================================================================
Both expose /v1/chat/completions with the SAME request body shape.
Client code is portable: change the host:port, nothing else.

Request body (identical for both):
  POST /v1/chat/completions
  Content-Type: application/json
  {"messages": [{"content": "Hello", "role": "user"}], "model": "llama3.2", "stream": false, "temperature": 0.7}

  -> Ollama:    POST http://localhost:11434/v1/chat/completions
  -> LM Studio: POST http://localhost:1234/v1/chat/completions

Only the HOST:PORT differs. The path, the JSON schema, and the
streaming SSE response format are the same (OpenAI-compatible).

Example: swap a curl call between the two by changing one number:
  curl http://localhost:11434/v1/chat/completions -d '{"messages": [{"content": "Hello", "role": "user"}], "model": "llama3.2", "stream": false, "temperature": 0.7}'
  curl http://localhost:1234/v1/chat/completions -d '{"messages": [{"content": "Hello", "role": "user"}], "model": "llama3.2", "stream": false, "temperature": 0.7}'

==========================================================================
  SECTION F: LINEAGE - raw llama.cpp -> Ollama + LM Studio
==========================================================================
WHY these exist: raw llama.cpp requires manual compile + manual GGUF
download + manual flag tuning. The UX gap kept local LLMs out of reach
for most users. Two different philosophies filled it:

stage              what changed                                             why
------------------------------------------------------------------------------
raw llama.cpp      compile C++ yourself, hunt for GGUFs, memorize -ngl/-c/-t flags powerful but hostile UX; no model discovery; no API
Ollama (2023)      Go server + CLI + Modelfile + registry + OpenAI API, all CLI-driven developer-friendly: `ollama run` just works; scriptable; CI-ready
LM Studio (2023)   Electron GUI + HuggingFace browser + chat + side-by-side compare non-developer-friendly: point-and-click discovery; no terminal

Both converge on the SAME outcome: hide llama.cpp's rough edges. The
divergence is WHO they hide them from -- devs (Ollama) vs everyone else
(LM Studio). When you outgrow single-user local serving, the next step
is vLLM (production multi-user throughput). See VLLM_SERVING.md.
