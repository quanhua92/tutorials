========================================================================
  SECTION D: float32 upcast is MANDATORY in bfloat16  (why we upcast)
========================================================================
Problem: bfloat16 has only 8 mantissa bits (vs float32's 23). Squaring
small values AND summing them loses precision; in the worst case the sum
of squares underflows to 0, RMS becomes sqrt(eps), and the output EXPLODES.

PART 1: precision loss  (small activations, eps = 1e-5)

Input x (8 small values, stored as bfloat16):
  ['0.0010', '0.0020', '0.0015', '0.0030', '0.0025', '0.0018', '0.0022', '0.0017']
mean(x^2) computed each way:
  bfloat16 (no upcast) : 4.202127e-06
  float32  (upcast)    : 4.189518e-06
  ratio (bf16 / f32)   : 1.00301  <- bf16 over-estimates

max| no-upcast  -  upcast | over the output = 1.278e-03
bfloat16 out: ['0.2656', '0.5312', '0.3984', '0.7969', '0.6641', '0.4785', '0.5820', '0.4512']
float32  out: ['0.2653', '0.5306', '0.3990', '0.7980', '0.6643', '0.4780', '0.5833', '0.4517']

The two normalizations DIFFER because bf16's 8 mantissa bits cannot
hold the sum of squares faithfully. In production (E=4096+) this error
compounds across the reduction. Upcasting to float32 before squaring is
cheap insurance and is what every production RMSNorm does.

PART 2: catastrophic underflow  (tiny activations, eps = 0)

Input x (8 tiny values, stored as bfloat16):
  ['1.00e-22', '2.00e-22', '2.99e-22', '4.00e-22', '5.00e-22', '5.99e-22', '7.01e-22', '8.01e-22']
mean(x^2) with eps = 0:
  bfloat16 (no upcast) : 0.000e+00   <- rounds to EXACTLY 0
  float32  (upcast)    : 2.550e-43   <- tiny but nonzero

x / sqrt(mean(x^2))  with eps = 0:
  bfloat16 (no upcast) : contains NaN/inf? True   <- DEAD
  float32  (upcast)    : max |out| = 1.5855  (finite, correct)

Without the upcast, the bf16 sum of squares underflows to 0 and (with
eps=0) the division blows up to NaN. Adding eps=1e-5 prevents the NaN
but still yields a WRONG scale whenever the true mean-square is below
eps. Upcasting to float32 before squaring is the robust fix: it preserves
the true (tiny) mean-square so the division stays well-conditioned.

This is why every production RMSNorm (LLaMA, Qwen, Mistral, Gemma) upcasts
x to float32 before squaring, then casts back. See rmsnorm_from_scratch.

========================================================================
  SECTION G: the lineage  LayerNorm -> RMSNorm  (why each step)
========================================================================

LayerNorm   Ba, Kiros, Hinton 2016  (arXiv:1607.06450)
    out = (x - mean) / sqrt(var + eps) * gamma + beta
    - 2 reduction passes (mean, then var)
    - 2 affine params per feature (gamma, beta)
    - gives re-centering AND re-scaling invariance
    - GPT-2 / nanoGPT / original Transformer use this

        |
        |  Zhang & Sennrich (2019) hypothesized:
        |  "re-centering invariance is dispensable;
        |   re-scaling invariance is what actually helps."
        |  -> drop the mean subtraction, keep only the RMS scaling.
        v

RMSNorm     Zhang & Sennrich 2019  (arXiv:1910.07467, NeurIPS 2019)
    out = (x / sqrt(mean(x^2) + eps)) * gamma
    - 1 reduction pass (mean of squares only)
    - 1 affine param per feature (gamma; NO beta)
    - gives re-scaling invariance only
    - 7%-64% faster than LayerNorm at comparable accuracy (the paper)
    - LLaMA / Mistral / Qwen / Gemma / Falcon all use this

MANDATORY in both: float32 upcast of x before squaring when running in
bfloat16/fp16 (else sum-of-squares underflows -> NaN). See Section D.
