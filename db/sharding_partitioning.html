① The three partitioning strategies — INSERT a row, watch it route

What to do: switch strategy, type a partition-key value, hit INSERT. The matching partition glows green. The same routing table is printed by sharding_partitioning.py Sections A/B/C. The hash is the same FNV-1a the .py uses, so user_id=42 lands in ph3 in both.
Read it as: the logical table orders/users/events is physically several smaller tables. The partition key's value (or its hash) is the address. That address is what makes pruning possible in panel ②.

② Partition pruning — a WHERE clause greys out partitions the executor never opens

What to do: choose a predicate shape per strategy. Kept partitions stay bright; pruned ones grey out. Try HASH + range — it CANNOT prune (hash is not monotonic), so nothing greys. Mirrors sharding_partitioning.py Section D.
Rule of thumb: RANGE prunes ranges, LIST prunes IN/eq, HASH prunes only equality. Plan a range scan on a hash key and you fan out to every partition.

③ Distributed sharding (Citus / Vitess) — route a query across nodes

What this shows: 4 hash-range shards, round-robin placed on 2 worker nodes (exactly sharding_partitioning.py Section F). A query with the shard key hits ONE node (zero fan-out); a query without it fans out and the coordinator gathers / gather-merges the streams.
Shard key = 42 hashes to …, which lives in shard S2 on node0 — matching the .py's Section F exactly. Co-located joins (same shard key) run locally per node; no data moves. That is the design lever: pick the shard key so most queries stay single-shard.
