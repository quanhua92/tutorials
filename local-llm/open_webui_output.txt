========================================================================
  SECTION A: what Open WebUI is (frontend, not a model runner)
========================================================================
Open WebUI is a SELF-HOSTED WEB FRONTEND. It does not run models; it
connects to backends and adds RAG, tools, multi-user RBAC on top.

Backends it connects to (docs.openwebui.com/features):
| backend         | local/remote | endpoint                       | serves                  |
|-----------------|--------------|--------------------------------|-------------------------|
| Ollama          | local        | http://localhost:11434         | any GGUF model via /api/chat |
| OpenAI API      | remote       | https://api.openai.com         | GPT-4o, o-series        |
| Anthropic       | remote       | https://api.anthropic.com      | Claude (via OpenAI-compat endpoint) |
| OpenAI-compat   | either       | <any base_url>                 | vLLM, LM Studio, llama.cpp server, Together, Groq, ... |

The flagship feature set:
  * Multi-model chat   - talk to any model, switch mid-chat, run two side-by-side
  * RAG / Knowledge    - upload docs -> chunk -> embed -> vector search -> inject
  * Hybrid search      - BM25 (keyword) + vector (semantic) + cross-encoder rerank
  * 9 vector DBs       - ChromaDB, PGVector, Qdrant, Milvus, Elasticsearch, ...
  * MCP                - Model Context Protocol: external tools the model can call
  * Pipelines          - modular plugins that filter / transform / route messages
  * Multi-user RBAC    - roles, groups, per-model access, SSO/OIDC/LDAP, SCIM
  * Open Terminal      - AI writes + executes Python in a real environment

Deploy:
  pip install open-webui && open-webui serve   # one command
  docker run -d -p 3000:8080 ... ghcr.io/open-webui/open-webui:main

========================================================================
  SECTION B: the RAG pipeline + chunking strategies
========================================================================
RAG = give the model YOUR documents without stuffing them all in context.
The pipeline (Open WebUI's Knowledge feature):

  Document -> [CHUNK] -> [EMBED] -> [VECTOR DB]
  Query    -> [EMBED] -> [cosine top-k] -> [RERANK] -> [INJECT] -> [LLM]

CHUNKING is where retrieval quality is won or lost:
  * TOKEN-based  (e.g. 512 tokens): production default, good throughput.
  * SENTENCE-based: preserves meaning boundaries (no mid-sentence cuts).
  * OVERLAP (e.g. 50 tokens): repeats context at the seam so a fact split
    across two chunks is not lost. step = size - overlap.
  * too small -> lose context; too large -> dilute relevance + cost.

Demo document (15 words):
  'The cat sat on the mat. The dog ran in the park. Cats love fish.'

Chunk it at size=5, overlap=2 (step=3):
| # | chunk                          | words |
|---|--------------------------------|-------|
| 0 | The cat sat on the             | 5     |
| 1 | on the mat. The dog            | 5     |
| 2 | The dog ran in the             | 5     |
| 3 | in the park. Cats love         | 5     |
| 4 | Cats love fish.                | 3     |

Effect of chunk size on chunk count (overlap=2 fixed):
| size | overlap | step | #chunks |
|------|---------|------|---------|
| 3    | 2       | 1    | 13      |
| 5    | 2       | 3    | 5       |
| 7    | 2       | 5    | 3       |
| 10   | 2       | 8    | 2       |
Smaller chunks => more windows => finer recall but more embeddings to store.

========================================================================
  SECTION C: fake embeddings + cosine vector search
========================================================================
We cannot ship a real embedding model (nomic-embed-text, 768 dims) in pure
stdlib. Instead: the FEATURE-HASHING TRICK. Each word -> FNV-1a hash -> one
of 256 dims -> count -> L2-normalize. Two texts sharing words land in
the same buckets, so cosine tracks lexical overlap. Deterministic (FNV-1a is
NOT Python's randomized hash()).

Corpus vocabulary (15 words) -> hash bucket (mod 256):

========================================================================
  SECTION D: hybrid search (BM25 + vector) + cross-encoder rerank
========================================================================
Pure vector search misses exact keyword matches (a product code, a name).
BM25 is the classic keyword ranker; it rewards rare terms and short docs.
HYBRID = alpha*vector + (1-alpha)*BM25. Then a CROSS-ENCODER reranks the
top-k for precision. Open WebUI ships both.

BM25 index stats (k1=1.5, b=0.75):
  N (chunks) = 5,  avgdl = 4.60 tokens
  query content terms: ['cats', 'eat']

| chunk                          | bm25  | vector  | rerank |
|--------------------------------|-------|---------|--------|
| The cat sat on the             | 0.000 | 0.0000  | 0.0000 |
| on the mat. The dog            | 0.000 | 0.0000  | 0.0000 |
| The dog ran in the             | 0.000 | 0.0000  | 0.0000 |
| in the park. Cats love         | 0.843 | 0.2236  | 0.2000 |
| Cats love fish.                | 1.038 | 0.2887  | 0.3333 |
