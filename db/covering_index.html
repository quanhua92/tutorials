① Build the index — add INCLUDE columns, watch it grow

What to do: flip INCLUDE columns on. Each adds its byte width to every leaf entry → fewer entries per leaf → more leaves → a bigger index. The key column email is always present (and is the only sort key).
INCLUDE name (30 B) INCLUDE age (4 B) INCLUDE city (20 B)
8 KB pages · 40 tuples/heap page
The plain index (email only) is the baseline. Each INCLUDE column is stored in the leaf but not compared, so the B-tree sort order (and key fanout) never changes — only the entry size and leaf capacity do. (COVERING_INDEX.md §4)

② The access path — regular (2 trips) vs covering (1 trip)

What to do: switch the query's index. Regular: index leaf has only (email, TID) → follow the TID to the heap → read name (2 logical steps, a random heap read). Covering: name is already on the card → skip the heap. Toggle the visibility map off and watch the covering path re-insert a heap fetch to check MVCC visibility.
visibility map: ON (all-visible) SELECT name FROM users WHERE email = 'cara@x.com'
