========================================================================
  SECTION A: the algorithm -- Lomuto partition, last-element pivot
========================================================================
Worked array: [3, 8, 2, 5, 1, 4, 7, 6]   (n = 8)

The whole algorithm is two lines of recursion around ONE pass:

    partition arr[lo..hi] around a pivot  ->  pivot lands in its
    FINAL sorted position p; everything left is < pivot,
    everything right is > pivot. Then recurse on both halves.

Top-level partition of arr[0..7], pivot = arr[7] = 6:

  stash pivot 6 at end:     [3, 8, 2, 5, 1, 4, 7, 6]
  compare j=0 (val 3):  <= pivot -> grows small region
    swap i=0<->j=0:          [3, 8, 2, 5, 1, 4, 7, 6]
  compare j=1 (val 8):  > pivot -> skip
  compare j=2 (val 2):  <= pivot -> grows small region
    swap i=1<->j=2:          [3, 2, 8, 5, 1, 4, 7, 6]
  compare j=3 (val 5):  <= pivot -> grows small region
    swap i=2<->j=3:          [3, 2, 5, 8, 1, 4, 7, 6]
  compare j=4 (val 1):  <= pivot -> grows small region
    swap i=3<->j=4:          [3, 2, 5, 1, 8, 4, 7, 6]
  compare j=5 (val 4):  <= pivot -> grows small region
    swap i=4<->j=5:          [3, 2, 5, 1, 4, 8, 7, 6]
  compare j=6 (val 7):  > pivot -> skip
  place pivot at index 5: [3, 2, 5, 1, 4, 6, 7, 8]   <- pivot locked

Pivot 5 final index. Left = [3, 8, 2, 5, 1], right = [7, 6].

Full quicksort (last pivot): [3, 8, 2, 5, 1, 4, 7, 6] -> [1, 2, 3, 4, 5, 6, 7, 8]
Total comparisons = 15   (n-1 + n-2-1 + ...; theoretical avg ~ 1.39 n log2 n = 33.4)

========================================================================
  SECTION B: the four pivot strategies  (first / last / random / median-of-3)
========================================================================
Worked array: [3, 8, 2, 5, 1, 4, 7, 6]

For the TOP-LEVEL call arr[0..7], each strategy picks:
| strategy    | pivot index | pivot value | how chosen                      |
|-------------|-------------|-------------|---------------------------------|
| first       | 0           | 3           | just arr[lo]                    |
| last        | 7           | 6           | just arr[hi]                    |
| random      | 6           | 7           | rng.randint(0,7) (seed 0)       |
| median3     | 3           | 5           | median of arr[0]=3, arr[3]=5, arr[7]=6 |

first/last are the dangerous ones on sorted/reversed input (see Section C).
median-of-3 costs 2-3 extra compares per call but avoids the worst case
on the two commonest real-world inputs (already-sorted, reverse-sorted).
random makes the WORST case a probability event, not an input pattern.

n = 32. comparisons per strategy x input:

| input       |     first |      last |    random |   median3 | best possible |
|--|--|--|--|--|-
| random      |       145 |       125 |       131 |       113 |           160 |

Read it row by row:
  - 'sorted'   with first/last pivot -> the WORST CASE. Each partition
    peels off ONE element, giving n+(n-1)+...+1 = n(n-1)/2 comparisons.
    first/last on sorted/reversed ~ 496 = O(n^2).
  - median-of-3 and random stay near n log2 n on EVERY input.
  - 'best possible' ~ n log2 n = 160. No comparison sort
    beats this in the worst case (decision-tree lower bound).

========================================================================
  SECTION D: quicksort vs mergesort vs heapsort  (counts on n=64)
========================================================================
Input: 64 random ints (seed 1). All three return the same sorted
output: [1, 2, 3, 3, 4, 4]...[98, 98, 99]

| sort          | comparisons | auxiliary memory | stable | worst case   |
|---------------|-------------|------------------|--------|--------------|
| quicksort(m3) | 306         | O(log n) stack   | no     | O(n^2)*      |
| mergesort     | 311         | O(n) extra array | yes    | O(n log n)   |
| heapsort      | 568         | O(1) in-place    | no     | O(n log n)   |

  * median-of-3 quicksort's O(n^2) is vanishingly rare on random input;
    introsort (std::sort) makes the GUARANTEE O(n log n) by switching to
    heapsort after 2*log2(n) recursion levels.

Why quicksort usually WINS in practice despite no guarantee:
  - tightest inner loop: one compare + one branch + maybe one swap, all on
    a contiguous array -> superb cache locality + branch prediction.
  - heapsort hops around a tree stored in an array (parent/child are 2i+1,
    2i+2 apart) -> cache-unfriendly, ~2-3x slower on real hardware.
  - mergesort touches O(n) extra memory -> more cache traffic, slower than
    quicksort unless you specifically need STABILITY or linked-list sort.

========================================================================
  SECTION E: when to use quicksort (and when NOT to)
========================================================================
USE quicksort when:
  - you sort general arrays in RAM and want the fastest average speed
    (this is why std::sort / sorted() are quicksort/introsort-based).
  - you can afford O(log n) stack + no stability requirement.

AVOID quicksort when:
  - you need a WORST-CASE guarantee without introsort's fallback
    (hard real-time, adversarial input) -> heapsort.
  - you need STABILITY (equal keys keep their order) -> mergesort.
  - the data is a linked list or streamed -> mergesort (O(1) merge).
  - n is tiny -> insertion sort (fewer ops, no recursion overhead);
    real quicksorts switch to insertion sort below ~10-20 elements.

Rule of thumb: the default in nearly every stdlib is INTROSORT
(quicksort + median-of-3 + heapsort fallback + insertion-sort cutoff).
You get quicksort's speed AND heapsort's O(n log n) guarantee.
