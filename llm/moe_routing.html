Mixture-of-Experts Routing — top-k gating, load balance, DeepSeek-V3 — interactive MOE_ROUTING.md moe_routing.py

For newcomers: a dense FFN runs ALL its params per token. MoE replaces it with a router (the receptionist) + E expert MLPs (specialists), and each token goes to only the top-k. Drag token m, E, and k below — every number is recomputed live in JS from the identical formula in moe_routing.py.  |  → dense SwiGLU   → GQA   → tensor parallel   → sampling

① Router: logits H → KeepTopK → softmax gate G (pick a token)

How to use: pick a token m. Yellow bars are the 4 raw router logits H = x·W_gᵀ. KeepTopK zeroes all but the top-k, then softmax renormalizes the survivors to sum=1 — those are the orange gate weights G. Only the lit experts actually run.
H(x) router logits (raw scores) G(x) gate weights (active top-k, sum=1) inactive expert (G=0, never runs)

② Expert routing: token → top-k SwiGLU experts → weighted sum → output

How to use: the selected token (left) is sent to its top-k experts (middle, each a full SwiGLU MLP 🔗 mlp_activation). Outputs are weighted by G and summed into the per-token output vector (right, D=8 dims). Slide k to see more/fewer experts contribute.
input token x[D] active expert (SwiGLU) + its gate G inactive expert output y = Σ Gᵢ·Eᵢ(x)
Structure: y = Σᵢ G(x)ᵢ · Eᵢ(x), each Eᵢ(x) = down(silu(gate(x)) · up(x)) (Shazeer 2020 SwiGLU). Total params ∝ E, active params ∝ k.

③ Load balance across the batch + the router-collapse problem

How to use: these bars count how many of the 4 tokens each expert received (the fᵢ in the load-balance loss L_bal = E·Σ fᵢPᵢ). With small E/k an imbalance is easy — that is router collapse, fixed by L_bal (+ z-loss). The dashed line is the capacity C = capacity·N·k/E: bars past it would be dropped in expert-parallel training.
