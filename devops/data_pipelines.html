A pipeline is an assembly line for data: Extract → Transform → Load. Below: an ETL pass with data-quality quarantines, the DAG an orchestrator runs (topological order, fan-out, cycle rejection), backpressure throttling a producer, and exactly-once delivery. Toggle batch vs streaming to watch the latency/throughput tradeoff flip. All numbers recompute live from the same model as data_pipelines.py.

① ETL — extract, transform (clean + enrich), load

6 raw events enter. Transform validates each: amount must be a non-negative number, currency must be known, then normalizes to USD via the FX table {usd:1.0, eur:1.10}. Bad rows are quarantined; good rows load to the warehouse.

② Pipeline DAG — topological order, parallel fan-out, cycle rejection

The orchestrator (Airflow/dbt/Dagster) reads the task dependency graph and groups tasks into levels: each level has no inter-deps and runs in parallel. The deepest chain (critical path) bounds wall-clock time. A cycle is rejected — pipelines must be acyclic.

③ Backpressure — a slow consumer throttles the producer

10 events pushed as fast as possible into a bounded buffer; the consumer drains rate per tick. When the buffer is full the producer stalls (backpressure) instead of drowning memory.

④ Delivery semantics — at-most / at-least / exactly-once

What this proves: 5 events, a crash injected at ids 2 and 4 (each retried once). The JS recomputes all three delivery modes from the same model as data_pipelines.py Section D. The blue check: OK confirms exactly-once yields 0 duplicates + 0 losses.

⑤ Batch vs streaming — the latency / throughput tradeoff

8 events, setup=2 ticks, proc=1 tick/record. Batch amortizes setup over the batch (higher throughput) but every record waits for the window (higher latency). Streaming pays setup per record (lower throughput) but skips the window (lower latency).
