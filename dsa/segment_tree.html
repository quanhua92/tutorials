Segment Tree & Fenwick Tree — interactive SEGMENT_TREE.md segment_tree.py

Both make range-sum query and point-update O(log N). The segment tree stores a SUM per segment; a query tiles [l,r] with the fewest precomputed blocks. The Fenwick tree uses the i & -i bit trick to do the same in N space.  |  ← Trie

① The segment tree for [1, 3, 5, 7, 9, 11]

Each node stores the SUM over its segment [lo,hi]; the root covers the whole array; leaves are single elements. #i = the node's 1-indexed slot in the 4N backing array (children of i are 2i, 2i+1).

② Range query — tile [l, r] with the fewest maximal segments (O(log N))

What to do: drag l and r. Blue nodes are visited; green nodes are FULL segments summed directly (no scan). Notice how few segments tile even a wide range.
0-indexed, inclusive. r ≥ l enforced.

③ Point update — set arr[i] = v, propagate up one root-to-leaf path (O(log N))

What to do: pick an index and a new value. The gold leaf is rewritten; every green ancestor on the path up to the root is recomputed from its children.
old value shown faded in the readout.

④ Fenwick tree (BIT) — the i & -i bit trick, prefix sums in N space

What to do: drag k. The walk peels off lowbit(k) each step (k -= k & -k). Each highlighted slot is added; the gold box shows the covered range. sum(l..r) = prefix(r+1) − prefix(l).
1-indexed Fenwick slots; slot i covers [i−lowbit(i)+1, i].

⑤ Segment tree vs Fenwick — head to head (recomputed live in JS)

Same array, same queries: both trees agree with brute force. Fenwick can't do min/max (no inverse), so use the segment tree for general associative combines.
