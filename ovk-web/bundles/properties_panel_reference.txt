# properties_panel — web provenance log
# Every URL has a "Verifies:" line stating exactly what it confirms.
# STEP 2 of HOW_TO_RESEARCH.md is non-skippable; ≥2 sources per concept.

https://developer.mozilla.org/en-US/docs/Web/API/Node/textContent
  Verifies: the "light DOM patch vs full re-render" concept. MDN: "If the node is a ... text node, textContent returns, or sets, the text inside the node, i.e., the Node.nodeValue" — so setting textContent on (or mutating) the ONE text node bound to a field is a text-only update, not an HTML re-parse. MDN also warns "Setting textContent on a node removes all of the node's children and replaces them with a single text node with the given string value" — a localized mutation, far cheaper than re-stamping the whole slide HTML and remounting the sub-comp. Contrast (same page): innerHTML "needs to invoke the HTML parser" — that is the full re-stamp path live-bind avoids.

https://developer.mozilla.org/en-US/docs/Web/API/Text
  Verifies: a text node is a discrete, independently-mutable DOM node. MDN: "The Text interface represents a text node in a DOM tree." Each text node is its own object, so editing a field's value = mutate the one Text node's data (CharacterData) — the surrounding element tree is untouched. This is the DOM-level justification for live-bind: the bound text node is the smallest possible patch target.

https://developer.mozilla.org/en-US/docs/Web/HTML/Element/input
  Verifies: the per-field-type → form-control mapping. MDN documents `<input type="file">` as the file-upload control (the "image" field's control). Pairing with the MDN <textarea> element page (https://developer.mozilla.org/en-US/docs/Web/HTML/Element/textarea — "the <textarea> HTML element represents a multi-line plain-text editing control"), this confirms the AGENTS.md "Layout field types" mapping: text → <textarea>, voiceover → <textarea> (amber-styled), image → <input type="file">.

(in-repo) docs/rfc-0001.md §7 "Editor Surfaces" — the "Properties / data" row
  Verifies: the Properties panel binds to "slide index.json" and its job is to "edit fields, asset refs, voiceover text, transition override." Also confirms "All surfaces operate on the same workspace files; switching is lossless," and "Undo/redo falls out naturally as file/JSON mutations, not HTML diffs."

(in-repo) docs/rfc-0001.md §5.6 "Data binding (how fields reach the HTML)"
  Verifies: the three stamping modes the Properties panel lives inside — (1) "Stamps on structural change (slide add/remove/reorder, layout swap)"; (2) "Live-binds text fields for responsive editing (textarea -> immediate preview update via a light DOM patch, no full re-stamp)"; (3) "Fully re-stamps at export." Live-bind is the Properties panel's per-keystroke behavior.

(in-repo) docs/rfc-0001.md §5.2 (root transition_default) + §5.3 (slide.transition override)
  Verifies: root index.json carries "transition_default": { "type": "crossfade", "duration": 0.4 }; a slide may optionally carry "transition": { "type": "push", "duration": 0.5 } which overrides the root default for that slide. This is the transition-override UI the panel exposes (absent = use root).

(in-repo) docs/AGENTS.md "Layout field types" + "Common pitfalls" #7
  Verifies: the field-type → form-control table (text → <textarea> + __FIELD_ID__ placeholder replacement; image → <input type="file"> + byte swap to assets/; voiceover → <textarea> amber styled + batch TTS pipeline). Also pitfall #7: "The slide editor intercepts form submit and reads all textarea values from the DOM via data-field attributes. The syncFromDOM() function must run before any re-render (add/remove/reorder) to avoid losing typed values."
