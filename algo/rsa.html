RSA is the one-way padlock: anyone can encrypt with the public key (c = m^e mod n), only you can decrypt with the private key (m = c^d mod n). This page uses the toy keys from rsa.py (p=61, q=53) so every number is printable — the math is identical for 2048-bit keys.  |  SHA-256 feeds RSA padding →

① Key generation — two primes become a public/private pair

What to do: pick two secret primes. The page derives the public key (n,e) and the private key (n,d) live. The secret is d — found via the extended Euclidean algorithm so that e·d ≡ 1 (mod φ).
Derived keys:
PUBLIC (publish) PRIVATE (secret)
The extended Euclid table finds x with e·x + φ·y = 1; then d = x mod φ. Knowing φ is exactly as valuable as knowing p,q — both yield d.

② Encrypt + decrypt — the one-way padlock in motion

What to do: type a message m (must be < n). The public key encrypts it to a scrambled c; the private key decrypts it back. Try encrypting with the private key (that is a signature, see ③).
as ASCII:
m = 65
── encrypt (e) ──▶
c = 2790
── decrypt (d) ──▶
m' = 65
pow(m, e, n) uses square-and-multiply: it is cheap, but finding the e-th root mod n without d means factoring n — believed infeasible for 2048-bit n.

③ Sign + verify — flip the keys for digital signatures

What this shows: signing flips encrypt/decrypt. You sign with the private key (only you can), anyone verifies with the public key. A forged signature recovers the wrong value and is rejected.
Genuine: verify(sign(m)) == m. Forged: a random s' recovers garbage ≠ m → rejected. This is the basis of code signing, TLS certificates, and blockchain transactions.
