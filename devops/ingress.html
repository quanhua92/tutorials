An Ingress is a set of L7 routing rules (host + path → backend Service). The Ingress Controller (nginx/traefik/envoy) reads them and serves traffic. Type a request below and watch the controller walk the 2-level decision tree. ① The routing engine — (host, path) → backend, decided live

What to do: type a host + path. The controller asks 1) does any host rule match? 2) which is the longest matching path prefix? 3) forward to that backend. No host rule → default-backend (404).
Rules live in the Ingress (config). The controller watches them and generates its proxy config — without a controller deployed, Ingress resources do nothing.

③ TLS termination — HTTPS in, HTTP out

What to look at: the controller holds the cert (Secret) and decrypts HTTPS, then talks plain HTTP to the backend. One cert at the edge secures many backends; the backend sees the controller's IP, not the client's (hence X-Forwarded-For).
HTTPS (TLS) controller terminates TLS plain HTTP to backend cert-manager → Secret

④ Annotations — rewrite-target (the #1 footgun)

What this shows: rewrite-target=/ strips the matched prefix before forwarding. Toggle it off and the full path is sent. These keys are nginx-only — not portable across controllers (a core reason Gateway API exists).
