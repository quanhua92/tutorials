# preview_engine — web provenance log
# Every URL has a "Verifies:" line stating exactly what it confirms.
# STEP 2 of HOW_TO_RESEARCH.md is non-skippable; >=2 sources per concept.

https://gsap.com/docs/v3/GSAP/Timeline/time()
  Verifies: GSAP Timeline.time() is the playhead setter — the "drive from the
  playhead" primitive. The page's signature reads "time( value:Number,
  suppressEvents:Boolean ) : [Number | self]" and the Details state it "Gets or
  sets the local position of the playhead (essentially the current time)". The
  page's own code example confirms seeking without playing: "//sets time,
  jumping to new value just like seek()  tl.time(2);". This is exactly what
  preview_engine.ts Section B asserts — the preview calls tl.time(playheadTime)
  on the PAUSED timeline (paused:true per AGENTS.md), moving the playhead without
  auto-playing. Deterministic given t.

https://en.wikipedia.org/wiki/Linear_interpolation
  Verifies: the interpolation math a timeline evaluates between two keyframes.
  The "Linear interpolation between two known points" section derives
  y = y0 + (x - x0) * (y1 - y0) / (x1 - x0). The "Programming language support"
  section gives the canonical computer-graphics "lerp" form (quoting the page):
  "float lerp(float v0, float v1, float t) { return v0 + t * (v1 - v0); }" and
  notes "the basic operation of linear interpolation between two values is
  commonly used in computer graphics ... sometimes called a lerp". Confirms the
  lerp(a,b,t) = a + t*(b-a) identity used in preview_engine.ts Section C, the
  pinned values lerp(0,1,0.5)=0.5 and the multi-segment lerp(1.0,0.5,0.5)=0.75.

https://gsap.com/docs/v3/GSAP/Tween
  Verifies: GSAP's default ease. GSAP tween/timeline tweens default to the ease
  "power1.out" (a quadratic ease-out), NOT linear. This is why preview_engine.ts
  models the LINEAR ("none") easing case for its hand-reproducible math: under
  linear easing the local t passed to lerp is the raw segment fraction, so
  lerp(0,1,0.5) === 0.5 exactly. Any other ease is just a remap of t before the
  lerp; the lerp identity itself is unchanged. Keeps the runnable deterministic
  and the gold-check a mathematical identity rather than a GSAP-version-dependent
  number.

(in-repo) docs/rfc-0001.md §8 "App-Owned Preview Engine"
  Verifies: the load-bearing quotes printed verbatim in Sections B/D/E —
  "Renders the active slide's index.html (the bare <template>) in the browser via
  the SlideRenderer interface"; "Drives GSAP timelines from the playhead time";
  "The preview is app-owned — the editor drives it; HyperFrames is not in the
  editing loop as a black box"; and "Real-time, not frame-accurate. That is the
  point of decoupling: the preview is responsive for editing; the export is
  rigorous (§10)." Primary source for the whole concept.

(in-repo) docs/rfc-0001.md §9.1 "The SlideRenderer Interface"
  Verifies: the EXACT interface quoted verbatim in Section A —
  `load(slide_html, fields, assets)`, `render_at(timecode) -> frame`,
  `duration() -> seconds`. The engine sits behind this interface; preview and
  export share it (§9.4 — fidelity drift "structurally impossible").

(in-repo) docs/rfc-0001.md §9.4–§9.5 "Why this kills fidelity drift" / "make HF smaller"
  Verifies: Section E's app-owned rationale — HF's role contracts to "render this
  one slide at time t" and it becomes a pluggable rendering primitive behind the
  SlideRenderer seam.

(in-repo) docs/rfc-0001.md §11 "Determinism"
  Verifies: why real-time-not-frame-accurate preview is acceptable — "Visual
  determinism, not byte-identical. Same input -> visually equivalent output
  across re-renders." Determinism is enforced at export (§10), NOT in preview.
  Primary source for Section D's decoupling rationale.

(in-repo) docs/rfc-0001.md §5.2 "Root index.json"
  Verifies: the canvas fps the export cadence uses —
  `"canvas": { "width": 1920, "height": 1080, "fps": 30 }`. Section D steps
  export frames at exactly 1/fps = 1/30 = 0.033333s each.

(in-repo) docs/AGENTS.md "Layout file format" — the slide <script>
  Verifies: the timeline-registration line quoted verbatim in Section B — the
  slide creates `var tl = gsap.timeline({ paused: true });` and registers it as
  `window.__timelines['__SLIDE_ID__'] = tl;`. This is the registry the preview
  engine looks up by active slide id to seek the right timeline (Sections B & F).
