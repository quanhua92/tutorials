# visual_determinism — web provenance log
# Every URL has a "Verifies:" line stating exactly what it confirms.
# STEP 2 of HOW_TO_RESEARCH.md is non-skippable; ≥2 sources per concept.

https://en.wikipedia.org/wiki/Hamming_distance
  Verifies: Hamming distance is "the number of positions at which the corresponding
  symbols are different"; for binary strings it equals the population count of
  a XOR b; and the distance between two identical strings is 0. This is the
  metric Section D uses to compare two aHash fingerprints, and the exact basis
  for the cache HIT/MISS threshold in Section E (distance 0 = identical).

https://github.com/JohannesBuchner/imagehash  (imagehash/__init__.py, average_hash)
  Verifies: the aHash (average hash) algorithm recipe implemented in Section C,
  verbatim from the canonical open-source implementation:
    image = image.convert("L").resize((hash_size, hash_size), ANTIALIAS)   # grayscale + 8x8
    avg = mean(pixels)                                                      # mean luminescence
    diff = pixels > avg                                                     # 64-bit hash (strictly >)
  with default hash_size=8 -> 64-bit hash. Also verifies that the library's
  hash subtraction (`hash - otherhash`, documented as "# hamming distance") is
  `numpy.count_nonzero(h1.flatten() != h2.flatten())` — i.e. Hamming distance.
  README also confirms the design intent: "Image hashes tell whether two images
  look nearly identical. This is different from cryptographic hashing algorithms
  (like MD5, SHA-1) where tiny changes in the image give completely different
  hashes." — the locality-sensitive property RFC §11 relies on for cache keying.

https://en.wikipedia.org/wiki/Perceptual_hashing
  Verifies: a perceptual hash "is a type of locality-sensitive hash, which is
  analogous if features of the multimedia are similar. This is in contrast to
  cryptographic hashing, which relies on the avalanche effect of a small change
  in input value creating a drastic change in output value." This is precisely
  the visual-vs-byte distinction Section A draws (RFC §11): perceptual hashing
  tolerates minor byte differences that cryptographic/byte hashing would flag.

(in-repo) docs/rfc-0001.md §11 (Determinism)
  Verifies: "Visual determinism, not byte-identical"; FFmpeg-with-fixed-params is
  near-byte-deterministic while browser capture is the variable part (own-renderer
  only); cache keying + "did this edit change the output" diffs use perceptual
  hashing; and byte-identical CAN be tightened later (pin Chromium, force CPU
  rasterization, full asset preload) without rearchitecting.

(in-repo) docs/rfc-0001.md §3 (Non-Goals)
  Verifies: "Byte-identical determinism (visual determinism suffices — see §11)"
  is an explicit Non-Goal of this RFC.

(in-repo) docs/rfc-0001.md §9.3 + §18
  Verifies: §9.3 — the own-renderer is "Headless-Chromium frame capture + FFmpeg
  encode/mux, targeting visual determinism (§11)" — i.e. the only place browser-
  capture non-determinism enters. §18 — the "Determinism engineering when we ship
  our own renderer" risk row, mitigated by the visual-determinism bar (§11).
