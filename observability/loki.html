Loki — label-indexed log aggregation, chunks on S3 LOKI.md loki.py

"Prometheus-for-logs." Loki indexes ONLY labels (metadata) — never the log content. Log lines batch into compressed chunks on cheap object storage (S3/MinIO); a substring search |= SCANS those chunks at query time. The killer trade vs Elasticsearch: tiny storage + cheap ingest, slow needle-in-haystack (until 3.0 Bloom filters). Recompute the whole pipeline live below.

② Chunk storage — raw logs → compressed chunks → S3

What this shows: raw log bytes batch into ~1.5 MiB chunks, compress (snappy/gzip/zstd), and land on S3. You pay for compressed bytes. Adjust the daily raw ingest + codec and watch the S3 bill move.
chunk_target_size ~1.5 MiB, max_chunk_age ~2h. Index (BoltDB/TSDB) is ~0.001% of chunk bytes — labels only, never content.

③ Label indexing — why high-cardinality labels hurt

What this shows: one label-set = one STREAM. Streams = product of label cardinalities. Each stream opens an in-memory chunk (~64 KiB head) + an index entry (~1.5 KiB). Add a toxic label (trace_id) and the ingester OOMs.
Loki wants FEWER labels than Prometheus (aim 5–10). Keep high-cardinality data IN the log line; filter with |= at query time.
