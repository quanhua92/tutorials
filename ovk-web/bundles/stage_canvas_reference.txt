# stage_canvas — web provenance log
# Every URL has a "Verifies:" line stating exactly what it confirms.
# STEP 2 of HOW_TO_RESEARCH.md is non-skippable; ≥2 sources per concept.

https://en.wikipedia.org/wiki/16:9_aspect_ratio
  Verifies: 1920×1080 is a 16:9 resolution. The "Common Resolutions" table lists "Width 1920 / Height 1080 / Full HD" as a 16:9 entry, and the lede states "16:9 ... is an aspect ratio with a width of 16 units and height of 9 units" (1.77:1). Confirms aspect = width/height = 1920/1080 = 16/9 ≈ 1.777778, the standard 16:9 (Full HD) canvas used by RFC 0001 §5.2 root.canvas and AGENTS.md data-width="1920" data-height="1080".

https://en.wikipedia.org/wiki/1080p
  Verifies: 1080p = "1920 × 1080 progressively displayed pixels" — the Full HD standard. Independent second source that the canvas dims {width:1920, height:1080} are a real, standardized video raster (so scaling them into a viewport while preserving aspect ratio — the stage's core job — is the well-defined "scale-to-fit / contain" problem).

https://developer.mozilla.org/en-US/docs/Web/CSS/object-fit
  Verifies: the scale-to-fit / "contain" math. MDN defines object-fit: contain as "The replaced content is sized to maintain its aspect ratio while fitting within the element's content box." This is exactly `scale = min(vw/cw, vh/ch); scaled dims = (cw*scale, ch*scale)` — the limiting axis wins and the remainder is letterboxed (never stretched). Confirms the scaleToFit() formula used in stage_canvas.ts Section B and the letterbox behavior rendered by stage_canvas.html.

https://developer.mozilla.org/en-US/docs/Web/CSS/transform-function/scale
  Verifies: CSS scale() as a uniform multiplier on both axes. The stage applies a SINGLE uniform scale (min of the two axis ratios) to preserve the 16:9 aspect ratio; a non-uniform scale (different x/y) would stretch the canvas. This is why the scale-to-fit formula takes the min, not a per-axis fit. Confirms the "never stretched" invariant.

(in-repo) docs/rfc-0001.md §7 "Editor Surfaces" — Stage / Canvas row
  Verifies: the stage's binding and job, quoted verbatim: "Stage / Canvas (center) | binds to: active slide | job: live preview via SlideRenderer; play/scrub." Primary in-repo source for Section A (one active slide) and Section D (swap on active-slide change).

(in-repo) docs/rfc-0001.md §8 "App-Owned Preview Engine"
  Verifies: the stage renders "the active slide's index.html (the bare <template>) in the browser via the SlideRenderer interface"; "Drives GSAP timelines from the playhead time"; and the load-bearing decoupling quote "Real-time, not frame-accurate. That is the point of decoupling: the preview is responsive for editing; the export is rigorous (§10)." Primary source for Sections A, C, and E.

(in-repo) docs/rfc-0001.md §9.1 "The SlideRenderer Interface"
  Verifies: the EXACT interface quoted verbatim in Section A — `load(slide_html, fields, assets)`, `render_at(timecode) -> frame`, `duration() -> seconds`. The stage drives the engine through this interface.

(in-repo) docs/rfc-0001.md §5.2 "Root index.json (minimal, project-wide)"
  Verifies: the canvas dims come from root `index.json`: `"canvas": { "width": 1920, "height": 1080, "fps": 30 }`. The stage renders at these dims; the .html gold-check recomputes 1920/1080 === 16/9.

(in-repo) docs/rfc-0001.md §11 "Determinism"
  Verifies: why real-time-not-frame-accurate preview is acceptable — "Visual determinism, not byte-identical ... Same input → visually equivalent output across re-renders." Determinism is enforced at export (§10), NOT in the preview. Primary source for Section E's decoupling rationale.

(in-repo) docs/AGENTS.md "Host div format" + "Why NOT flex/absolute centering"
  Verifies: the host-div rule quoted verbatim in Section F — `style="position:absolute;inset:0;z-index:100;"` with the note "position:absolute;inset:0; is REQUIRED on the host div — without it, sub-comp content collapses to zero size." Also verifies the bare-<template> composition mounted into that div carries `data-width="1920" data-height="1080"` (the canvas dims the stage scales). Primary source for Section F and the canvas-dim assertion in Section B.
