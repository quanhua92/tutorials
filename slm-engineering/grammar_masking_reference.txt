--------------------------------------------------------------------------------
[1] https://www.aidancooper.co.uk/constrained-decoding/
    Aidan Cooper (2024) "A Guide to Structured Generation Using Constrained Decoding"
    Verifies: the core mechanism -- "constrained decoding ... manipulates a
    generative model's token generation process to constrain its next-token
    predictions to only tokens that do not violate the required output structure";
    disallowed tokens are removed before the draw so the model's relative
    preferences WITHIN the allowed set are preserved (the "model does not know the
    constraints in advance" pitfall). Also confirms constrained decoding works on
    LOCAL models only (must expose the full next-token distribution), and that it
    lets "relatively small and inexpensive models perform comparably to much
    larger alternatives" -- the SLM reliability lift this bundle demonstrates.

[2] https://lmsys.org/blog/2024-02-05-compressed-fsm/
    Yin, Sheng, Zheng (LMSYS/SGLang, 2024) "Fast JSON Decoding for Local LLMs with
    Compressed Finite State Machine"
    Verifies: the FSM-from-regex pipeline -- "transforming the JSON schema into a
    regular expression. We can then construct a Finite State Machine(FSM) based on
    the regular expression ... For every state within the FSM, we can calculate the
    permissible transitions and identify the acceptable next tokens. This allows us
    to track the current state during decoding and filter out invalid tokens by
    applying logit bias to the output." (Section "Method 1: Finite State Machine
    Based"). This is exactly Section A's DFA + Section B's logit mask. Also
    documents the tokenization-boundary pitfall (multi-char tokens) that motivates
    llama.cpp's full-token mask.

[3] https://arxiv.org/abs/2307.09702
    Willard, B. T.; Louf, R. (2023) "Efficient Guided Generation for Large Language
    Models" (the Outlines paper, arXiv:2307.09702)
    Verifies: the formal basis for regex/FSM-guided decoding -- reformulates neural
    text generation as transitions over a finite-state machine compiled from a
    regular expression; at each step the set of admissible tokens is computed from
    the current FSM state and the disallowed tokens are masked. Cross-checked
    against the HuggingFace papers page (https://huggingface.co/papers/2307.09702)
    and Semantic Scholar (377+ citations). This is the "regex-guided decoding"
    stage of the lineage.

[4] https://github.com/ggml-org/llama.cpp/blob/master/grammars/README.md
    llama.cpp project -- "GBNF Guide" (GGML BNF grammars, official docs)
    Verifies: GBNF (GGML Backus-Naur Form) -- a BNF-style grammar with regex sugar
    (`[a-z]`, `+ * ?`, `|`) that expresses JSON schemas; the sampler constrains
    generation by masking tokens the grammar rejects before sampling. Documents
    `--grammar-file` / `--grammar` / `-j` (json_schema) CLI flags and the server's
    `grammar` / `json_schema` / `response_format` fields. This is the "GBNF /
    CFG-guided" stage of the lineage (the production reference this bundle
    simulates).

[5] https://github.com/ggml-org/llama.cpp/blob/master/common/grammar-parser.h
    llama.cpp project -- `common/grammar-parser.h` + `llama-grammar.cpp` (source)
    Verifies: the production implementation is a byte-level LL(1) parser with a
    stack of rule positions; at each decode step it tests EVERY vocabulary token
    against the current grammar state (the whole token text, not just the first
    byte) and masks the rejected ones to -inf before softmax. Confirms the
    "full-token mask" (vs the broken first-byte mask) and the O(vocab x steps)
    cost.

[6] https://github.com/ggml-org/llama.cpp/issues/4218
    llama.cpp issue #4218 -- "grammar performance" (the canonical perf thread)
    Verifies: the ~10-30% decode-time overhead of grammar-constrained sampling
    ("every token in the vocabulary is checked against the grammar state at every
    step"); documents the `x? x? x? ...` exponential-blowup pathology and the
    `x{0,N}` rewrite fix. Cited verbatim by ../local-llm/GRAMMAR_OUTPUT.md.

[7] https://github.com/dottxt-ai/outlines
    dottxt-ai/outlines -- "Structured generation for type-safe LLMs" (library)
    Verifies: the library form of the FSM approach -- "ensures outputs conform to
    specified formats (JSON schemas, regex patterns, grammars)" by a logit
    processor that "sets the probability of illegal tokens" to zero at every step
    (cross-checked against the AWS Outlines blog,
    https://aws.amazon.com/blogs/machine-learning/generate-structured-output-from-llms-with-dottxt-outlines-in-aws/ ).
    Independent of llama.cpp; same -inf-before-softmax mechanism over a BPE vocab.

[8] ./GRAMMAR_OUTPUT.md  (in-repo: /Volumes/data/workspace/tutorials/local-llm/GRAMMAR_OUTPUT.md)
    This repo's ../local-llm/GRAMMAR_OUTPUT.md -- the production grammar-output
    reference this bundle simulates at toy scale.
    Verifies: the lineage table (free sampling -> first-byte mask -> FULL TOKEN
    mask / GBNF -> JSON-Schema->GBNF compiler), the ~10-30% cost figure, the
    worked mask table (state 0 valid = ['{']; state 1 valid = in-class letters),
    and the pitfalls (first-byte-only mask, whitespace handling, token boundary
    mid-rule). The sibling this bundle cross-references.

[9] ../llm/SAMPLING.md  (in-repo: /Volumes/data/workspace/tutorials/llm/SAMPLING.md)
    This repo's ../llm/SAMPLING.md -- the logits -> softmax -> sample pipeline.
    Verifies: the `-inf` masking idiom (top-k/top-p set disallowed logits to `-inf`
    so their softmax probability is exactly 0 and they are never drawn), the
    numerically-stable `log_softmax(z) = z - logsumexp(z)`, and the
    filter-then-draw ordering. Grammar masking slots INTO this exact pipeline as
    one more filter (the grammar filter) applied before the draw.
