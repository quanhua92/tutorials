BlockManager — Paged Alloc + Prefix Cache — interactive BLOCK_MANAGER.md block_manager.py

The library-card idea: each page is fingerprinted by a chained hash — its own tokens folded onto the hash of the page before it. Two requests that share a token prefix get identical fingerprints and share the same physical page (ref_count++), skipping the KV recompute. block_size = 2. A = [10,11, 12,13, 16,17] (3 blocks). B copies A's first N blocks, then diverges.
green = shared (identical hash, ref_count 2) blue = A's divergent block purple = B's divergent block
What to do: drag N. At N=1, block0 hashes match exactly (the shared system-prompt page) and block1 diverges. Notice the shared hash is identical even though the block-1 hashes differ — that is the whole reason prefix sharing works. The hash under each green node is the gold value recomputed from block_manager.py's FNV-1a.

Manager state
Caption: Allocate A fills blocks 0,1 and registers their hashes. Allocate B finds block0's hash in the catalog → reuses block 0 (ref 1→2, no recompute) and grabs a fresh block 2 for B's divergent tokens. Finish A deallocates: block1 frees (ref 0) but block0 only ticks to ref 1 (B still reads it) — shared blocks survive preemption. See BLOCK_MANAGER.md §7.

③ can_allocate walk for B — cache hit (block0) then miss (block1)

What this shows: when B arrives (after A was allocated+hashed), can_allocate chains B's block hashes and asks the catalog "do I have this fingerprint AND do the tokens match?". Block0 → HIT (reuse, ref++). Block1 → MISS (divergent tokens). Returns num_cached_blocks = 1 → B skips prefill for 2 tokens. (Allocate A and B in panel ② first.)
Caption: the green HIT row is the prefix-cache payoff. The red MISS is the divergence — the walk stops at the first miss. If no catalog match existed, num_cached_blocks=0; if free blocks ran out, it would return -1 (OOM → preempt).
