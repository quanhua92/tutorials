==========================================================================
  SECTION E: flat vs HNSW -- why brute force wins at small N on edge
==========================================================================
| index          | query cost       | build cost          | space          |
|----------------|------------------|---------------------|----------------|
| Flat (brute-force) | O(N*d)           | O(0)                | N*d*4          |
| HNSW (graph ANN) | O(log N * d)     | O(N log N * d)      | N*d*4 + M*N*4  |
| IVF-PQ (quantized) | O(nprobe * d/m)  | O(N * nlist * d)    | ~N*m           |

Concrete at the edge regime (d=384, single-threaded, cosine in fp32):
| N        | flat query work (N*d MACs) | flat memory (MB) |
|----------|----------------------------|------------------|
|     1000 |                    384,000 |             1.54 |
|    10000 |                  3,840,000 |            15.36 |
|   100000 |                 38,400,000 |           153.60 |

A modern phone NPU does >1 GMAC/s, so even N=100k (38.4M MACs) is
<40 ms. HNSW's log-N win only matters at N > ~1M (server scale);
below that its BUILD cost + graph memory are pure overhead.

DECISION RULE (mirrors ../vector-db/VECTOR_DATABASES.md):
  N < 100k, on-device, recall must be perfect  -> FLAT (this bundle).
  100k..10M, real-time updates                 -> HNSW.
  10M+, memory-tight, batch-indexed            -> IVF-PQ.

  [check] HNSW adds M*N*4 bytes of graph edges on top of the matrix: OK

==========================================================================
  SECTION F: lineage recap -- server RAG -> edge RAG (RAG-Slim)
==========================================================================
| component | SERVER RAG (fat)               | EDGE RAG / RAG-Slim             |
|-----------|--------------------------------|---------------------------------|
| embedder  | 1024-dim on GPU (BGE-large)    | 384-dim on CPU/NPU (MiniLM ~22M / BGE-small) |
| index     | HNSW graph (M=16) + reranker   | FLAT cosine matrix (N*d*4 bytes, no build) |
| context   | top-5..10 chunks, ~2-4k tokens | top-1..3 chunks, ~512 tokens    |
| runtime   | datacenter GPU + tens of GB RAM | phone NPU + hundreds of MB RAM  |
| recall    | highest (big embedder + reranker) | traded for footprint (tiny embedder, no rerank) |

Each row is a deliberate DOWNSIZE that keeps the pipeline shape
(embed -> index -> query -> top-k -> inject -> generate) identical
while cutting footprint by 10-100x. The trade is recall: a 384-dim
embedder has lower top-k recall than a 1024-dim one + cross-encoder
reranker -- but it fits on a phone where the fat stack would not boot.

  [check] every slim component is strictly smaller than its server counterpart: OK
