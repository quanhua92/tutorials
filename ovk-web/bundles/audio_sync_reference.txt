# audio_sync — web provenance log
# Every URL has a "Verifies:" line stating exactly what it confirms.
# STEP 2 of HOW_TO_RESEARCH.md is non-skippable; ≥2 sources per concept.

https://developer.mozilla.org/en-US/docs/Web/API/HTMLMediaElement/currentTime
  Verifies: the sync primitive — HTMLMediaElement.currentTime. "A double-precision floating-point value ... the current playback time in seconds." Setting it seeks the media: "If the media is already playing, the playback position is moved to the new position." This is what the timeupdate handler WRITES to each external <audio> element (ext_music.currentTime = ...; ext_voiceover.currentTime = ...).

https://developer.mozilla.org/en-US/docs/Web/API/HTMLMediaElement/timeupdate_event
  Verifies: the sync TRIGGER. "The timeupdate event is fired when the time indicated by the currentTime attribute has been updated." Frequency "between about 4Hz and 66Hz". Confirms AGENTS.md's claim that "Both sync to _player.currentTime via timeupdate events" — i.e. the editor listens for the player's timeupdate and re-seeks the external <audio> elements each tick.

https://developer.mozilla.org/en-US/docs/Web/API/HTMLMediaElement/loop
  Verifies: the loop attribute semantics. "The HTMLMediaElement.loop property reflects the loop HTML attribute, which controls whether the media element should start over when it reaches the end." Boolean. Confirms ext-music (loop=true) restarts at 0 at end while ext-voiceover (loop=false) does not — and therefore that the modulo math (t % musicDuration) for the looped bed is required: without manual modulo on currentTime, a looped short bed would drift out of phase with the playhead across multiple loops.

https://developer.mozilla.org/en-US/docs/Web/API/HTMLMediaElement
  Verifies: the interface both external elements implement — HTMLMediaElement (inherited by HTMLAudioElement / <audio>). Confirms the two external <audio> elements (ext-music, ext-voiceover) expose the SAME currentTime/volume/loop properties the handler reads and writes, and that they are distinct media elements the player cannot internally mute (AGENTS.md: "bypasses the player's internal muting").

https://developer.mozilla.org/en-US/docs/Web/API/HTMLAudioElement
  Verifies: that <audio> elements are HTMLMediaElements usable purely as off-screen audio nodes. Confirms the preview can spawn <audio id="ext-music"> / <audio id="ext-voiceover"> that are NOT wired through the video player's own audio pipeline — they are independently controllable media elements (volume, loop, currentTime), which is the whole point of the bypass.

(in-repo) docs/AGENTS.md "Preview audio"
  Verifies: the EXACT pattern quoted verbatim in the runnable. "The preview page (render_player_page()) uses external <audio> elements synced to the HyperFrames player via event listeners. This bypasses the player's internal muting. — ext-music: music-bed, looped, volume 0.08 — ext-voiceover: voiceover track (if assets/voiceover.mp3 exists), volume 1.0, not looped — Both sync to _player.currentTime via timeupdate events." Primary in-repo source for track config + the bypass rationale.

(in-repo) docs/rfc-0001.md §8 "App-Owned Preview Engine"
  Verifies: the RFC sentence that elevates AGENTS.md's pattern to architecture. "Syncs audio via the external-<audio> pattern documented in AGENTS.md ('Preview audio')." Also §8 confirms the preview is playhead-driven ("Drives GSAP timelines from the playhead time") — i.e. _player.currentTime is the single source of truth both ext-audio elements follow.

(in-repo) docs/rfc-0001.md §5.2 "Root index.json"
  Verifies: where the audio refs live — the root index.json `audio` object: `"music": { "asset": "sha256:...", "volume": 0.08, "loop": true }`, `"voiceover": { "asset": "voiceover.mp3", "auto_generated": true }`. Confirms the editor edits audio.music.asset / audio.voiceover.asset here, and render_player_page() reads them to wire <audio src> for preview. Cross-ref ROOT_INDEX_JSON bundle.
