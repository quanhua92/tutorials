① LoRA forward: y = W₀x + (α/r)·B(Ax) — frozen base + a tiny rank-r side-path

Two paths, one sum. The input x flows through the frozen W₀ (blue, never trained) AND a thin (α/r)·B(Ax) side-path (orange, the only thing trained). Their sum (green) is the output. At init B=0 so the orange path is zero — the model is identical to the base until training moves B.
d = 8, k = 8, r = 2, α = 16 → scale α/r = 8. Fixed x, W₀, A, trained B (all identical to peft_lora.py Section B).
y[0] is the pinned gold value — the footer re-checks it equals 0.1808 from peft_lora.py. The orange adapter column is the entire per-task "learned" signal: ~0.02 on top of a ~0.2 base.

② Rank r → trainable-parameter savings (full FT d·k vs LoRA r·(d+k))

Drag r, watch LoRA win. Full fine-tuning trains d·k weights; LoRA trains only r·(d+k). The savings ratio is ≈ r/min(d,k). Tiny r → tiny adapter, but too tiny under-fits hard tasks (LoRA recovers full FT as r → rank(W₀)).
full FT (blue) vs LoRA (orange) trainable params — log scale

③ Multi-adapter serving — Punica grouped GEMM (one kernel, many adapters)

One base, many adapters, one batch. Each token in a batch picks its own adapter (A_i, B_i). A naive loop runs one matmul per adapter; Punica's grouped-GEMM kernel runs them all in one pass. Both give identical numbers — only the GPU utilisation differs.
S-LoRA extends this: thousands of adapters live in paged memory (like PagedAttention for KV cache 🔗), paged in sub-millisecond. One frozen base serves all of them — paid once.

④ QLoRA memory — NF4 frozen base (4×) + tiny fp16 adapters 🔗

Crush the frozen base, keep the adapters precise. The base W₀ is frozen anyway, so QLoRA stores it as 4-bit NF4 (4× smaller); the tiny adapters A,B stay fp16 (the only things back-prop touches). Bars are the frozen-base footprint of one [896,896] Qwen3-0.5B layer.
NF4 = 16 levels on the quantiles of a normal (dense near 0), the normal-quantile cousin of W4A16's uniform levels. QLoRA paper: a 65B model trains on one 48 GB GPU.
