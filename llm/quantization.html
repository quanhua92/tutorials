① Quantize 8 floats → int4 (0..15) with one per-group scale & bias

8 weights → 16-level ruler. Type 8 weight values. The quantizer finds ONE ruler (a scale step) + ONE offset (a bias start point) for the whole group, then snaps each weight to its nearest of 16 marks (0..15). The orange element is the edge (the heaviest extreme) — it lands dead on a mark with zero error.
group_size = 8 (real models use 128), bits = 4 → 16 levels. The orange element is the edge (heavier extreme) — it maps to an int4 level with zero error by MLX's scale = edge/q0 trick.

② Pack 8 nibbles into ONE uint32 — bit layout (LSB = w_int[0])

8 nibbles → 1 box. Each int4 fits in a nibble (4 bits = half a byte). Eight nibbles squeeze into one 32-bit box (uint32). Read the green hex right-to-left to recover the weights in order — that is what the chip stores on disk.
packed = Σ w_int[k] << (4*k); bits[4k:4k+4] = w_int[k]. Read the hex right-to-left to recover the nibbles. Unpack reverses it: nibble_k = (packed >> 4*k) & 0xF.

③ Quantized matmul vs float matmul — dequant fused inside the MAC loop

Same answer, 4× less memory traffic. The weights arrive packed. The kernel unpacks each box, rebuilds the weight (scale×w_int + bias), and multiplies it by the input on the fly — it never builds the full float matrix. Compare the green (quantized) column to the float column: they match to ~2.6%.
x = [0.5, −0.3, 0.8, −0.1, 0.4, −0.6, 0.2, 0.7] (activation, float16). W is [4, 8]: row 0 = the 8 values above; rows 1–3 are fixed. Each row quantized independently.
The kernel pulls a uint32, unpacks 8 nibbles, dequantizes (scale*w_int + bias), and immediately multiplies by the activation — it never builds the full float weight matrix. That fusion is what makes W4A16 a bandwidth win.
