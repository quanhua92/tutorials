What to do: the two grids below are the on-disk bitmaps. Each row is a tuple; each column is one distinct value. Click a cell to set that row to that value — the row's single 1 slides to the clicked column (the bitmaps partition the rows, so exactly one bit per row is on). Index size (uncompressed) = D × N / 8. For status: 4 × 16 / 8 = 8 bytes. Each column's bits are a partition of the 16 rows — every row is on in exactly one value's bitmap.

③ RLE compression — sparse bitmaps shrink, periodic ones don't

What to do: toggle the source bitmap and the compressed/uncompressed view. A sparse bitmap (long 0-runs) shrinks dramatically; a regular 1-in-8 pattern has many short runs and actually grows.
128-bit bitmap, uncompressed = 16 bytes
Naive RLE wins when runs are LONG. Production systems (BBC, Roaring) are adaptive: each 65536-row chunk picks whichever of {array | bitmap | RLE} is smallest, so neither sparse nor dense regions are punished.

④ Cardinality — when bitmap wins / loses vs B-tree (N = 1,000,000)

What to do: drag D (distinct values). Bitmap size = D × N / 8 grows linearly in D; a B-tree stays ~O(N). At D ≈ N (a unique key) the bitmap explodes to N²/8.
N = 1,000,000 rows · B-tree ≈ 16 B/entry
