Hardware Landscape — capacity fits it, bandwidth decodes it 📖 HARDWARE_LANDSCAPE.md 📄 hardware_landscape.py

Will it fit? is answered by memory capacity (GB). How fast is decode? is answered by memory bandwidth (GB/s). They are different specs — no single box wins both. Decode is bandwidth-bound (tps ~= bw / bytes_per_token), so doubling GB/s doubles tok/s; prefill is compute-bound (FLOPS). Pick a model + quant below and watch which hardware can run it and at what speed. The gold-check pins the GPT-OSS 120B bandwidth-bound diagnostic to the exact .py output.

③ Bandwidth insight — the GPT-OSS 120B proof

What this shows: the canonical 2026 benchmark (GPT-OSS 120B MXFP4). The key column is GB/token = bandwidth / decode_tps. If decode is bandwidth-bound, it is ~constant across very different hardware — and it is (~7–8 GB/token for CUDA/ROCm). That constancy is the bandwidth-bound signature.
Source: hardware-corner.net + aimultiple.com/dgx-spark-alternatives. DGX Spark (1 PFLOP FP4) wins prefill (compute-bound); 3×RTX 3090 (936 GB/s) wins decode (bandwidth-bound). Mac Studio reads ~11.6 GB/token (Metal/MLX overhead) vs ~7.5 for CUDA.
