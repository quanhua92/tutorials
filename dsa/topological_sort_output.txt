========================================================================
  SECTION A: DFS topological sort (post-order reversal)
========================================================================
THE ALGORITHM in one line: run DFS, push each node onto a stack
when you FINISH it (post-order), then REVERSE the stack.

WHY it works: DFS finishes a node only after finishing all nodes
that depend on it (its descendants). So 'finished last' = 'has no
unmet prerequisites' = should come FIRST. Reversing the finish
order gives a valid topological order.

Worked DAG:
Nodes: [0, 1, 2, 3, 4, 5]   (V = 6)
Edges (6):
    0 -> 1    (0 must come before 1)
    0 -> 2    (0 must come before 2)
    1 -> 3    (1 must come before 3)
    2 -> 3    (2 must come before 3)
    3 -> 4    (3 must come before 4)
    4 -> 5    (4 must come before 5)
Adjacency list (sorted for determinism):
    adj[0] = [1, 2]
    adj[1] = [3]
    adj[2] = [3]
    adj[3] = [4]
    adj[4] = [5]
    adj[5] = []

DFS trace (events in order, outer loop over nodes 0..5):
    node 0: enter
    node 1: enter
    node 3: enter
    node 4: enter
    node 5: enter
    node 5: finish -> push
    node 4: finish -> push
    node 3: finish -> push
    node 1: finish -> push
    node 2: enter
    node 2: finish -> push
    node 0: finish -> push

Stack after all finishes (top = last pushed):  [5, 4, 3, 1, 2, 0]
Reversed  -> topological order:                 [0, 2, 1, 3, 4, 5]

========================================================================
  SECTION B: Kahn's algorithm (BFS via in-degree = 0)
========================================================================
THE ALGORITHM in one line: repeatedly pluck out a node whose
in-degree is 0 (no unmet prerequisites), 'remove' it by
decrementing each neighbour's in-degree, and repeat.

WHY it works: an in-degree-0 node has ALL prerequisites met, so it
is safe to place next. Removing it mirrors 'task is done, its
dependents lose one prerequisite.' A queue of ready nodes drives a
Breadth-First layer-by-layer peeling of the DAG.

Step 1 - compute in-degrees:
  | node | in-degree | incoming edges        |
  |------|-----------|-----------------------|
  | 0    | 0         | []                     |
  | 1    | 1         | ['0']                  |
  | 2    | 1         | ['0']                  |
  | 3    | 2         | ['1', '2']             |
  | 4    | 1         | ['3']                  |
  | 5    | 1         | ['4']                  |

Initial queue (in-degree == 0): [0]

Step 2 - repeatedly pop a 0-in-degree node, output it, decrement
its neighbours (the queue drives the BFS):

    pop 0, output it, decrement [1, 2]  -> [1, 2] now ready
    pop 1, output it, decrement [3]
    pop 2, output it, decrement [3]  -> [3] now ready
    pop 3, output it, decrement [4]  -> [4] now ready
    pop 4, output it, decrement [5]  -> [5] now ready
    pop 5, output it, decrement []

Kahn's topological order: [0, 1, 2, 3, 4, 5]

========================================================================
  SECTION C: cycle detection - if |output| < V, a cycle exists
========================================================================
A topological order EXISTS iff the graph is a DAG. If we add a
back-edge that closes a directed cycle, neither algorithm can place
all V nodes: DFS never 'finishes' the cycle, and Kahn's queue
empties before everything is peeled. So: |output| < V  <=>  cycle.

Cyclic graph = DAG + back-edge 5 -> 2 (closes 2->3->4->5->2):
Nodes: [0, 1, 2, 3, 4, 5]   (V = 6)
Edges (7):
    0 -> 1    (0 must come before 1)
    0 -> 2    (0 must come before 2)
    1 -> 3    (1 must come before 3)
    2 -> 3    (2 must come before 3)
    3 -> 4    (3 must come before 4)
    4 -> 5    (4 must come before 5)
    5 -> 2    (5 must come before 2)
Adjacency list (sorted for determinism):
    adj[0] = [1, 2]
    adj[1] = [3]
    adj[2] = [3]
    adj[3] = [4]
    adj[4] = [5]
    adj[5] = [2]

Kahn on the cyclic graph:
    pop 0, output it, decrement [1, 2]
    pop 1, output it, decrement [3]

Kahn output = [0, 1]   (length 2 < V = 6)
Remaining nodes (stuck in the cycle, never reach in-degree 0):
    [2, 3, 4, 5]

========================================================================
  SECTION D: DFS vs Kahn - same goal, different mechanics
========================================================================
Both produce a VALID topological order, but via opposite mechanics:

| aspect        | DFS topo sort                | Kahn's algorithm            |
|---------------|------------------------------|------------------------------|
| core idea     | finish-time stack reversal   | peel in-degree-0 nodes      |
| traversal     | DFS (recursive)              | BFS (queue, iterative)      |
| key structure | recursion stack + result stk | in-degree[] + FIFO queue    |
| cycle detect  | GRAY back-edge in recursion  | output length < V           |
| stack depth   | O(V) recursion (watch Python | O(1) - no recursion         |
|               |   recursion limit on big V)  |                              |
| order flavour | tends to go DEEP first        | tends to go BREADTH first   |
| parallelism   | hard (single DFS path)       | NATURAL: all 0-indeg nodes  |
|               |                              | are independent -> run in   |
|               |                              | parallel (build systems!)   |

DFS  order: [0, 2, 1, 3, 4, 5]
Kahn order: [0, 1, 2, 3, 4, 5]

Note both are valid (all edges go forward) yet DIFFERENT: a DAG can
have many valid topological orders. DFS dives deep (0 -> 1 -> 3 -> 4
-> 5) before backtracking to 2; Kahn peels layer by layer (0, then 1
and 2 become ready, then 3, ...).

========================================================================
  SECTION E: applications + GOLD CHECK
========================================================================
Topological sort is the engine behind every 'do things in dependency
order' system:

  - Build systems (Make, Gradle, Cargo, Bazel, npm): compile targets
    in dependency order; Kahn's independent 0-in-degree set is the
    basis for PARALLEL builds.
  - Course prerequisites: a valid semester-by-semester plan is a
    topological order of courses (edge: 'X is a prereq of Y').
  - Package managers (pip, apt, brew): resolve install order; a
    dependency CYCLE is a hard error (|output| < V).
  - Task scheduling / dataflow (Airflow, Spark, spreadsheets):
    compute stages once all inputs are ready.
  - Circuit layout / Verilog: signal propagation order.
  - Git commit history (DAG of commits): linearised history.

Worked example - course prerequisites (6 courses, diamond at 1,2->3):
  prereqs: 0->{1,2}, 1->3, 2->3, 3->4, 4->5
