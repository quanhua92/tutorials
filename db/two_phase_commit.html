Two-Phase Commit (2PC) — interactive 📖 TWO_PHASE_COMMIT.md 📄 two_phase_commit.py

📖 Read the full guide — architecture diagrams, decision tables, killer gotchas, cheat sheet. This page is the interactive companion.
One coordinator drives a strict two-step handshake with N participants: phase 1 PREPARE (everyone fsyncs a durable promise and votes YES/NO) → phase 2 COMMIT/ROLLBACK (the global decision). ② Crash lab — click a node to crash it, watch blocking & recovery

What this proves: crash the coordinator after PREPARE and every prepared participant blocks (locks held, can't decide) until recovery. Crash a participant and it runs a recovery transaction (asks the coordinator). Toggle coordinator recovers to resolve blocking. The engine is the same as the .py.
The gold invariant of 2PC: after any crash/recovery, ALL participants end in the same final state (all committed XOR all aborted) — verified live below.

③ Latency & message cost — slide N and watch the critical path

Participant fsyncs run in parallel, so critical-path latency does not grow with N; what grows is the message count (4N) and the total fsync work (2N+1). Matches two_phase_commit.py Section F.
Formula: latency = 2·RTT + 3·fsync (3 serialized fsyncs: prepare-phase, coordinator decision, commit-phase).  messages = 4·N  |  total fsync ops = 2·N+1 (throughput load).
