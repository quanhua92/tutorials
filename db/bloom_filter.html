Bloom Filter — interactive 📖 BLOOM_FILTER.md 📄 bloom_filter.py

📖 Read the full guide — architecture diagrams, decision tables, killer gotchas, cheat sheet. This page is the interactive companion.
A Bloom filter answers "might x be in the set?" with m bits and k hashes, storing no keys. Insert sets k bits; query checks them — all 1 → "maybe", any 0 → "definitely not". No false negatives, ever; false positives bounded by (1 - e^(-kn/m))^k. The worked example recomputes live from the identical seeded hashes as bloom_filter.py.  |  RocksDB uses one per SST file →

① Insert — watch k bits flip into the m-bit array

What to do: click a key (or type your own + Insert). Each insert computes k = 3 positions via mix32(fnv1a(key||byte(i))) % m and sets those bits. Shared bits = future false positives.
keys:
set (1) empty (0) just flipped this step
empty filter: m = 32 bits, fill 0/32 (0.000)

② Query — check all k bits. All set = "maybe"; any 0 = "definitely not"

What to do: type a key and Query. The k positions are highlighted (blue = being checked). Try a real member (cherry) and a non-member (mango).
A true member's bits were all set by its own insert, so "maybe" is correct. A non-member sharing all k bits with others would be a false positive; a single 0-bit proves absence.

③ The false-positive dial — slide m, k, n and watch the FPR formula

What this shows: FPR = (1 - e^(-kn/m))^k live. Increase m (more room → lower FPR); tune k toward k* = (m/n)·ln2 (the green marker). Defaults match bloom_filter.py Section C.

④ False-positive simulation — measured vs the formula

What to do: builds a fresh filter with the m, k, n above, inserts n members, then probes 2000 guaranteed non-members. The measured bar should land near the theoretical bar — the residual is finite-sample noise. (Deterministic: seeded non-members.)
