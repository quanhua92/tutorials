========================================================================
  SECTION A: the algorithm  (array = [64,25,12,22,11])
========================================================================
Input  : [64, 25, 12, 22, 11]
Sorted : [11, 12, 22, 25, 64]
n = 5  ->  divide log2(5) = 2.32 -> 3 levels of recursion, 4 merges.

Two phases repeat at every node of the recursion tree:
  DIVIDE : cut arr[lo..hi] at mid=(lo+hi)//2 -> recurse on each half.
  MERGE  : after both halves return sorted, merge them in O(n).

Base case: a range of length <= 1 is already sorted (nothing to do).
All the real work is in the MERGE step, never in the divide.

========================================================================
  SECTION B: the divide tree  (how the array gets chopped up)
========================================================================
Every node is a recursive call. Leaves (length 1) are sorted.

depth 0: [64, 25, 12, 22, 11] [0..4]   
depth 1:   [64, 25, 12] [0..2]   [22, 11] [3..4]   
depth 2:     [64, 25] [0..1]   [12] leaf   [22] leaf   [11] leaf   
depth 3:       [64] leaf   [25] leaf   

Read top-down: the whole array splits into [64,25,12] and [22,11],
each of those splits again, until every piece is a single element.
The number of splits along the longest root-to-leaf path is the
tree HEIGHT = 3 = ceil(log2 5) = ceil(2.32).
[check] height == ceil(log2 n):  OK

n = 5 leaves (one per element). An internal node has exactly
2 children, so #merges = #internal nodes = 4.

========================================================================
  SECTION C: the merge steps  (each merge is O(n), take-left on tie)
========================================================================
Merges run bottom-up (deepest first), as each pair of children returns.
Rule: compare the two fronts; take the SMALLER; on a TIE take LEFT
(-> stable). When one pile empties, copy the rest of the other.

-- merge 0: [0..0] + [1..1] --
   left  : [64]
   right : [25]
   result: [25, 64]   (1 comparisons)

-- merge 1: [0..1] + [2..2] --
   left  : [25, 64]
   right : [12]
   result: [12, 25, 64]   (1 comparisons)

-- merge 2: [3..3] + [4..4] --
   left  : [22]
   right : [11]
   result: [11, 22]   (1 comparisons)

-- merge 3: [0..2] + [3..4] --
   left  : [12, 25, 64]
   right : [11, 22]
   result: [11, 12, 22, 25, 64]   (3 comparisons)

Total comparisons across all 4 merges: 6
Upper bound n * ceil(log2 n) = 5 * 3 = 15.
Measured 6 <= 15: OK

Each LEVEL of the tree touches every element exactly once during its
merges -> O(n) work per level. With ceil(log2 n) levels, total work
= O(n log n). That is the Master theorem on T(n) = 2*T(n/2) + Theta(n).

========================================================================
  SECTION D: complexity analysis  (BEST = AVERAGE = WORST = Theta(n log n))
========================================================================
Merge sort ALWAYS divides in half and ALWAYS merges linearly. The
input's order only changes HOW MANY comparisons each merge makes
(best case ~ n/2 per level if already sorted, worst case ~ n per
level), NOT the number of levels. So there is no degenerate input.

  case                         comparisons  merges   sorted?
  best    (already sorted)     7            4        True
  average (random)             6            4        True
  worst   (reverse sorted)     5            4        True

Comparison count varies only by a constant factor (~2x) across
cases; the n log n SHAPE is identical. n log2 n for n=5 = 11.6.

Summary table:
  | case    | comparisons        | time         | space     | stable |
  |---------|--------------------|--------------|-----------|--------|
  | best    | ~(n/2) log n       | Theta(n log n)| Theta(n) | yes    |
  | average | ~ n log n          | Theta(n log n)| Theta(n) | yes    |
  | worst   | ~ n log n          | Theta(n log n)| Theta(n) | yes    |

Space: Theta(n) auxiliary for the merge buffer (NOT in-place).
Contrast selection_sort (🔗 SELECTION_SORT.md Section C): Theta(n^2)
comparisons in EVERY case, but O(1) auxiliary space, in-place.

========================================================================
  SECTION E: stability  (YES -- the take-from-left rule preserves order)
========================================================================
A sort is STABLE if equal keys keep their original relative order.

Merge sort is stable BY CONSTRUCTION: the merge rule says 'on a tie,
take from the LEFT half first'. Since the left half held the elements
that came EARLIER in the input, ties always resolve in input order.

The ONLY way to break this is to write `<` instead of `<=` in the
merge (taking right on a tie). The implementation above uses `<=`.

Demonstration: tag two equal keys with their original index.

Input (tag, value): [('2a', 2), ('1', 1), ('2b', 2), ('3', 3)]
Output             : [('1', 1), ('2a', 2), ('2b', 2), ('3', 3)]

The two 2's ended up as ['2a', '2b']; a stable sort keeps ['2a', '2b'].
[check] stable?  True  ->  merge sort IS stable

Compare with selection_sort (🔗 SELECTION_SORT.md Section D): the
swap leapfrogs equal keys -> NOT stable. Merge sort pays O(n) space
and GETS stability for free.

========================================================================
  SECTION F: when to use merge sort  (and where it hides in real systems)
========================================================================
USE merge sort when:
  * You need GUARANTEED O(n log n) with no bad input. Quicksort's
    worst case is Theta(n^2); merge sort's never is. Mission-critical
    / real-time systems often prefer the predictability.
  * You need STABILITY. It is the canonical stable O(n log n) sort.
  * Sorting LINKED LISTS: merge is O(1) extra space on linked lists
    (relink pointers, no buffer), and divide is pointer-based. The
    one place merge sort beats quicksort on space.
  * EXTERNAL sorting (data > RAM): write sorted runs to disk, then
    k-way merge. Merge sort's sequential access pattern is ideal for
    slow spinning/sequential media; random-access sorts thrash.

DO NOT use plain merge sort when:
  * You are tight on MEMORY: it needs Theta(n) auxiliary space. Use
    heapsort (in-place, O(1)) or an in-place merge variant.
  * You want cache-friendly in-place sorting on arrays: quicksort's
    partitioning is in-place and cache-friendlier; merge sort's
    buffer copy hurts. (This is why libc qsort is usually quicksort.)
  * n is SMALL (< ~64): insertion sort's lower overhead wins. This is
    exactly the hybrid trick TimSort uses.

WHERE IT HIDES:
  * Python list.sort / sorted()  -> TimSort (merge sort + run detect).
  * Java Arrays.sort(Object[])   -> TimSort since Java 7.
  * Java Arrays.sort(int[])      -> dual-pivot quicksort (NOT merge;
    chose speed + in-place over stability for primitives).
  * V8 / libstdc++ std::stable_sort -> merge sort (stability required).

See SELECTION_SORT.md for the Theta(n^2) minimum-swap foil.
