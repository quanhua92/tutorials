MLX Inference — Apple's unified-memory-native array framework 📖 MLX_INFERENCE.md 📄 mlx_inference.py

"Built for the RAM the CPU and GPU already share." MLX is the only major tensor framework written natively for Apple Silicon's unified memory: arrays live in one pool, you pick the device at op time (stream=, not .to()), so the GPU and CPU read the same bytes with zero copy. Ops are lazy (build a graph, run nothing until mx.eval), which lets the compiler fuse a chain of K elementwise ops into one Metal kernel — cutting memory traffic K-fold. Recompute it all live below.

① Unified memory — CPU and GPU share ONE RAM pool (zero copy)

What this shows: on Apple Silicon the CPU and GPU address the same RAM. An MLX array lives in that pool; you pick the device at op time (stream=mx.gpu). A GPU matmul and a CPU print(a[0]) hit the same bytes — no copy. PyTorch MPS, on the same hardware, keeps separate buffers and copies on .to('mps') / .to('cpu').
MLX — unified memory (no copy arrows)
PyTorch MPS — split buffers (explicit copy arrows)
MLX: the GPU and CPU point at the same pool (no arrow = no copy). PyTorch MPS: .to('mps') + .to('cpu') = 2 copies per layer. Same formulas as mlx_inference.py Section A.

② Lazy evaluation — build the graph, nothing runs until “eval”

What this shows: MLX ops are lazy. a = mx.array([1,2,3,4]), b = a*2, c = b+5 build a graph of three nodes — but b and c hold no value until you click eval. That is what lets the compiler see the whole chain and fuse it (Panel 3). Edit the constants and re-run.
Dashed = LAZY (no value). Solid green = materialised by mx.eval. Until eval, the output array exists only as a recipe. Mirrors mlx_inference.py Section B.

③ Op fusion — K elementwise ops → 1 Metal kernel, K-fold less bandwidth

What this shows (the gold value): a chain of K elementwise ops run eagerly = K kernels, each doing 1 read + 1 write of the full array = 2K array-sized transfers. MLX fuses the chain into 1 kernel: 1 read + 1 write = 2 transfers. Bandwidth cut = K-fold. For K=3 → 6 down to 2 → 3× less.
"transfer" = one full-array read OR write to main memory (what fusion eliminates). Eager = 2K transfers across K kernel dispatches; fused = 2 transfers in 1 kernel. Mirrors mlx_inference.py Section C.

④ Functional transforms — value_and_grad, vmap, compile (stackable)

What this shows: MLX borrows JAX's model — transforms are higher-order functions that stack. value_and_grad(fn)(x) returns value + gradient in ONE pass; vmap(fn)(batch) batches; compile(fn) traces+fuses once. Compose freely: vmap(grad(compile(fn))).
grad of x² is 2x (analytic; MLX does real autodiff). vmap runs the transform over the batch as one fused op. Mirrors mlx_inference.py Section D.

⑤ Decode performance — bandwidth-bound; MLX spends less of it

What this shows: decode is bandwidth-bound — the engine reads ~the whole model per token. Roofline tokens/s ≈ unified_memory_BW / weight_bytes. MLX's three edges (no copy + fusion + compiled kernels) push utilisation toward the roofline. Reported ~2–3× over llama.cpp.
Roofline = BW / weight_bytes. MLX utilises more of it (no copy + fusion + compiled kernels); llama.cpp's Metal backend copies more and fuses less. Efficiencies are order-of-magnitude (modelled). Mirrors mlx_inference.py Section E.
