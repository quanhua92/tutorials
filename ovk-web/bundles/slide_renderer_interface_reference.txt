# slide_renderer_interface — web provenance log
# Every URL has a "Verifies:" line stating exactly what it confirms.
# STEP 2 of HOW_TO_RESEARCH.md is non-skippable; ≥2 sources per concept.

https://pptr.dev/api/puppeteer.page.screenshot
  Verifies: headless-Chromium frame capture is the standard approach for Impl 2
  (RFC §9.3 "our own renderer"). Puppeteer's Page.screenshot() is documented as
  "Captures a screenshot of this page" returning Promise<Uint8Array> (raw frame
  bytes) — i.e. drive a headless Chromium page to time t, screenshot it, get a
  raster frame. This is exactly the "headless-Chromium frame capture" primitive
  RFC §9.3 names. Confirms IMPL_2_OWN_RENDERER.tools "headless-Chromium
  (Page.screenshot / CDP)" in slide_renderer_interface.ts Section C, and the
  capture step "headless-Chromium capture every frame index 0..N-1" in the
  export path. (CDP = Chrome DevTools Protocol, the lower-level alternative to
  Page.screenshot; both are standard Puppeteer/Chromium capture mechanisms.)

https://ffmpeg.org/ffmpeg.html
  Verifies: FFmpeg as the standard encode/mux step (the own-renderer's backend).
  The FFmpeg documentation states: "ffmpeg is a universal media converter. It can
  read a wide variety of inputs - including live grabbing/recording devices -
  filter, and transcode them into a plethora of output formats." Synopsis:
  "ffmpeg [global_options] {[input_file_options] -i input_url} ... {[output_file_options]
  output_url}". This is exactly the encode/mux step Impl 2 needs: take the
  captured PNG/RGBA frames from headless Chromium (the inputs) and encode them
  into an MP4 (the output). Confirms IMPL_2_OWN_RENDERER.tools "FFmpeg encode/mux"
  in slide_renderer_interface.ts Section C, and RFC §9.3's "FFmpeg encode/mux"
  wording. Also confirms RFC §11's note that "FFmpeg with fixed params is
  near-byte-deterministic."

https://developer.mozilla.org/en-US/docs/Web/API/HTMLCanvasElement/toDataURL
  Verifies: an ALTERNATIVE standard in-browser frame-capture primitive (in
  addition to Puppeteer's Page.screenshot / CDP). MDN documents
  HTMLCanvasElement.toDataURL() as returning "a data URL containing a
  representation of the image in the format specified" — i.e. a canvas (which a
  preview/screenshot can be drawn into) can be serialized to a raster frame. This
  is the same family of "capture the current visual frame as bytes" primitives
  that Impl 2 relies on; it confirms headless-Chromium capture is not exotic but
  the standard Web-API approach. (Cited as the MDN anchor per the brief; the
  Puppeteer doc above is the primary capture source for a headless pipeline.)

(in-repo) docs/rfc-0001.md §9.1 "The interface"
  Verifies: the EXACT interface quoted verbatim in Section A —
  `interface SlideRenderer: load(slide_html, fields, assets);
  render_at(timecode) -> frame; duration() -> seconds` — and the load-bearing
  sentence "Preview and export both go through this interface." Primary in-repo
  source for Section A and the TypeScript SlideRenderer shape.

(in-repo) docs/rfc-0001.md §9.2 "Implementation 1 (POC, now): HyperFrames"
  Verifies: Impl 1's preview path ("instantiate HF's <hyperframes-player> per
  active slide; drive it to the playhead") and export path ("assemble slides +
  data into one HF composition; call npx hyperframes render once (§10). Proven,
  reuses today's render path."). Primary source for Section B and
  IMPL_1_HYPERFRAMES in slide_renderer_interface.ts.

(in-repo) docs/rfc-0001.md §9.3 "Implementation 2 (future): our own renderer"
  Verifies: Impl 2 = "Headless-Chromium frame capture + FFmpeg encode/mux,
  targeting visual determinism (§11). Swaps in behind the same interface; the
  editor and export pipeline are unchanged." Primary source for Section C and
  IMPL_2_OWN_RENDERER in slide_renderer_interface.ts.

(in-repo) docs/rfc-0001.md §9.4 "Why this kills fidelity drift (decision record)"
  Verifies: the structural-impossibility argument quoted in Section D — "Preview
  and export share one interface → one engine at every stage. POC: both use HF →
  consistent. Later: both swap to our renderer together → still consistent. 'It
  looked different when I exported' is structurally impossible, because preview
  and render are never two different engines." Primary source for Section D's
  three-scenario model (POC match / LATER match / forbidden MIXED case).

(in-repo) docs/rfc-0001.md §9.5 "Why 'make HF smaller'"
  Verifies: HF's contracted role quoted in Section E — "HF's role contracts from
  'the whole composition engine' to 'render this one slide at time t.' ... Timeline
  orchestration, the document model, and the editor are all ours; HF is a pluggable
  rendering primitive." Primary source for HF_ROLE_CONTRACT in
  slide_renderer_interface.ts.

(in-repo) docs/rfc-0001.md §10 "Export Pipeline"
  Verifies: the export pipeline goes through the SAME engine — step 5 is
  "npx hyperframes render <workspace> --output scene.mp4" — i.e. export does not
  bypass the SlideRenderer interface. Confirms Section A's "editor (§8 preview)
  and the export pipeline (§10) BOTH program to this interface."

(in-repo) docs/rfc-0001.md §11 "Determinism"
  Verifies: why Impl 2 targets VISUAL (not byte) determinism — "Same input →
  visually equivalent output across re-renders. Minor byte differences
  (encoder/browser non-determinism) are allowed. FFmpeg with fixed params is
  near-byte-deterministic; the variable part is browser capture." Confirms the
  mock's purity contract (Section F): render_at is a pure function of t after
  load, even though a real browser-capture impl would only be visually (not
  byte) deterministic. Cross-ref 🔗 VISUAL_DETERMINISM.

(in-repo) docs/rfc-0001.md §18 "Risks & Tradeoffs"
  Verifies: the risk row "Preview ↔ export fidelity drift | Structurally
  eliminated — preview and export share one SlideRenderer (§9.4)." Primary source
  for Section D's claim that drift is "the single biggest risk of a
  decoupled-preview architecture, eliminated by construction."

(in-repo) docs/AGENTS.md "Module responsibilities" + "Commands"
  Verifies: the HF CLI surface Impl 1 reuses — rendering.py "npx hyperframes
  render subprocess + job tracking" and the commands `npx hyperframes render
  templates/<id> --output out.mp4` and `npx hyperframes lint templates/<id>`.
  Confirms IMPL_1_HYPERFRAMES.tools and the "reuses today's render path" claim
  in Section B. Also documents `npx hyperframes inspect`/`snapshot` for frame
  capture/verification, the debugging analog of Impl 2's headless capture.
