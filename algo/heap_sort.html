A heap is an array pretending to be a tree: node i's children are 2i+1, 2i+2. In a max-heap the biggest element sits at the root. Heapsort builds the heap, then repeatedly swaps the root-max to the end and sifts the new root back down. Guaranteed O(n log n), in place, O(1) extra memory — no pathological input. Step through both phases, then compare it with quicksort's cache-friendly speed.  |  quick_sort: the faster-but-risky sibling →

① Build-heap then extract-max — bars + the tree they form

What to do: press play. Phase 1 sifts every internal node down to form a max-heap (root = max). Phase 2 swaps the root to the end (becoming sorted tail), shrinks the heap, and sifts the new root down. The bars are the raw array; the tree is the same array drawn as a complete binary tree — watch the max-heap property (parent ≥ children) hold after Phase 1.

Note build-heap is O(n) (not O(n log n)) — most nodes are near the bottom and barely move (Section C). The n−1 extractions dominate.

② The array IS the tree — index arithmetic, no pointers

What this shows: the max-heap of the worked array as both an array and a tree. parent(i)=(i-1)//2, left(i)=2i+1, right(i)=2i+2. Because it's a complete tree stored level-by-level, there are zero pointers — just index math.
Heap invariant: every parent ≥ its children. The root [0] is therefore the global maximum — that's the whole reason extraction works.

③ Heapsort vs quicksort — guarantee vs cache speed

What to do: drag n. Both are O(n log n) on random data, but heapsort does ~2× the comparisons AND is cache-hostile (sift-down jumps to indices 2i+1, 2i+2, far apart in memory). Quicksort's linear partition scan wins on real hardware — which is why introsort = quicksort + heapsort fallback.
bars = comparison counts (lower = fewer ops)
Takeaway: heapsort is the guarantee (worst-case O(n log n), O(1) space). Quicksort is the speed. std::sort keeps both: quicksort for the common case, heapsort as the safety net, insertion-sort for tiny subarrays.
