tts_kokoro.py - Kokoro-82M text-to-speech pipeline simulation.
Pure Python stdlib. Numbers below feed TTS_KOKORO.md.
Sources: https://huggingface.co/hexgrad/Kokoro-82M (Model Facts),
         StyleTTS2 arxiv:2306.07691, ISTFTNet arxiv:2203.02395.

Pipeline: normalize -> phonemize -> prosody -> mel-spectrogram -> vocoder.

  SECTION C: AUDIO FUNDAMENTALS - sample rate, hop, mel frames
==========================================================================
Kokoro audio config (verified: Model Facts + ISTFTNet):
  sample_rate = 24000 Hz  (24000 samples per second)
  hop_length  = 256 samples per mel frame
  fft_size    = 1024 samples (STFT window)
  n_mel_bands = 80 frequency bands

mel frames per second = sample_rate / hop_length
                     = 24000 / 256 = 93.75
                     ~ 94 frames/sec

1 second of audio:
  samples    = 24000
  mel frames = 93.75  (≈ 94)
  floats     = 93 frames * 80 bands = 7440 mel values

Per mel frame: 80 floats = a snapshot of frequency content
  -> frame covers 256 samples = 10.67 ms of audio

Duration in seconds from frames:
  seconds = frames * hop / sample_rate

  SECTION D: MEL-SPECTROGRAM - the acoustic model's output
==========================================================================
Each phoneme has a characteristic formant profile (energy across the
80 mel bands). The duration predictor REPEATS that profile
across the phoneme's frames, producing horizontal stripes. This is the
classic look of a TTS mel-spectrogram.

Input : phonemes=['/h/', '/ə/', '/l/', '/o/', '/ʊ/'], durations=[5, 12, 8, 15, 20]
Output: mel-spectrogram = 60 frames x 80 bands

  frame 5   :   = @ @ @ % :         (segment 1 /ə/ dur=12)
  frame 17  : . @ @ @ + . + @ @ :   (segment 2 /l/ dur=8)
  frame 25  : : @ @ % .             (segment 3 /o/ dur=15)
  frame 40  : = @ @ :               (segment 4 /ʊ/ dur=20)

==========================================================================
  SECTION F: STYLETTS 2 ARCHITECTURE (what Kokoro inherits)
==========================================================================
Kokoro's architecture = StyleTTS 2 (arxiv:2306.07691) + ISTFTNet
vocoder (arxiv:2203.02395). Decoder only: NO diffusion, NO released
encoder. Five components, all sharing one backbone:

  1. STYLE ENCODER
     extracts a speaking-style vector from reference audio (or a
     learned 256-dim vector per voice). This single vector
     carries identity + prosody style across the whole utterance.

  2. TEXT ENCODER
     phonemes -> hidden representations (the phoneme embedding +
     transformer/conv stack). Pre-computed phoneme embeddings keep
     the embedding table tiny.

  3. DURATION PREDICTOR
     predicts how many mel frames each phoneme spans (alignment).
     No external alignment labels needed.

  4. DECODER (NON-AUTOREGRESSIVE)
     aligned text + style -> mel-spectrogram. PARALLEL: all frames
     generated at once, not one-at-a-time. This is the speed win.

  5. VOCODER (ISTFTNet)
     mel-spectrogram -> waveform via inverse short-time Fourier
     transform. Cheaper than HiFi-GAN, lighter than diffusion.

The trick: ADVERSARIAL TRAINING + STYLE DIFFUSION. A discriminator
judges generated mel vs real mel; a diffusion process samples the
style vector. Together they yield natural prosody WITHOUT explicit
duration/pitch labels during training.

Voice storage: 54 voices * 256-dim * 4 bytes
             = 55296 bytes = 54.0 KB (negligible)

==========================================================================
  SECTION G: WHY KOKORO IS ONLY 82M PARAMS (vs 300M+ for other TTS)
==========================================================================
| # | Trick                              | Effect                          |
|---|------------------------------------|---------------------------------|
| 1 | Shared encoder backbone            | Text/duration/decoder share ONE feature extractor, not 3 duplicates |
| 2 | Non-autoregressive decoder         | All mel frames generated in parallel, no per-step recurrence stack |
| 3 | Compact 256-dim style vectors      | 256-dim style per voice vs huge speaker-embedding tables |
| 4 | Pre-computed phoneme embeddings    | Phoneme vocab is tiny (~100 IPA symbols), so the embedding table is tiny |
| 5 | ISTFTNet vocoder                   | Inverse-STFT is cheaper than HiFi-GAN's multi-block upsampling stack |

Result: 82M params total, runs in <2GB VRAM and on CPU at ~3x real-time.

VRAM budget (weights dominate):
  weights fp32 = 82000000 * 4 = 328 MB
  weights fp16 = 82000000 * 2 = 164 MB
  voices       = 54 * 256 * 4 = 54.0 KB
  + activations + mel buffers       ~ a few tens of MB
  => total well under 2 GB on any GPU, and runs on CPU.
