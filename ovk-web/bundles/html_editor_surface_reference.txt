# Every URL has a "Verifies:" line stating exactly what it confirms.

https://hyperframes.heygen.com/packages/cli
  Verifies: HyperFrames ships a `lint` CLI command. Verbatim from the page: "The `hyperframes` CLI ... handles project creation, live preview, rendering, linting, and diagnostics"; under §lint — "Check a composition for common issues: `npx hyperframes lint [dir]` / `npx hyperframes lint [dir] --verbose` / `npx hyperframes lint [dir] --json` # machine-readable JSON output". Also: "The linter detects missing attributes, missing adapter libraries (GSAP, Lottie, Three.js), structural problems, and more." This is the exact command the RFC §18 mitigation names ("AGENTS.md lint gate (`npx hyperframes lint`) before accepting"), and the `--json` flag is what makes it callable as a programmatic gate (the editor parses the JSON to decide accept/revert).

https://developer.mozilla.org/en-US/docs/Web/HTML/Element/template
  Verifies: the bare-`<template>` invariant the gate's R2 rule enforces. Verbatim: "By default, the element's content is not rendered. The corresponding `HTMLTemplateElement` interface includes a standard `content` property ... This `content` property is read-only and holds a `DocumentFragment`." The `<template>` is an inert carrier whose `.content` is what HF extracts and mounts — so an `<html>`/`<head>`/`<body>` wrapper *outside* the `<template>` is the authoring mistake that breaks extraction (the wrapper is not part of `.content`). Cross-cites AGENTS.md "Layout file format — CRITICAL" / "Why bare `<template>`".

https://hyperframes.heygen.com/concepts/compositions
  Verifies: HyperFrames' runtime "automatically fetches the file, extracts the `<template>` content, mounts it, executes scripts, and registers the timeline" and "each external composition file wraps its content in a `<template>` tag" — i.e. the bare `<template>` IS HF's native sub-composition format, which is why the gate's R2/R3 rules (bare `<template>` + `data-composition-id`) are non-negotiable: they enforce HF's own extraction contract so an accepted edit mounts correctly.

(in-repo) docs/rfc-0001.md §7 (Editor Surfaces — "HTML editor (per slide)" row), §16 (Animation Framework), §18 (Risks — "Tier-2 AI writes broken/invalid slide HTML" → mitigation)
  Verifies: §7 — the HTML editor "binds to" slide `index.html` and its job is "edit within-slide animation; the AI Tier-2 surface; power users." §16 — "GSAP + vanilla CSS inside slide `index.html` ... Do NOT introduce Tailwind or a motion library into composition files" (the R4 rule). §18 — the risk row and its mitigation: "AGENTS.md lint gate (`npx hyperframes lint`) before accepting an HTML edit (RFC 0002)" — i.e. accept-on-pass, revert-on-fail.

(in-repo) docs/AGENTS.md — "Layout file format — CRITICAL", "Why bare `<template>` (no `<html>` wrapper)", "Validate a template: `npx hyperframes lint templates/eco-bottle`"
  Verifies: the export-target contract the editor's accepted edit must obey — layout files MUST be a bare `<template>` (no `<html>`/`<head>`/`<body>`, no `data-composition-variables`); an `<html>` wrapper causes HF to NOT extract content and the sub-comp renders BLANK (verified HF v0.7.3); and the explicit lint command `npx hyperframes lint templates/eco-bottle` is the AGENTS.md validation step. This is the predicate set the gate encodes (R1/R2/R3).
