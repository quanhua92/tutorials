Multi-Version Concurrency Control (MVCC) — interactive 📖 MVCC.md 📄 mvcc.py

📖 Read the full guide — architecture diagrams, decision tables, killer gotchas, cheat sheet. This page is the interactive companion.
Every row is a stack of drafts: an UPDATE writes a new version and stamps the old one's xmax, but never overwrites. A transaction picks the one draft whose xmin/xmax match its snapshot — so readers never block writers. Drag the snapshot horizon and watch the visible version step forward. The visibility engine is recomputed in JS from the identical rules as mvcc.py.  |  visibility map (Index-Only Scan) →

① Version chain — slide the snapshot horizon, watch the visible version step

What to do: drag xmax (snapshot horizon = next xid). The querier sees inserts with xmin < xmax (committed). As xmax grows past 200, then 300, the visible version steps v1 → v2 → v3. Same three drafts on the page; the snapshot picks one.
chain: v1(100,200) · v2(200,300) · v3(300,0)   committed = {100,200,300}
VISIBLE version (the one this txn reads) dead / superseded (xmax committed-visible) inserter not yet visible (xmin ≥ xmax)
At xmax=251 (mvcc.py Section A, snapshot@250) the visible version is v2: T200 committed & visible, but T300 (the superseder) hasn't started (300 ≥ 251), so v2's delete doesn't count. (MVCC.md §1)

② The snapshot — xmin horizon, xmax horizon, and the in-progress list (xip)

What this shows: a snapshot carves the xid line into committed-visible (started, not running, committed), in xip = running → invisible, and ≥ xmax = not started → invisible. A row is visible iff its inserter is committed-visible AND its deleter is not.
snapshot S = {xmin:200, xmax:400, xip:[250,350]}   committed = {100,150,200,300}
Note p5: an old row whose deleter is in xip stays visible — the delete hasn't happened yet from this snapshot's view. That same rule is why a long-running txn blocks VACUUM (MVCC.md §2, §5).

③ Isolation — READ COMMITTED (non-repeatable) vs REPEATABLE READ (pinned snapshot)

What to do: toggle the isolation level. READ COMMITTED takes a fresh snapshot per statement → the second read sees T501's update (non-repeatable read). REPEATABLE READ pins the snapshot at t1 → xid 501 stays "not started" → both reads return 100.
T500 reads, T501 UPDATE balance 100→200 + COMMIT between the reads
The only difference is one line: whether t4 uses a fresh S2 (xmax=502) or the pinned S (xmax=501). (MVCC.md §3, §4)

④ VACUUM — UPDATE bloats, VACUUM removes dead tuples

What to do: hit VACUUM. It removes every tuple that no live snapshot can see, frees the line pointers, and sets the visibility-map bit. Toggle the old snapshot open switch: v1 is still visible to it, so VACUUM must retain it → bloat stays.
old snapshot open: OFF
dead(v) = not visible under ANY live snapshot. A single long-running transaction pins the xmin horizon and stalls VACUUM for the whole DB. (MVCC.md §5)

⑤ XID wraparound — freeze old tuples before the 32-bit counter wraps

What to do: drag current_xid past the thresholds. At 50M VACUUM freezes opportunistically; at 200M a FORCED anti-wraparound autovacuum runs (cannot be disabled); at 2³¹ the signed comparison flips → data loss unless frozen.
oldest unfrozen xmin = 1,000  ·  32-bit space = 2³²
Freezing overwrites xmin with FrozenTransactionId(2) = "committed forever", so the old xid can be reused after the wrap. (MVCC.md §6)
