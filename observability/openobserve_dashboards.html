OpenObserve — Dashboards, Alerts, Reports, RUM, Traces & Functions OPENOBSERVE_DASHBOARDS.md openobserve_dashboards.py

The user-facing layer of O2 is six modules over one Rust/S3 backend. A panel, an alert, and a report all start as the same SQL query against a stream — the difference is what happens to the result: draw it, test it, or screenshot it. RUM ships browser vitals, traces draw an OTLP span waterfall, and functions (VRL + Python) transform rows in flight. Build every module live below.

① The six UI modules — one backend under them all

What this shows: O2 is a single binary doing logs+metrics+traces+RUM on Parquet/S3. The six modules are six lenses over the same streams. Click a module to see what it does. Every one starts from a SQL query against a stream — the backend never moves data between systems.

② Dashboard panel builder — stream + SQL → preview

What this shows: a panel is built in 4 steps: stream → SQL → panel type → gridPos. Pick a stream and panel type; the matching SQL template and a mini-rendering update live. The panel type is dictated by the question, not the data.

③ Alert rule simulator — evaluate → fire → notify → resolve

What this shows: the alert lifecycle over a 40-tick error-rate series (seeded; an incident spikes above threshold in ticks 10–22). Drag the knobs: threshold (when it bites), for (consecutive true evals before FIRING), cooldown (gap between re-notifications). See exactly when it fires and resolves, and how many notifications actually go out.

④ Report scheduler — PDF/CSV via cron

What this shows: a report snapshots a dashboard on a schedule and emails it. Toggle schedules and the window; see run counts and storage. Rendering is a headless-browser PDF — expensive, so keep schedules coarse.

⑤ RUM session visualizer — Core Web Vitals → perf score

What this shows: the browser SDK ships Core Web Vitals to a rum stream. Drag LCP / INP / CLS to their p75 values; each falls into a good / needs / poor bucket (100/50/0). The weighted score is what the RUM dashboard surfaces. Replay + error tracking close the loop.

What this shows: OTLP spans land in a traces stream; the UI renders a Gantt waterfall. This is a 6-span checkout trace (total 120 ms). The critical path (longest parent→child chain) is what latency optimization targets. Click a span to see its detail.
