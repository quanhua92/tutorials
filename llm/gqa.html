Grouped-Query Attention (GQA) — interactive GQA.md gqa.py

Think of H_kv as the number of filing cabinets the workers (query heads) share. Slide it down from H_q (MHA: every worker has their own cabinet) to 1 (MQA: everyone shares one cabinet) and watch the cabinet count — and the memory bar — shrink. ① The H_kv dial: MHA → GQA → MQA, and the KV cache it buys you

What to do: drag H_kv down from H_q and watch the cabinet count — and the memory bar — shrink. Fewer cabinets = less to re-read each step = faster generation.
D = 64, layers = 28, S = 4096, fp16 — matches gqa.py Section B
In words: the KV cache = the cabinets, and its size is set by H_kv (the cabinet count), never H_q. Halve the cabinets and you halve the memory re-read every step. KV cache = layers × 2 × H_kv × S × D; n_repeats = H_q // H_kv (workers per cabinet).

② The sharing trick — each group of n_repeats workers points at ONE cabinet

What this shows: instead of photocopying each cabinet, we tell the computer "these workers are a group" and let them all read the same cabinet. Toggle the button to see the #1 bug: group them in the wrong order and workers grab the wrong cabinet — silently.
tiny model: H_q = 4, H_kv = 2, n_repeats = 2
query head Q_h shared KV head (broadcast) wrong pairing (scrambled)
Correct: Q-heads 0,1 → KV0 ; 2,3 → KV1. Wrong order stripes the groups so Q1 and Q2 grab the wrong KV head — no error raised, just garbage (GQA.md §7).
