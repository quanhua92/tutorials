KV Cache Quantization — FP16 vs Q8_0 vs Q4_0 📖 KV_CACHE_QUANT.md 📄 kv_cache_quant.py

The KV cache stores K and V for every token, every KV head, every layer. At long context it dominates VRAM. Quantize it from FP16 (2 B/E) to Q8_0 (1.0625 B/E, ~47% saved) or Q4_0 (0.5625 B/E, ~72% saved) — independently of the weight quant. Drag the context slider and watch all three bars shrink. The gold-check pins Llama-3-8B Q8_0 KV at 32K to the exact .py output.

② Savings comparison — FP16 vs Q8_0 vs Q4_0

What this shows: side-by-side bar chart of KV cache GB for the selected model + context. Q8_0 nearly halves; Q4_0 nearly quarters. The block format adds a small overhead (2-byte fp16 scale per 32 elements), so Q4_0 is 3.6× not 4×.
Block layout: fp16 scale (2 B) + data. Q8_0 = 32×int8 (34 B/block), Q4_0 = 32×int4 (18 B/block). Same GGML block format as weight quants.

③ Quality trade-off — perplexity increase (approximate)

What this shows: approximate perplexity ratio (1.000 = FP16 baseline) from llama.cpp benchmarks and community reports. Q8_0 is nearly free (<1% increase). Q4_0 is moderate (2–8%). These are empirical, model-dependent values — not computed from a formula.
Rules of thumb: Q8_0 KV → always on (free). Q4_0 KV → chat/retrieval OK, avoid for code/math. Longer contexts spread the error, so relative impact per token is smaller.
