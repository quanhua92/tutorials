① Model size = params × bytes/param → drag params and quant, watch the Q4 column land where mobile lives

How to read: size_GB = params × bytes/param / 1e9. fp16=2.0, Q8=1.0625, Q4=0.5625. The gold anchor: 1B @ Q4 = 0.5625 GB. Quantizing fp16 → Q4 cuts size ~3.6× at every scale — and mobile deployment lives almost entirely in the Q4 column.

② Active RAM = weights + KV-cache + overhead → the KV-cache grows LINEARLY with context (the #1 mobile OOM)

How to read: fix the model at 1B @ Q4 (gold) and drag seq_len. The weights bar is flat; the red KV bar doubles every time context doubles. Past ~4K it overtakes the weights; at 32K it's 8× bigger and OOMs a 4GB phone. The pie on the right shows the live weights/KV/overhead split for the current context.

③ Device recommender → drag device RAM + NPU, find the largest (params, quant) that fits the active budget

How to read: active_ram = total × fraction (phone 0.5 — OS takes half; browser 1.0 — the 2GB IS the cap). The recommender picks the largest params that fits at seq=2048, preferring Q8 over Q4 for quality. A 1B model fits everywhere; Q4 (~0.56GB) is the universal fallback quant — the mobile default.

④ Three runtimes, three catches — NPU (fast, op gaps) vs browser (portable, GC) vs unified (one export, EPs)

How to read: the NPU wins on speed but has op-coverage gaps; the browser wins on reach but is GC'd and tab-capped; the unified runtime wins on portability. No free lunch — each bills a different tax. (mobile_runtime.py Section C.)
