=== Defense in Depth — independent layers, multiplicative defense
========================================================================
  A castle with MANY concentric walls. No single wall is perfect;
  each layer has independent controls and an independent probability
  of stopping an attack. Because the layers are INDEPENDENT, the
  combined breach probability is the PRODUCT of per-layer leak
  rates — so a stack of good-enough layers leaks far less than any
  one perfect layer.

  layer     stop_prob   leak_rate   controls
  ------------------------------------------------------------
  network       85%          15%      WAF, firewall + ACLs, DDoS scrubbing, segmentation
  host          80%          20%      OS hardening, patching, EDR / antivirus, host firewall
  app           90%          10%      input validation, authN + authZ, secure coding, SCA / SAST
  data          95%           5%      encryption at rest, tokenization, DLP, access audit log

  combined breach = product of leak rates
                  = 0.15 x 0.20 x 0.10 x 0.05
                  = 0.00015  (0.015%)
                  = 15 breaches per 100,000 attacks

  best single layer (data, stop=95%) leaks 5% = 5,000 per 100,000
  defense-in-depth improves on the best single layer by 333x

  SIMULATION (100,000 attacks, LCG seed=20260627):
    stopped at network   = 85,017   (expected   85,000)
    stopped at host      = 11,947   (expected   80,000)
    stopped at app       =  2,744   (expected   90,000)
    stopped at data      =    280   (expected   95,000)
    breached ALL layers   =     12   (expected     15.0)

=== TLS Handshake — TLS 1.2 (2-RTT) vs TLS 1.3 (1-RTT) vs 0-RTT
========================================================================
  TLS provides encryption (confidentiality), authentication (server
  identity via X.509 certificate), and integrity (tamper detection).
  TLS 1.3 halved the handshake to a single round trip and added
  0-RTT resumption; both use ECDHE for forward secrecy — past
  sessions stay secure even if the server's long-term key leaks.

  TLS 1.2 full handshake (2 RTT before app data):
    C->S  ClientHello
          version, random_client, cipher_suites, SNI, key_share (1.3)
    S->C  ServerHello + Certificate + ServerKeyExchange + HelloDone
          random_server, chosen cipher, DH params, cert chain
    C->S  ClientKeyExchange + ChangeCipherSpec + Finished
          encrypted premaster / DH public, switch to encrypted
    S->C  ChangeCipherSpec + Finished
          server switches to encrypted

  TLS 1.3 full handshake (1 RTT before app data):
    C->S  ClientHello (+ key_share, + early_data on resumption)
          one flight: proposed ciphers + ECDHE public key
    S->C  ServerHello + {EncryptedExtensions, Cert, CertVerify, Finished}
          from here on EVERYTHING is encrypted
    C->S  {Finished}
          client confirms; app data follows

  RTT = 50ms (representative)
    TLS 1.2        = 2 RTT = 100ms   (handshake)
    TLS 1.3        = 1 RTT = 50ms   (handshake)
    TLS 1.3 0-RTT  = 0 RTT = 0ms   (resumed session)
    1.3 saves 50ms per connection vs 1.2

  FORWARD SECRECY (ECDHE): each session derives EPHEMERAL keys from
  a fresh Diffie-Hellman exchange. If the server's long-term private
  key is stolen tomorrow, PAST recorded sessions CANNOT be decrypted
  — the ephemeral keys were destroyed at session end.

  TLS 1.3 saves exactly 1 RTT (50ms) vs 1.2?    [check] OK
  TLS 1.3 has fewer flights than TLS 1.2?        [check] OK

  [check] OK   (TLS 1.3: 1-RTT handshake, 0-RTT resumption, ECDHE)

  GOTCHA: 0-RTT is vulnerable to REPLAY — an attacker can replay the
  early-data request (e.g. a duplicate POST). Only allow 0-RTT for
  idempotent operations, or enforce idempotency keys server-side.
  GOTCHA: TLS authenticates the CHANNEL, not the USER. A valid TLS
  connection to api.evil.com still reaches the attacker's server —
  always validate the certificate's hostname (SAN), not just the CA.

=== mTLS — mutual authentication (the client proves identity too)
========================================================================
  Standard TLS authenticates only the SERVER. mTLS (mutual TLS)
  requires the CLIENT to ALSO present a certificate, signed by a CA
  the server trusts. The server verifies it — both sides are now
  authenticated and the channel is encrypted. This is the identity
  backbone of service meshes and zero-trust internal networks.

  server-only TLS:
    - Client validates SERVER certificate against trusted CA
    - Server is authenticated; client is anonymous (or auth via app layer)

  mTLS (adds the client-cert flight):
    - Client validates SERVER certificate against trusted CA
    - Server requests + validates CLIENT certificate against internal CA
    - Server maps client cert to a workload identity (SPIFFE ID / service)
    - Server authorizes the request based on that identity

  HANDSHAKE COST (mTLS vs server-only TLS 1.3):
    server-only handshake  = 3,500 bytes (1 RTT)
    mTLS handshake         = 5,564 bytes (+2000 client cert, +64 CertVerify)
    overhead               = +2,064 bytes (~59% larger), still 1 RTT

  WORKLOAD IDENTITY (SPIFFE / SPIRE):
    - SPIFFE ID: a URI-shaped identity, e.g.
      spiffe://prod.acme.com/payments/service
    - issued by SPIRE (or the mesh control plane) as a short-lived
      X.509 SVID (rotated automatically, TTL ~7d)
    - the cert IS the identity; no shared passwords, no long-lived keys

  WHERE mTLS IS USED:
    service mesh (Istio / Linkerd) : sidecar enforces mTLS transparently
    microservice-to-microservice   : every hop authenticated
    internal APIs / banking         : strong client identity required
    Kubernetes pod-to-pod          : mesh + NetworkPolicy together

  authZ (zero trust) so a stolen workload identity cannot impersonate
  arbitrary users.

=== Secrets Management — the rotation lifecycle
========================================================================
  A secret (API key, DB password, signing key) is valuable ONLY while
  it is secret. The lifecycle keeps it that way: generate outside the
  app, distribute via a sidecar, rotate on a schedule, revoke on
  compromise, and audit every access. NEVER bake secrets into images,
  env vars, or source control.

  lifecycle:
    generate     - create in HSM/KMS; plaintext never leaves the boundary
    distribute   - inject at runtime via sidecar (Vault Agent) / CSI volume
    use          - fetch + cache with TTL; never log, never return to client
    rotate       - periodic dual-key overlap; old + new valid during window
    revoke       - kill compromised key via KMS; propagates to all consumers
    audit        - log every read/decrypt; alert on anomalous access

  LEAK-RISK MODEL (daily leak rate = 0.1%):
    risk(days) = 1 - (1 - leak_rate)^days   (compounds daily)
    1 day   -> risk = 0.1000%
    30 days -> risk = 2.96%
    90 days -> risk = 8.61%   (canonical rotation interval)
    365 days-> risk = 30.6%   (a year-old key is risky)

    short-lived token (TTL 1h) -> risk = 0.0042%
    1h token vs 90-day key: 2,066x lower leak risk

  DUAL-KEY ROTATION (zero-downtime key change):
    day 0   : create key_v2 alongside key_v1 (both can DECRYPT)
    day 0+  : new writes use key_v2; old writes still readable
    day 7   : all data re-encrypted to key_v2; revoke key_v1
    overlap = 7 days, so a rolling deploy never sees a
              key the other instances cannot decrypt

  SHORT-LIVED vs LONG-LIVED:
    long-lived (90-day key) : blast radius = 90 days of data
    short-lived (1h token)  : blast radius = 1 hour of data
    2,066x reduction in the window of compromise

  1h token > 1000x lower risk than 90-day key?  [check] OK
  overlap window shorter than rotation interval?[check] OK

  [check] OK   (secrets: rotate often, overlap keys, prefer short TTLs)

  GOTCHA: secrets in ENV VARS are leaked via /proc/<pid>/environ,
  process listings, and crash dumps. Inject via a mounted volume or
  sidecar that fetches on demand and caches with a short TTL.
  GOTCHA: a secret in git is compromised FOREVER — the commit history
  is forever. Rotate immediately AND scrub history (git filter-repo),
  because anyone may have already cloned the repo.

=== Zero Trust — never trust, always verify (every request)
========================================================================
  Castle-and-moat security trusts anything INSIDE the network (a VPN
  or office IP grants broad access). Once an attacker breaches the
  perimeter — a phished credential, a vulnerable pod — they move
  LATERALLY with little resistance. Zero trust (BeyondCorp, NIST
  800-207) eliminates implicit trust: EVERY request is authenticated
  and authorized, regardless of where it originates, based on identity
  + device posture + context — not network position.

  THE MODEL (one compromised internal credential):
    estate              = 100 internal services
    compromised cred    = authorized for 2% (least privilege) = 2 service(s)

  castle-and-moat (trust by location):
    attacker is 'inside' the VPN -> trusted by the network
    lateral movement reaches ALL 100 services
    detection: ~7 days (anomalous access unnoticed)

  zero-trust (verify every request):
    every request hits a PEP (policy enforcement point)
    -> authenticates identity (mTLS / token)
    -> authorizes THIS action on THIS resource (least privilege)
    -> continuous eval flags anomalous access in ~5 min
    attacker reaches only 2 service(s) the cred can access

  BLAST-RADIUS COMPARISON:
    castle-and-moat : 100 services, detected in ~7 days
    zero-trust      : 2 service(s),  detected in ~5 min
    reduction       : 50x smaller blast radius (+ 2016x faster detection)

  COMPROMISE COST (services x hours exposed):
    castle-and-moat : 100 svc x 168h = 16,800 svc-hours
    zero-trust      : 2 svc x 0h = 0.2 svc-hours
    ratio           : 100,800x lower impact under zero trust
