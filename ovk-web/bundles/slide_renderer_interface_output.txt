SECTION A: the interface — preview AND export both go through it
============================================================
  RFC 0001 §9.1 — the SlideRenderer interface (verbatim):

    interface SlideRenderer:
        load(slide_html, fields, assets)   # mount a slide composition
        render_at(timecode) -> frame       # render at time t (preview or capture)
        duration() -> seconds

  Three methods. The editor (§8 preview) and the export pipeline
  (§10) BOTH program to this interface — they never reach around it.
  That single constraint is what makes the engine replaceable (§9.3)
  and fidelity drift structurally impossible (§9.4).

SECTION D: §9.4 — why fidelity drift is STRUCTURALLY IMPOSSIBLE
============================================================
  RFC 0001 §9.4 (decision record): 'Preview and export share ONE
  interface → ONE engine at every stage. POC: both use HF →
  consistent. Later: both swap to our renderer together → still
  consistent. "It looked different when I exported" is structurally
  impossible, because preview and render are never two different
  engines.'

    POC   preview.renderAt(2) = {"slideId":"slide-0","time":2,"loaded":true,"hasContent":true}
    POC   export .renderAt(2) = {"slideId":"slide-0","time":2,"loaded":true,"hasContent":true}
    LATER preview.renderAt(2) = {"slideId":"slide-0","time":2,"loaded":true,"hasContent":true}
    LATER export .renderAt(2) = {"slideId":"slide-0","time":2,"loaded":true,"hasContent":true}
    MIXED preview.renderAt(2) = {"slideId":"slide-0","time":2,"loaded":true,"hasContent":true}  (HF backend)
    MIXED export .renderAt(2) = {"slideId":"slide-0","time":2,"loaded":true,"hasContent":true}  (own backend)

[check] POC stage: preview + export use the SAME backend (both HF) → frames match: OK
[check] LATER stage: preview + export use the SAME backend (both own) → frames match: OK
[check] the MIXED case (preview≠export engine) is exactly what the architecture FORBIDS: OK
  → there is no code path where preview and export pick different
    engines. The seam enforces one engine per stage; drift cannot arise.
    This is the single biggest risk of a decoupled-preview architecture,
    eliminated by construction (RFC §18 risk row).

SECTION E: §9.5 — make HF smaller (it contracts to one job)
============================================================
  RFC 0001 §9.5 — HF's role contracts from 'the whole composition
  engine' to 'render this one slide at time t.' Timeline orchestration,
  the document model, and the editor are all OURS; HF is a pluggable
  rendering primitive.

    HF owns   = "render one slide at time t"
    OURS      = timeline orchestration, document model, editor, asset library, AI

[check] HF's surface is the 3-method interface (load / render_at / duration): OK
[check] timeline + document model + editor + assets + AI are OURS (5 concerns, not HF's): OK
  → contracting HF to one job is what makes it replaceable (§9.3).
    If HF owned the timeline or document model, swapping it would mean
    rewriting the editor. It doesn't, so it doesn't.
