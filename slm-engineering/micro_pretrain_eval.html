📖 Read the full guide — the full-bench → held-out → slice → micro-bench lineage, correlation tables, killer gotchas, cheat sheet. This page is the interactive companion.
Stop running the full benchmark every checkpoint. Hold out tiny capability slices (code, reason, factual, commonsense), log their loss each checkpoint (cheap), and predict the downstream benchmark from the slice whose loss tracks it tightly (Pearson |r| ≥ 0.9). Every number below is recomputed in JS from the identical formula in micro_pretrain_eval.py.  |  compare with pretraining stability →

① Slice losses vs the benchmark across checkpoints — toggle which slices you watch

How to read: every slice loss (left axis, lower=better) falls as training advances; the benchmark (right axis) rises. The question is which slice's curve mirrors the benchmark. code drops fast then plateaus — exactly the benchmark's shape. commonsense wanders (note the upticks) — it does not track, which is why Section ② shows it's a misleading proxy.

② Pearson |r| per slice — the faithful-proxy threshold (0.85) and the micro-bench bar (0.9)

How to read: the height of each bar is |r| between that slice's loss-series and the benchmark. code clears 0.9 on its own → it is the micro-bench. commonsense (0.657) sits below 0.85 → a misleading slice you should drop.

③ Fit B_hat = a·L_code + b on early ckpts → predict the held-out late ckpts

How to read: the line is fit on checkpoints 0..4 (where you DID run the full benchmark), then used to predict checkpoints 5..7 from the code-slice loss alone. The predicted points (open circles) land within ~1% of the true benchmark (solid) — that is the micro-pretrain-eval payoff.
