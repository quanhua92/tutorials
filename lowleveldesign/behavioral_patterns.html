Strategy — swap algorithms at runtime

A family of algorithms behind one interface. The DataProcessor context delegates to whichever strategy is plugged in — no if/else, fully Open/Closed. Click an algorithm to swap it live.

Observer — publish / subscribe, fan-out

The OrderService subject notifies every subscribed observer when an order is placed. Toggle observers on/off to subscribe or unsubscribe, then publish an event and watch the fan-out.

Command — encapsulate a request, enable undo

Each insert is a Command object pushed onto a history stack. Undo pops and reverses it. A replay guard (set of applied ids) makes re-applying the same command idempotent.

State — step through a state machine

The OrderContext delegates pay/ship/cancel to its current state object. Each state implements the operations differently — PaidState.pay() is a graceful no-op, ShippedState.cancel() is rejected.

Transitions

current: CREATED
Invalid ops raise (e.g. ship a CREATED order). Redundant valid ops are no-ops.

Template Method — fixed skeleton, swappable steps

The base DataParser.parse() fixes the order read → parse → validate → save. Subclasses fill in each step. The Hollywood Principle: the base calls you, not the other way around.

Chain of Responsibility — middleware pipeline

A request flows Auth → RateLimit → Log. Each handler either short-circuits (returns false) or passes to the next. Toggle handlers and the token to see how the chain breaks early.
