MLP Activations — ReLU → GELU → SiLU / SwiGLU — interactive MLP_ACTIVATION.md mlp_activation.py

For newcomers: an activation is a decision gate — ReLU is a hard on/off switch, GELU/SiLU are smooth dimmers. A SwiGLU block adds a faucet (silu(gate) = the handle, up = the water).   Drag input x and toggle the bug below — every number is recomputed live in JS from the identical formula in mlp_activation.py.  |  → compare with RoPE

① ReLU vs GELU vs SiLU — drag x and read off exact values

How to use: move the input x slider and watch the three curves bend. The dots show each function's output at your x. ReLU (red) is a hard switch that floors at 0; GELU (yellow) and SiLU (blue) are smooth dimmers that dip slightly below zero for small negatives instead of snapping dead.
Formulas copied verbatim from mlp_activation.py Section A
ReLU(x) = max(0, x) — hard switch GELU_tanh(x) = 0.5x(1 + tanh(√(2/π)(x + 0.044715x³))) — dimmer SiLU(x) = x · sigmoid(x) — self-gated dimmer current x
What to notice: for x<0, ReLU is stuck at 0 (the "dying ReLU" zone), while GELU and SiLU let a small negative signal through — SiLU's dip is larger (floor ≈ −0.278 near x≈−1.28), GELU's is smaller (≈ −0.17 near x≈−0.75). Above x≈+2 all three converge to identity (the switch is fully on).

② SwiGLU block: gate → silu → ×up → down (live, fixed seeded input)

How to use: move the token m slider to step through the 4 tokens. Each column is one stage of the faucet: gate(x) = the handle signal, silu(gate) = the smoothed dial (0..1-ish), up(x) = the raw water, and silu(gate)·up = the gated features that survive. down then mixes them back to 8 outputs.
Fixed input x[0,0] (E=8) from mlp_activation.py FIXED INPUT, seeded weights (manual_seed=1). Token selector cycles through m=0..3 of the seeded batch.
FFN=16, E=8 (same tiny model as the .py)
gate path (gets silu — the handle) up path (raw, no activation — the water) gated feature = silu(gate)·up (survivors) down-projected output [E]
Structure: down( silu(gate(x)) * up(x) ) — 3 matrices, Shazeer 2020. SiLU goes ONLY on the gate branch (the dial), never on up (the water). Panel ③ shows why swapping them silently breaks everything.

③ The operand-order pitfall — toggle the bug and see the difference

How to use: click the two buttons to swap which branch gets silu. The readout always computes both wirings so you can compare. Notice the outputs differ and there is no error — that's what makes this bug so dangerous.
Same weights, same input — only the silu placement differs.
Why it bites: silu is non-linear, so silu(a)·b ≠ a·silu(b). Llama/Qwen checkpoints are trained with silu on the GATE path, so swapping it silently corrupts inference (same shapes, no error message, just slightly wrong numbers). This is the #1 SwiGLU implementation bug. See §7 in the guide →
