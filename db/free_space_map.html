The Free Space Map (FSM) — interactive 📖 FREE_SPACE_MAP.md 📄 free_space_map.py

📖 Read the full guide — architecture diagrams, decision tables, killer gotchas, cheat sheet. This page is the interactive companion.
Each heap page's free space becomes a 1-byte category (free space ÷ 32). The categories stack into a max-tree: every internal node holds the best (max) free space in its subtree, so the root alone tells you whether any page fits. Slide "bytes needed" to watch the search path light up; click a page to insert a tuple and watch the max bubble back up. Recomputed live in JS from the identical inputs as free_space_map.py.  |  Lineage: linear scan O(N) → FSM tree O(log N).

① The max-tree — slide "bytes needed", follow the search path down

What to do: drag bytes needed. The path turns gold as the search steps to a child whose value ≥ the needed category, ending on a green page that fits. Read the root first: if it's below the needed category, no page fits.

A category is a floor in 32-byte buckets, so the FSM guarantees the result page has ≥ cat×32 bytes — not strictly ≥ what you asked. PostgreSQL re-checks the real free space on the page before inserting (the FSM is a fast filter, not an exact ledger).

② The flat array — the tree is just header + bytes (no pointers)

heap-array: parent(i)=(i−1)//2, left(i)=2i+1, right(i)=2i+2  |  bytes 0..7 = FSMPageHeaderData (fp_next_slot), then fp_nodes[0..14]
Leaves are the last 8 slots (indices 7..14 = pages 0..7); the root is index 0. Searching streams through contiguous bytes — no pointer chasing, perfect cache locality. Real PostgreSQL packs ~4092 leaves per 8 KB FSM page, then stacks FSM pages in a 3-level tree (4092³ > 2³² blocks).
