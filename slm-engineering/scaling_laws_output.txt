==========================================================================
  SECTION A: the two loss formulas -- Kaplan vs Chinchilla
==========================================================================
KAPLAN 2020 (param-only power law):
  L(N)        = (N_c / N) ** alpha_N
  alpha_N     = 0.076   (steep: loss falls SLOWLY with N)
  N_c         = 8.8e+13   (critical parameter count)
  -> recipe:  funnel compute into BIGGER models, starve data.
     GPT-3 followed this: 175B params on 300B tokens ~= 1.7 tokens/param.

CHINCHILLA 2022 (param + data power law):
  L(N, D)     = E + A / N**alpha + B / D**beta
  E           = 1.69     (irreducible entropy floor)
  A           = 406.4    alpha = 0.34  (param term)
  B           = 410.7    beta  = 0.28  (data term)
  -> alpha ~= beta => scale N and D TOGETHER; D_opt ~= 20 * N.

Same loss, very different prescriptions. Sample at N=1B, D=20B:
  Kaplan   L(1B)              = 2.3756  (ignores data entirely)
  Chinchilla L(1B, 20B)       = 2.5800  (the 20:1 optimal point)

  [check] Chinchilla optimal tokens/param ratio in [15, 25]: OK

==========================================================================
  SECTION B: the compute-optimal frontier  (N_opt = sqrt(C/120), D_opt = 20*N)
==========================================================================
Chinchilla: C ~= 6*N*D and D_opt ~= 20*N  =>  C = 6*N*(20*N) = 120*N^2.
So the optimal model size for a budget C is  N_opt = sqrt(C / 120),
and the optimal token count is  D_opt = 20 * N_opt  (always a 20:1 split).

| FLOP budget C | N_opt (params) | D_opt (tokens) | D/N  | check C ~= 6ND |
|---------------|----------------|----------------|------|----------------|
| 1e+18      |      9.129e+07 |      1.826e+09 | 20.0 |      1.28e-16 |
| 1e+19      |      2.887e+08 |      5.774e+09 | 20.0 |      0.00e+00 |
| 1e+20      |      9.129e+08 |      1.826e+10 | 20.0 |      1.64e-16 |
| 1e+21      |      2.887e+09 |      5.774e+10 | 20.0 |      0.00e+00 |
| 1e+22      |      9.129e+09 |      1.826e+11 | 20.0 |      2.10e-16 |

Read each row: doubling C grows N_opt and D_opt by sqrt(2) ~= 1.41x each.
The D/N column is a flat 20.0 everywhere -- that is Chinchilla's whole point.

==========================================================================
  SECTION C: the overtraining curve -- why deployed SLMs run D >> 20*N
==========================================================================
Fixed model size N = 1.35e+08 (SmolLM2-135M).
Chinchilla-optimal data for this N: 20*N = 2.7e+09 tokens.
Sweep overtrain factor k = D / (20*N).  Inference ~= 2*N FLOPs per generated token; 1024 tokens/query.

| k (overtrain) | D (tokens) | modeled loss | equiv Chinchilla N' | extra pretrain FLOPs | break-even query-tokens | break-even queries |
|---------------|------------|--------------|---------------------|----------------------|--------------------------|--------------------|
| 1             |    2.7e+09 |       3.3284 |            1.35e+08 |                    0 |                        0 |                  0 |
| 5             |   1.35e+10 |       2.9877 |           2.897e+08 |            8.616e+17 |                2.784e+09 |          2.719e+06 |
| 50            |   1.35e+11 |       2.7033 |           6.527e+08 |            5.823e+19 |                5.624e+10 |          5.492e+07 |
| 300           |    8.1e+11 |       2.5794 |           1.002e+09 |            5.356e+20 |                3.088e+11 |          3.015e+08 |
| 3000          |    8.1e+12 |       2.4891 |           1.426e+09 |            6.317e+21 |                2.446e+12 |          2.389e+09 |

Reading the table like a story:
  k=1     : the Chinchilla-optimal point. N' = N, no advantage, no cost.
  k=5     : a little more data; tiny loss drop; breaks even quickly.
  k=3000  : a 135M model overtrained to match a ~1.4B
            Chinchilla model -- you serve ~7x fewer FLOPs per token,
            repaying the extra pretrain FLOPs after ~2.4 billion queries.
  => For a DEPLOYED SLM queried billions of times, overtraining wins; for
     a one-shot research checkpoint, Chinchilla-optimal is still right.
