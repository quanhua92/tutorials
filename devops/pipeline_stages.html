A pipeline is an assembly line: Source → Build → Test → Package → Deploy → Verify. Toggle the test stage between sequential (jobs run back-to-back, total = sum) and parallel (jobs run at once, total = max). All numbers recompute live from the same model as pipeline_stages.py.

① Sequential vs Parallel — the test stage wall-clock (sum vs max)

What to do: the Gantt bars show every job on a shared 6m00s timeline. In sequential mode the test jobs (unit, integration, lint) stack one after another. In parallel mode they overlap — the group ends when the slowest job (integration, 1m30s) finishes.

② Matrix build — Python 3.10/3.11/3.12 × Ubuntu/Mac (6 cells, all parallel)

What this shows: a matrix fans one job out to 3 × 2 = 6 cells. Each cell is independent and runs at the same time. Sequential total = sum of all cells; parallel total = max cell.

③ Caching — cache HIT skips the 45s dependency download

What this shows: the cache is keyed on the lockfile hash. On a HIT, the 45s download becomes 0s — build drops from 1m32s to 48s.
