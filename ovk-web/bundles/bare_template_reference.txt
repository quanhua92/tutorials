# bare_template — web provenance log
# Every URL has a "Verifies:" line stating exactly what it confirms.
# STEP 2 of HOW_TO_RESEARCH.md is non-skippable; ≥2 sources per concept.

https://developer.mozilla.org/en-US/docs/Web/HTML/Element/template
  Verifies: the HTML <template> element's content is NOT rendered by default — its read-only `.content` property holds an inert DocumentFragment (the DOM subtree HF extracts and mounts). Also verifies that <html>, <head>, and <body> opening/closing tags INSIDE a <template> are syntax errors the HTML parser ignores — i.e. the <template> is the inert carrier, and a real <html>/<body> wrapper around it (outside the template) is the authoring mistake that breaks HF extraction.

https://hyperframes.heygen.com/concepts/compositions
  Verifies: HyperFrames' runtime "automatically fetches the file, extracts the <template> content, mounts it, executes scripts, and registers the timeline" and "each external composition file wraps its content in a <template> tag" — i.e. the bare <template> IS HF's native sub-composition format (the external-file example shown there is a bare <template>, no <html> wrapper). This is exactly what RFC §5.5 cites as "zero translation at export" (§10).

(in-repo) docs/rfc-0001.md §5.4 and §5.5
  Verifies: the slide index.html format — a bare <template> wrapping <div data-composition-id="__SLIDE_ID__" data-width="1920" data-height="1080"> with content + <style> + <script> — and the decision record "Why bare <template> compositions" (HF extracts <template> content; an <html> wrapper causes a blank render; using it as the source format means zero translation at export, §10).

(in-repo) docs/AGENTS.md "Layout file format — CRITICAL" / "Why bare <template> (no <html> wrapper)" / "Why NOT data-variable-values / getVariables()" / "Why NOT flex/absolute centering" / "Sub-composition model"
  Verifies: the export-target contract the editor must emit — layout files MUST be bare <template> (no <html>/<body>/data-composition-variables); an <html> wrapper causes HF to NOT extract content and the sub-comp renders BLANK (verified HF v0.7.3); getVariables() returns {} in v0.7.3 sub-comps so CSS is scoped via [data-composition-id="__SLIDE_ID__"] and values are stamped via __PLACEHOLDER__; vertical centering must use text-align:center + padding-top:XXvh, NOT flex/absolute.
