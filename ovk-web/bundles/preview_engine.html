Pinned interpolation values (match preview_engine.ts Section C)

Between two keyframes a timeline evaluates lerp(a,b,t) = a + t*(b-a) (Wikipedia, "Linear interpolation"). Multi-keyframe = piecewise lerp.

The @1.5 → 0.75 row is the multi-segment case: localT=(1.5−1)/(2−1)=0.5 → lerp(1.0, 0.5, 0.5) = 0.75.

Why it works this way — the load-bearing rules

RuleMechanismWhat breaks without it
tl.time(t), not tl.play()Seek the PAUSED timeline per paint; playhead is a setter ("jumping to new value just like seek()" — GSAP docs).play() races GSAP's clock vs the editor playhead; preview stops being a pure function of t.
paused: true on registerSlide <script> creates gsap.timeline({paused:true}) → window.__timelines[id] = tl (AGENTS.md).A running timeline self-advances; same t yields different frames depending on wall-clock.
lerp(a,b,t) = a + t*(b-a)Linear interpolant between two keyframes (Wikipedia). Easing only remaps t before the lerp.Hand-verifying a tween under default power1.out seems to disagree — it doesn't, t is just remapped.
real-time, not frame-accuraterAF-paced seeks; dt variable → frames repeat/skip. Export is the rigorous path (§10/§11).Locking preview to 1/fps makes scrubbing sluggish for zero fidelity gain.
one SlideRendererPreview + export share load/render_at/duration (§9.4). HF is a pluggable primitive, not a black box.Two engines → "it looked different when I exported" — the drift this architecture exists to kill.
