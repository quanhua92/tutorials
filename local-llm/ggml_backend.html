GGML Compute Graph & Backend — interactive 📖 GGML_BACKEND.md 📄 ggml_backend.py

PyTorch runs ops eagerly; GGML builds a DAG, then a backend topo-sorts it, allocates once, and runs kernels in order. Explore four ideas interactively: build & compute a graph (drag the input), topological sort (post-order DFS vs Kahn’s), backend dispatch (same graph on CPU/Metal/CUDA), and op fusion (the bandwidth win). All values recomputed live in JS — identical to ggml_backend.py.

② Topological sort — post-order DFS vs Kahn’s

ggml_build_forward_expand does a post-order DFS from the output (deps before consumers, one pass over src[]). Kahn’s algorithm emits zero-in-degree tensors first (BFS). Both are legal topo orders; ggml uses the DFS form.

④ Op fusion — mul → add → silu in ONE kernel

Consecutive element-wise ops fuse into a single kernel: intermediates (T1, T2) stay in registers. The math is exact. For a 2-element chain: separate = 64B traffic, fused = 32B (−50%).
