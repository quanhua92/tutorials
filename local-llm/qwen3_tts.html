Qwen3-TTS — end-to-end multi-codebook LM for streaming TTS 📖 QWEN3_TTS.md 📄 qwen3_tts.py

"Speech generation as language modelling." Qwen3-TTS is a single transformer that autoregressively predicts a short grid of discrete audio tokens — the Qwen3-TTS-Tokenizer-12Hz compresses audio to 12 frames/s × 4 codebooks = 48 tokens/s (480 bps, 533× vs PCM). No cascade to compound errors, no DiT denoising loop to block streaming — so the first audio packet ships at 97 ms. Plus voice clone (3 s ref), design (natural language), and control (instructions). ① Multi-codebook tokenizer — 12 Hz × 4 codebooks = 48 tokens/s

What this shows: one second of audio becomes a 12×4 grid of integers. Each column is a frame (83.33 ms of audio); each colored cell is one codebook id. The 4 codebooks are parallel streams that together reconstruct near-losslessly. This grid is what the LM predicts.
codebook 0 (phonetic) codebook 1 (prosody) codebook 2 (timbre) codebook 3 (acoustic env)
token rate = Hz × codebooks. bitrate = token rate × 10 bits. compression vs 16kHz PCM. Same formula as qwen3_tts.py Section A.

② Non-DiT — one LM, no cascade, no denoising loop

What this shows: cascade = 2 stages (errors compound); DiT = 25 denoising passes (whole audio, blocks streaming); Qwen3 LM = 1 pass per frame, each frame final when emitted. That finality is why streaming works.
cascade accuracy = stage1 × stage2 (compounds). DiT revisits the whole audio per step. Qwen3 revisits once per 83.33 ms frame — and ships it. Mirrors qwen3_tts.py Section B.

③ Dual-track streaming — first packet at 97 ms (gold)

What this shows (the gold value): the streaming track ships frame 0 after 97 ms (12+35+20+30), then streams at realtime. The non-streaming track must generate ALL frames first — its latency scales with audio length. Streaming latency is fixed.
green = audio shipped to user. orange = buffered (non-streaming waits for all). First packet breakdown: 12 (text) + 35 (prefill) + 20 (gen frame) + 30 (decode) = 97 ms. Mirrors qwen3_tts.py Section C.

④ Voice design & control — description → parameters

What this shows: a natural-language description is parsed into a voice parameter vector (emotion / pace / volume / accent / gender). For control, the instruction overlays only the matching fields on a base voice. Clone needs 3 s of reference.
design = description → new voice. control = instruction overlaid on base (only matching fields change). Clone = 3 s reference → embedding. Mirrors qwen3_tts.py Section D.
