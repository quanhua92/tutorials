① Dense vs Paged memory map — drag tokens used

Dense reserves max_len=16 up-front (the 8192 problem in miniature). Paged uses page_size=4.
DENSE — one pre-allocated slab per request (grey = wasted shelf space, reserved but never written)
PAGED — only the pages you need; waste < 1/page_size (last page partial, shown grey)
Caption: dense lights up its full max_len shelf the moment a request arrives — most of it stays dark forever. Paged lights up only as many pages as the request actually fills. At used=1: dense wastes 15/16 = 93.75% (the 512/8192 example in miniature); paged wastes 3/4 = 25%. Scale dense to max_len=8192, used=512 and you get exactly the 93.75% from PAGED_ATTENTION.md §1.

③ Block table: logical position → physical (page_id, slot) for request A

For each logical token seat t in A: page_idx = t//PS, slot = t%PS, page_id = block_table[page_idx]. This is the OS virtual→physical translation. Walk it below.
Caption: logical seats 0,1 live in physical page 0; logical seat 2 lives in physical page 2 — that's the non-contiguous storage the block table rescues. The kernel's indirect K address is k_idx = ((page_id*H_kv + head)*page_size + slot)*D + c — see PAGED_ATTENTION.md §6.
