
========================================================================
PART I  --  SOLID: five principles, each BAD vs GOOD
========================================================================
    Each section shows a violation, then the refactor, then asserts the
    observable behavior is preserved (LSP asserts the contract is restored).

========================================================================
S. SINGLE RESPONSIBILITY (SRP) -- BAD: one class, three jobs
========================================================================
BadEmployee.calculate_pay()  -> 1000.0
BadEmployee.save(db)         -> db={'Alice': 1000.0}
BadEmployee.format_report()  -> Alice: $1000.0

========================================================================
S. SINGLE RESPONSIBILITY (SRP) -- GOOD: one reason to change per class
========================================================================
PayCalculator.pay(emp)       -> 1000.0
EmployeeRepository.save(..)  -> db={'Alice': 1000.0}
PayReporter.line(emp, pay)   -> Alice: $1000.0
  [check] OK   SRP split preserves pay (40 * 25.0 = 1000.0)

========================================================================
O. OPEN/CLOSED (OCP) -- BAD: add a tier = edit the function
========================================================================
bad_discount(100, 'regular')   -> 100.0
bad_discount(100, 'vip')       -> 90.0
bad_discount(100, 'wholesale') -> 80.0

========================================================================
O. OPEN/CLOSED (OCP) -- GOOD: new tier = new class, no edit to engine
========================================================================
engine.price(100, RegularTier  ) -> 100.0
engine.price(100, VipTier      ) -> 90.0
engine.price(100, WholesaleTier) -> 80.0
engine.price(100, StudentTier  ) -> 85.0
  [check] OK   OCP engine matches if/elif; StudentTier added with zero edits to engine

========================================================================
L. LISKOV SUBSTITUTION (LSP) -- BAD: Square breaks Rectangle's contract
========================================================================
resize_to(BadRectangle, 5, 10) -> area=50.0   (5*10=50, OK)
resize_to(BadSquare,   5, 10) -> area=100.0   (expected 50, got 100! LSP BROKEN)

========================================================================
L. LISKOV SUBSTITUTION (LSP) -- GOOD: shared Shape, no false is-a
========================================================================
GoodRectangle .area() -> 50.0
GoodSquare    .area() -> 49.0
GoodRectangle .area() -> 6.0
total_area(shapes)              -> 105.0
  [check] OK   LSP fix: GoodSquare no longer pretends to be a Rectangle; Shape is the safe abstraction

========================================================================
I. INTERFACE SEGREGATION (ISP) -- BAD: fat interface forces dead stubs
========================================================================
BadRobot.work() -> beep boop, working
BadRobot.eat()  -> NotImplementedError: robots do not eat

========================================================================
I. INTERFACE SEGREGATION (ISP) -- GOOD: clients depend only on what they use
========================================================================
run_shift(GoodHuman) -> coding
run_shift(GoodRobot) -> beep boop, working
  [check] OK   ISP: GoodRobot implements Workable only; no dead eat()/sleep() stubs

========================================================================
D. DEPENDENCY INVERSION (DIP) -- BAD: high-level depends on low-level concrete
========================================================================
BadNotifier.notify('deploy ok') -> emailed: deploy ok

========================================================================
D. DEPENDENCY INVERSION (DIP) -- GOOD: both depend on an abstraction, injected
========================================================================
GoodNotifier(Email).notify('deploy ok') -> emailed: deploy ok
GoodNotifier(SMS).notify('deploy ok')   -> sms: deploy ok
  [check] OK   DIP: email result preserved; SMS swaps in with zero edits to GoodNotifier

PART II  --  THE 5-PHASE DESIGN PROCESS CHECKLIST
========================================================================
    Understand Requirements -> Identify Entities -> Define Relationships
    -> Apply Patterns -> Review Tradeoffs   (the SEDIE shape)

------------------------------------------------------------------------
Phase 1  UNDERSTAND REQUIREMENTS  -- name the boundary
------------------------------------------------------------------------
  Worked example: a Notification Service that fires on order events.
  IN scope : send a message on order placed / shipped.
  OUT scope: marketing campaigns, A/B delivery, template editor.
  Actors   : OrderService (caller), end customer (receiver).
  Use cases: notify(channel, message) -> delivery receipt.
  Constraint: must swap channel (email/sms) without touching callers.
  [x] boundary stated   [x] assumptions written   [x] alignment

------------------------------------------------------------------------
Phase 2  IDENTIFY ENTITIES  -- nouns to candidates, filter the weak
------------------------------------------------------------------------
  Nouns (candidates): Notifier, Message, Channel, EmailSender, SmsSender, Receipt.
  Verbs (methods)   : notify(), send(), format(), acknowledge().
  Filter:
    KEEP   Notifier (lifecycle + policy), MessageSender (varies), Receipt (value).
    COLLAPSE 'Channel' -> enum/value object, not a class (no behavior).
  Entity list: Notifier, MessageSender(ABC), EmailSender, SmsSender, Receipt.

------------------------------------------------------------------------
Phase 3  DEFINE RELATIONSHIPS  -- is-a / has-a / uses-a
------------------------------------------------------------------------
  EmailSender  is-a  MessageSender   (true specialization, swap target)
  SmsSender    is-a  MessageSender   (true specialization, swap target)
  Notifier     has-a MessageSender   (composition; injected at construct)
  Notifier    uses-a Message         (passed into notify(), not owned)
  Receipt      value  (immutable record of one delivery)

------------------------------------------------------------------------
Phase 4  APPLY PATTERNS  -- requirement signal -> SOLID -> pattern
------------------------------------------------------------------------
  Signal 'channel swaps at runtime'   -> OCP/DIP -> Strategy (MessageSender).
  Signal 'caller must not know how'   -> DIP      -> inject the abstraction.
  Signal 'one delivery record shape'  -> DRY      -> Receipt value object.
  Code lives in PART I (DIP section): GoodNotifier(EmailSender()).

------------------------------------------------------------------------
Phase 5  REVIEW TRADEOFFS  -- the weighted matrix (see PART III)
------------------------------------------------------------------------
  Pick the channel using the matrix below, not a gut call.
  Then prove extensibility: a new PushSender subclass needs zero edits to Notifier.
  [check] OK   5-phase process: each phase left a visible artifact on the board
