==========================================================================
  SECTION B: WHY DECODE IS BANDWIDTH-BOUND - weights read every token
==========================================================================
During DECODE (generating one token at a time, batch=1), the engine must
read (almost) ALL the active model weights out of memory ONCE per token.
Compute is idle waiting on memory. So:

    decode_tps ~= bandwidth_GBps / bytes_read_per_token

Doubling bandwidth doubles tok/s. Doubling FLOPS does ~nothing for decode
(it helps PREFILL - see Section D). This is the roofline model.

Worked: a DENSE 70B Q4_K_M model. weight = 70 * 4.5 / 8 = 39.38 GB.
Theoretical peak decode (bw / weight, efficiency=1.0) per hardware:
| hardware                 | bw(GB/s) | peak tps | est tps* |
|--------------------------|----------|----------|----------|
| 3x RTX 3090              |      936 |    23.8  |   13.1   |
| AMD Strix Halo           |      273 |     6.9  |    3.8   |
| DGX Spark (GB10)         |      273 |     6.9  |    3.8   |
| Mac Mini M4 Pro          |      273 |     6.9  |    3.8   |
| Mac Studio M3 Ultra      |      819 |    20.8  |   11.4   |
| Mac Studio M4 Max        |      546 |    13.9  |    7.6   |
| RTX 4090                 |     1008 |    25.6  |   14.1   |
| RTX 5090                 |     1792 |    45.5  |   25.0   |
| RX 7900 XTX              |      960 |    24.4  |   13.4   |
  * est = peak * 0.55 (KV-cache reads + attention overhead)

Two facts this table forces on you:
  1. 3x3090 (936 GB/s) decodes 3.4x faster than
     DGX Spark (273 GB/s) - PURELY because of bandwidth. Same model.
  2. Mac Mini M4 Pro (273 GB/s) is bandwidth-equal to DGX Spark
     (273 GB/s) - so it decodes a 70B at the SAME speed, for ~1/3 $.

[check] decode scales linearly with bandwidth (3x3090/DGX == bw ratio) :  OK

CAVEAT: this is for DENSE models. MoE models (GPT-OSS, DeepSeek, Qwen3-MoE)
read only the ACTIVE experts per token (~7-8 GB for GPT-OSS 120B, not the
full 60 GB on-disk), so they decode MUCH faster than a dense 120B would -
but the bandwidth-bound law still holds (Section C proves it).

==========================================================================
  SECTION D: PREFILL vs DECODE - compute-bound vs bandwidth-bound
==========================================================================
LLM serving has TWO phases with OPPOSITE bottlenecks. Mixing them up is
why people buy the wrong hardware.

| phase   | what it does              | bottleneck    | wins on...      |
|---------|---------------------------|---------------|-----------------|
| PREFILL | process the WHOLE prompt   | COMPUTE(FLOPS)| tensor cores    |
| DECODE  | emit ONE token at a time   | MEMORY(bw)    | bandwidth (GB/s)|

PREFILL batches the entire prompt into one giant matmul - compute units are
the limit, so FLOPS (tensor-core throughput) sets the speed. DECODE emits one
token at a time (batch=1), so each weight is read once per token and memory
BANDWIDTH is the limit - FLOPS sit idle waiting for data.

The GPT-OSS 120B benchmark makes the split vivid:
| system               | prefill | decode  | who wins prefill?   | who wins decode?  |
|----------------------|---------|---------|---------------------|------------------|
| DGX Spark (GB10)     |   1723  |  38.55  | 1 PFLOP FP4 -> WINS | 273 GB/s -> slow |
| AMD Strix Halo       |    340  |  34.13  | ~50 TOPS -> slow    | 273 GB/s -> slow |
| 3x RTX 3090          |   1642  | 124.03  | fast compute        | 936 GB/s -> WINS |

  DGX Spark prefill / Strix Halo prefill = 5.1x
  -> Spark's 1 PFLOP FP4 tensor cores crush prefill (compute-bound).
  -> Yet they DECODE at nearly the same speed (both 273 GB/s, bandwidth-bound).

RULE OF THUMB: if your workload is LONG PROMPTS / short answers (RAG, code
completion, document Q&A), prefill dominates -> buy COMPUTE (DGX Spark,
tensor cores). If it is CHAT / long generation, decode dominates -> buy
BANDWIDTH (3x3090, Mac Studio). Section F shows you can split the two.

Pick by your LARGEST model and your budget. Memory capacity is a hard gate

[check] Strix Halo beats DGX Spark on $/tps (half price, same bandwidth) :  OK

DECISION TREE (by what you actually want to run):

  Want <= 14B Q4 (chat, fast)?
    -> single discrete GPU: RTX 5090 (32GB/1792 GB/s) or RTX 4090 (24GB).
       Best tok/s per dollar; bandwidth is huge; capacity is the only cap.

  Want 70B Q4 (strong reasoning, fits in ~40GB)?
    -> 3x RTX 3090 (72GB, 936 GB/s): fastest decode, cheapest speed.
    -> Mac Studio M4 Max (128GB, 546 GB/s): single-box, quiet, MLX.
    -> offload on a single 24GB card is viable but slow (see gpu_offload).

  Want 120B-200B+ (GPT-OSS, command-R, huge context)?
    -> Mac Studio M3 Ultra (512GB): the ONLY single box that holds 200B+ Q4.
    -> DGX Spark / Strix Halo (128GB): hold up to ~120B Q4, CUDA vs ROCm.

  Want the CUDA ecosystem + fast prefill (dev, fine-tuning, agents)?
    -> DGX Spark. You pay for the stack, not the raw decode speed.

==========================================================================
  SECTION F: THE HYBRID TRICK - disaggregate prefill from decode
==========================================================================
Since prefill and decode have OPPOSITE bottlenecks (Section D), no single
box is optimal at both. EXO Labs' insight: network TWO boxes and let each
do the phase it is best at. Source: blog.exolabs.net (2026).

    DGX Spark (1 PFLOP FP4)  -->  fast PREFILL   [compute-bound phase]
    Mac Studio M3 Ultra      -->  fast DECODE     [bandwidth-bound phase]
                networked over 200 Gbps ConnectX-7

Measured result on GPT-OSS 120B: this disaggregated pair runs ~2.8x faster
END-TO-END than the Mac Studio alone (the Spark's prefill removes the Mac's
weakness; the Mac's 819 GB/s decode removes the Spark's 273 GB/s weakness).

  Mac Studio alone:  prefill ~420 t/s, decode 70.8 t/s
  Hybrid:            prefill ~1723 t/s (Spark), decode 70.8 t/s (Mac)
  -> prefill speedup ~4.1x, decode unchanged; end-to-end ~2.8x (EXO Labs).
[check] hybrid prefill (Spark) beats Mac-alone prefill :  OK  (1723.07 vs ~420)
[check] hybrid decode (Mac) beats Spark-alone decode :  OK  (70.79 vs 38.55)

This is PREFILL/DECODE DISAGGREGATION at the desktop scale - the same idea
datacenters use (separate prefill pools from decode pools). Cross-ref
llm/DISAGGREGATED_SERVING for the datacenter version. The win comes from
NOT forcing one machine's memory system to serve two opposite bottlenecks.
