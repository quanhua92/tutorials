① Dense vs Paged memory map — drag tokens used

Dense reserves max_len=16 up-front (tiny demo of the 8192 problem). Paged uses page_size=4.
DENSE — one pre-allocated slab per request (grey = wasted shelf space, reserved but never written)
PAGED — only the pages you need; waste < 1/page_size (last page partial, shown grey)
Caption: dense lights up its full max_len shelf the moment a request arrives — most of it stays dark forever. Paged lights up only as many pages as the request actually fills. At used=1: dense wastes 15/16 = 93.75% (the 512/8192 example in miniature); paged wastes 3/4 = 25%. Scale dense to max_len=8192, used=512 and you get exactly the 93.75% from KV_CACHE.md §4.

③ Offset correctness: cached decode == full precompute (the RoPE link)

What this proves: token 3's cached K (head h=0, first 4 dims) is byte-identical whether you compute all 4 tokens at once, or prefill 3 then decode token 3. The cache stores already-rotated K,V, so decode must rotate the new token at its TRUE position (offset=3), not at m=0. The WRONG path rotates at m=0 → diverges — that's the proof the offset is mandatory (see RoPE §10).
Caption: the three correct rows (green) show the SAME K[3] vector — the cache changes when K[3] is computed, never what it is. The wrong-offset row (red) differs — proving slice(3,4) is not optional. Numbers match KV_CACHE.md §7.
