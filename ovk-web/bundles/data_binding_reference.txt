# data_binding — web provenance log
# Every URL has a "Verifies:" line stating exactly what it confirms.
# STEP 2 of HOW_TO_RESEARCH.md is non-skippable; ≥2 sources per concept.

https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/String/replace
  Verifies: the special replacement-string patterns ($$, $&, $`, $', $n, $<Name>) that String.prototype.replace / replaceAll apply when the replacement is a STRING. Critically, "$n and $<Name> are only available if the pattern argument is a RegExp object. If the pattern is a string ... the pattern will be replaced as a literal." So a __FIELD__ stamp with a string search + string replacement corrupts on $$, $&, $`, $' but NOT on $5. Also verifies: "The above-mentioned special replacement patterns do not apply for strings returned from the replacer function" — the function-form fix used by stampSafe().

https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/String/replaceAll
  Verifies: String.prototype.replaceAll semantics — when the pattern is a string, ALL occurrences are replaced (vs replace() which does only the first), and the same special replacement-string patterns apply. This is why stamping uses replaceAll (a placeholder may appear many times in one template, e.g. __SLIDE_ID__ in CSS + script + attribute).

https://developer.mozilla.org/en-US/docs/Web/HTML/Element/template
  Verifies: the HTML <template> element — "content held in <template> is not rendered on page load, but can be instantiated later" — which is the static-site / placeholder-substitution pattern OpenVideoKit uses: layouts/ are bare <template> files with __PLACEHOLDER__ markers, stamped at edit/export time then mounted by HyperFrames.

(in-repo) docs/rfc-0001.md §5.6 "Data binding (how fields reach the HTML)"
  Verifies: the three stamping modes — (1) "Stamps on structural change (slide add/remove/reorder, layout swap)"; (2) "Live-binds text fields for responsive editing (textarea -> immediate preview update via a light DOM patch, no full re-stamp)"; (3) "Fully re-stamps at export." Also: "Each field id uppercases to its placeholder: field title -> __TITLE__", and "Stamping ... remains the data-injection mechanism, because HF's getVariables() returns {} in sub-compositions."

(in-repo) docs/rfc-0001.md §5.5 "Why NOT data-variable-values / getVariables()" + §10 step 2 "Stamp"
  Verifies: the rationale (HF getVariables() returns {} in v0.7.3 sub-compositions) and the export step that consumes stamping ("Stamp: replace __SLIDE_ID__ and every __FIELD__ in each composition from the corresponding slide index.json").

(in-repo) docs/AGENTS.md "Placeholder convention" + "Layout field types"
  Verifies: the placeholder convention table (__SLIDE_ID__, __TITLE__, __BODY__, __IMAGE__, __STEP__; "Each field ID is uppercased: field title -> __TITLE__") and the image-field rule ("image -> byte swap to assets/, __FIELD_ID__ path replacement" — the value is a PATH, not text).
