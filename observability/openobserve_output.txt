========================================================================
SECTION A - Architecture: Rust + S3 + Parquet
========================================================================

OpenObserve (O2) is a single Rust binary that does logs, metrics, and
traces. The storage tier is object storage (S3/MinIO/GCS/Azure Blob):
Parquet columnar files, queried directly off S3. No mandatory local disk
for data -- only a small WAL buffer on the ingester. This is the inverse
of Elasticsearch, which ties data to local disk + JVM heap.

Five core components (HA mode). In single-node mode all five collapse
into ONE process:

  Router        Client-facing HTTP/gRPC gateway + serves the web UI. A thin
                proxy: routes ingest -> Ingester, query -> Querier.
  Ingester      Receives ingest, parses, schema-evolves, writes WAL -> Memtable
                (256MB) -> Immutable -> local Parquet -> flush to S3. 3 buffers:
                Memtable, Immutable, wal Parquet not-yet-on-S3.
  Compactor     Merges small S3 Parquet files into big ones (<= 2 GB) so
                queries scan fewer files. Also enforces retention + deletions.
  Querier       FULLY STATELESS. Scans S3 Parquet. A LEADER partitions the file
                list across WORKER queriers over gRPC, then merges results.
                Caches Parquet in RAM (default 50% of free memory).
  AlertManager  Runs scheduled alert queries + report jobs; fires notifications
                to destinations (Slack/email/webhook/Teams/PagerDuty).

HA-mode dependencies (single-node uses SQLite, no extra deps):
  NATS        - cluster coordinator + node discovery + event bus
  PostgreSQL  - metadata: orgs, users, functions, alert rules,
                stream schema, file list (index of Parquet files)
  S3/MinIO    - ALL Parquet data (the actual telemetry)

Why Rust (vs Java/ES, vs Go/Loki):
  * Zero-cost abstractions + no GC pauses -> predictable ingest latency
  * SIMD-accelerated string parsing -> fast line-by-line ingest decode
  * ~50% lower resource use than Elasticsearch at equal ingest (O2 claim)
  * Memory-efficient: no JVM heap tax, no per-object overhead

Storage model comparison (the whole cost story):

  Engine         Lang  Storage         Format        Index      
  -------------  ----  --------------  ------------  -----------
  OpenObserve    Rust  S3 (object)     Parquet col.  Tantivy FTS
  Elasticsearch  Java  local EBS/disk  inverted idx  Lucene     
  Loki           Go    S3 (chunks)     gzip chunks   labels only

Single-node ingest (pinned, official, Apple M2): 31 MB/s
  = 2,678 GB/day  (31 x 86400 / 1000)
  ~ 3.9 MB/s per vCPU (M2 ~8 cores)

SECTION B - Day 0: Deploy with S3 (single-node + object storage)
========================================================================

Goal: one OpenObserve container, data on S3/MinIO, ready to ingest in
under 15 minutes. Single-node-with-S3 uses SQLite for metadata (no
Postgres/NATS needed) but stores all telemetry on object storage -- so
storage is durable and cheap from day one.

--- docker-compose.yml (single node + S3/MinIO) ---

version: "3.8"

services:
  openobserve:
    image: openobserve/openobserve:latest
    container_name: openobserve
    ports:
      - "5080:5080"          # Web UI + HTTP ingest API
    environment:
      # --- root credentials (created on first boot) ---
      ZO_ROOT_USER_EMAIL: "admin@openobserve.dev"
      ZO_ROOT_USER_PASSWORD: "admin123Complex#"

      # --- S3 / object storage (MinIO here; swap for AWS S3) ---
      ZO_S3_PROVIDER: "minio"           # aws | minio | gcs | azure
      ZO_S3_SERVER_URL: "http://minio:9000"
      ZO_S3_REGION: "us-east-1"
      ZO_S3_BUCKET: "openobserve"
      ZO_S3_ACCESS_KEY: "minioadmin"
      ZO_S3_SECRET_KEY: "minioadmin"

      # --- ingest tuning (defaults, shown for clarity) ---
      ZO_MAX_FILE_SIZE_IN_MEMORY: "256"  # MB -> Memtable -> Immutable
      ZO_MAX_FILE_SIZE_ON_DISK: "128"    # MB -> WAL cap
      ZO_FILE_PUSH_INTERVAL: "10"        # s -> merge + push to S3
      ZO_COMPACT_MAX_FILE_SIZE: "2048"   # MB (2 GB) merged file ceiling
    volumes:
      - o2-data:/data                   # local WAL buffer only (data lives on S3)
    depends_on:
      - minio

  minio:                                # S3-compatible object store
    image: minio/minio:latest
    command: server /data --console-address ":9001"
    ports:
      - "9000:9000"
      - "9001:9001"                      # MinIO console
    environment:
      MINIO_ROOT_USER: "minioadmin"
      MINIO_ROOT_PASSWORD: "minioadmin"
    volumes:
      - minio-data:/data

volumes:
  o2-data:
  minio-data:

SECTION D - Day 2: Scale, Compaction, Alerts, Dashboards
========================================================================

SCALING  add queriers (stateless, no data on them)

  Queriers are fully stateless: they only scan S3 + cache Parquet in RAM.
  Double query throughput by doubling querier pods -- no rebalancing, no
  shard pain. A LEADER querier partitions the file list across WORKERs

  over gRPC and merges partial results.

  1 querier(s): 100 files -> 100 files/querier
  2 querier(s): 100 files -> 50 files/querier
  5 querier(s): 100 files -> 20 files/querier

COMPACTION  merge small S3 Parquet files -> bigger files

  before: 1200 files x 2 MB = 2,400 MB (avg 2 MB/file)
  compactor target ceiling: ZO_COMPACT_MAX_FILE_SIZE = 2048 MB (2 GB)
  after : 2 files (1 x 2048 MB + 1 tail) ...
          = 2 files x up to 2048 MB = 2,400 MB
  file-count reduction: 1200 -> 2 (600x fewer files to plan/scan)
  exact merged sizes (MB): [2048, 352]

SECTION E - Storage & Cost Analysis (S3 Parquet, model-based)
========================================================================

Parquet is columnar + dictionary/RLE/zstd encoded. For repetitive log
fields (level, service, host) it crushes redundancy. Representative
compression on logs: O2 ~15x, Loki ~8x
(gzip chunks). ES keeps an inverted index + replicas on local EBS.

Assumptions (transparent, AWS us-east-1 2025 pricing):
  retention            = 30 days
  O2 storage mult.     = 1/15 (compressed Parquet, single copy, on S3)
  Loki storage mult.   = 1/8 (compressed chunks, on S3)
  ES storage mult.     = 2 x 1 (primary+replica, net ~raw, on EBS gp3)
  S3 Standard          = $0.023/GB-mo    S3 IA = $0.0125/GB-mo
  EBS gp3              = $0.08/GB-mo
  (STORAGE ONLY; ES also needs compute that scales with data -- see ROI)

Cost table -- STORAGE cost / month by daily ingest volume:

  ingest/day  O2 (S3)       Loki (S3)     ES (EBS)      O2 vs ES  
  ----------------------------------------------------------------
  1 GB/day    $0.046        $0.086        $4.80            104x
  10 GB/day   $0.460        $0.862        $48.00           104x
  100 GB/day  $4.60         $8.62         $480.00          104x
  1 TB/day    $46.00        $86.25        $4,800           104x

ROI -- when does O2's S3-native model win MASSIVELY?

  * The gap widens with VOLUME and RETENTION. ES cost grows with data
    (more EBS + more data nodes holding indices in RAM). O2 storage is
    ~flat (S3 price) and compute is a handful of stateless nodes that do
    NOT grow with historical data.

  * Cross-over: at ~10 GB/day x 30d retention, ES storage already costs
    ~104x O2. At 1 TB/day it is ~104x on storage ALONE, before
    ES compute. O2's official headline is '140x lower storage cost' vs ES
    (specific configurations); our transparent model shows ~100x+ on storage
    at 1 TB/day, which is the right order of magnitude.

  * O2 does NOT win on: tiny volumes (<1 GB/day, where ES single-node is
    fine), or queries needing ES's mature Lucene relevance ranking.

SCHEMA DETECTION  (auto, from ingested JSON)

  No schema declared up front. On first ingest O2 inspects the JSON
  records, infers Arrow types, and registers the stream schema in
  Postgres (HA) / SQLite (single-node). Example:

    _timestamp   JSON int   -> Arrow Int64
    latency_ms   JSON int   -> Arrow Int64
    level        JSON str   -> Arrow Utf8
    ok           JSON bool  -> Arrow Boolean
    service      JSON str   -> Arrow Utf8

SCHEMA EVOLUTION  (new fields appear -> auto-added)

  day 1: 4 fields  + new: ['_timestamp', 'level', 'log', 'service']
  day 2: 5 fields  + new: ['latency_ms']
  day 3: 6 fields  + new: ['user_id']
  Old Parquet files simply lack the new column; queries return NULL for
  it on those files. No reindex, no downtime.

MULTI-FORMAT  one platform, three signal types

  Metrics   Prometheus remote_write (/api/{org}/prometheus/api/v1/write) ;

            -> span model

PARTITIONING  time-based folders in S3

  Parquet files are laid out by hour/day under the object prefix, e.g.:
    files/logs/default/2024/05/29/00/<uuid>.parquet
    files/logs/default/2024/05/29/01/<uuid>.parquet
  A time-range query only lists the partitions it overlaps -> the file-
  list index (in Postgres/SQLite) prunes the rest BEFORE any S3 scan.
  This is why Querier->S3 stays fast: time predicate -> partition pruning.
