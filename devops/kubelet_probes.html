Kubernetes Kubelet Probes — interactive KUBELET_PROBES.md kubelet_probes.py

The kubelet asks each container three yes/no questions on a schedule. Liveness: fail → kill + restart. Readiness: fail → pull from Service endpoints (no restart). Startup: a one-shot gate that disables the other two until the app has booted. The liveness timeline below is recomputed in JS from the identical model as kubelet_probes.py.

① The three probe KINDS — what a failure actually does

What to do: each KIND answers a different question. The HANDLER (http/tcp/exec) only decides how the question is asked — every handler collapses to one pass/fail boolean, so the KIND is what matters.

liveness

fail threshold → KILL + RESTART container
  • "Are you still alive?" (deadlock, hung process)
  • about the container
  • raises restartCount

readiness

fail → remove from Service endpoints
  • "Ready for traffic?" (warming cache, dep down)
  • about the network — routes around you
  • never restarts

startup

gates liveness+readiness until it succeeds
  • "Finished booting?" (slow JVM, heavy init)
  • one-shot: runs once, then disabled
  • prevents boot-time kills
Handlers (all reduce to one bool): httpGet → status 200-399 · tcpSocket → TCP connect ok · exec → exit 0.

② Liveness timeline — 3 consecutive fails (failureThreshold) → restart

What to do: drag failureThreshold and period and watch the kill move. The app is unhealthy during the gray window; each probe in that window is a fail. Hit the threshold and the kubelet restarts the container. The gold bar below confirms the page matches kubelet_probes.py Section B.
probe OK (200) probe FAIL (500) RESTART app unhealthy window
Formula: restart_tick = first_fail + (failureThreshold − 1) × periodSeconds. After a restart the failure window is relative to the new container start, so a transient blip yields exactly one restart.

③ Startup probe — save a slow-booting app from a boot-time kill

What this proves: same slow JVM (boots at t≥45s), two configs. Without a startup probe, liveness fires while the app is still booting and kills it (crash-loop). With one, liveness is held back until the startup probe passes. Toggle and watch restartCount.
JVM boots at t_rel ≥ 45 · startup FT=6 · liveness FT=3, period=10
This is exactly kubelet_probes.py Section D: no-startup crash-loops (boot resets each restart), with-startup = 0 restarts.
