========================================================================
  SECTION A: WHY quantize - memory of a weight matrix (FP16 vs W4A16)
========================================================================
One linear layer weight [N=896, E=896] (Qwen3-0.5B hidden dim):
  FP16 (bfloat16): 896*896*2 =    1,605,632 bytes = 1.606 MB
  W4A16 group_size=  8: packed=  401,408 B + scales/biases=401,408 B =     802,816 bytes = 0.803 MB  -> 2.00x smaller
  W4A16 group_size=128: packed=  401,408 B + scales/biases=25,088 B =     426,496 bytes = 0.426 MB  -> 3.76x smaller

Decode is BANDWIDTH-bound: weights stream from HBM every token,
activations are computed and stay on-chip. So shrinking the STATIC
weight footprint (4x) directly speeds up inference; keeping
activations in 16-bit preserves accuracy (per LLM.int8() findings).

Tiny hero example (group_size=8, one row of 8 weights):
  FP16    : 8 * 2              = 16 bytes
  W4A16   : 1 uint32 (4 bytes) + scale(2)+bias(2) = 8 bytes  -> 2.0x
  (group_size=128 amortizes the scale/bias overhead -> ~3.8x)

========================================================================
  SECTION B: quantize ONE row of 8 floats -> scale, bias, 8 int4 values
========================================================================
Input weight row w (group_size=8, bits=4):
  w = [-1.8, 0.55, -0.35, 1.2, -0.85, 0.25, -1.15, 1.05]

Step 1-4 of the MLX affine quantizer (see quantize_group docstring):
  w_min=-1.80  w_max=+1.20
  mask = |w_min|>|w_max| = 1.80>1.20 = True
  edge = w_min = -1.80   (the 'heavier' extreme -> maps to int4 with ZERO error)
  scale = edge/q0 = -1.80/-9 = 0.200000
  bias  = edge    = -1.800000   (in FLOAT units, same scale as w)

Per-element quantize:  w_int = clip(round((w - bias)/scale), 0, 15)

| k |   w[k]   | (w-bias)/scale | round | w_int | dequant  |  error   |
|---|---------|----------------|-------|-------|----------|----------|
| 0 | -1.80   |        +0.0000 |     0 |     0 | -1.8000  | +0.0000   |
| 1 | +0.55   |       +11.7500 |    12 |    12 | +0.6000  | -0.0500   |
| 2 | -0.35   |        +7.2500 |     7 |     7 | -0.4000  | +0.0500   |
| 3 | +1.20   |       +15.0000 |    15 |    15 | +1.2000  | +0.0000   |
| 4 | -0.85   |        +4.7500 |     5 |     5 | -0.8000  | -0.0500   |
| 5 | +0.25   |       +10.2500 |    10 |    10 | +0.2000  | +0.0500   |
| 6 | -1.15   |        +3.2500 |     3 |     3 | -1.2000  | +0.0500   |
| 7 | +1.05   |       +14.2500 |    14 |    14 | +1.0000  | +0.0500   |

RESULT: scale=0.2000  bias=-1.8000  w_int=[0, 12, 7, 15, 5, 10, 3, 14]
Note: the extreme (w[0]=-1.80) maps to w_int=0 and the other
extreme (w[3]=+1.20) maps to w_int=15 - both ZERO error, by
construction (bias=edge, scale=edge/q0). Interior weights drift up
to +/- scale/2 = +-0.1000.

========================================================================
  SECTION C: PACK 8 int4 values into ONE uint32 (bit layout)
========================================================================
w_int = [0, 12, 7, 15, 5, 10, 3, 14]
       = ['0x0', '0xc', '0x7', '0xf', '0x5', '0xa', '0x3', '0xe']  (one nibble each)

Packing:  packed = sum_k  w_int[k] << (4*k)
          bits[4k : 4k+4] = w_int[k]

Bit layout of the uint32 (bit 31 = MSB, bit 0 = LSB):

  bit range  | k | shift | w_int[k] | hex nibble
  ------------|---|-------|----------|-----------
  [ 0:4 ]      | 0 |     0 |        0 | 0x0
  [ 4:8 ]      | 1 |     4 |       12 | 0xc
  [ 8:12]      | 2 |     8 |        7 | 0x7
  [12:16]      | 3 |    12 |       15 | 0xf
  [16:20]      | 4 |    16 |        5 | 0x5
  [20:24]      | 5 |    20 |       10 | 0xa
  [24:28]      | 6 |    24 |        3 | 0x3
  [28:32]      | 7 |    28 |       14 | 0xe

packed (decimal) = 3819304896
packed (hex)     = 0xE3A5F7C0
packed (binary)  = 11100011101001011111011111000000

Read the hex string RIGHT-TO-LEFT to recover w_int in order:
  0xE3A5F7C0  -> nibbles k=0..7 = [0, 12, 7, 15, 5, 10, 3, 14]

========================================================================
  SECTION F: SIGN-CONVENTION PITFALL - MLX (+bias) vs textbook (-zero_point)
========================================================================
Using the hero group: scale=0.2000, bias=-1.8000 (float units),
w_int[1] = 12 (the second weight, w[1]=+0.55).

Three dequant formulas, all claiming to be 'the' affine dequant:

(1) MLX actual (this file, mlx/backend/cpu/quantized.cpp):
    w = scale*w_int + bias = 0.20*12 + (-1.80) = +0.6000  CORRECT

(2) Textbook zero-point (GPTQ/AWQ papers):
    zero_point = -bias/scale = -(-1.80)/0.20 = 9.0000  (INT units!)
    w = scale*(w_int - zero_point) = 0.20*(12 - 9.0000) = +0.6000  CORRECT

(3) Source-guide PROSE 'w = (w_int + bias)*scale' (MISLEADING):
    w = (w_int + bias)*scale = (12 + (-1.80))*0.20 = +2.0400  WRONG!

  (3) is off by 1.4400 - silently. If you plug MLX's
  float-unit `bias` into the textbook `(w_int + X)*scale` shape, you get
  garbage. The two conventions are algebraically equal ONLY when
  zero_point = -bias/scale (note the /scale: int units vs float units).

[check] MLX (1) == textbook (2) when zp=-bias/scale :  OK
[check] source-guide prose (3) != MLX (1)            :  OK (it's wrong)

RULE: pick ONE convention and store its parameters with the right units.
  MLX checkpoint  -> dequant = scale*w_int + bias     (bias: float units)
  GPTQ checkpoint -> dequant = scale*(w_int - zp)     (zp:   int units)
Never cross them.
