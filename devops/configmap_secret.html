Two config objects — ConfigMap (plain) and Secret (base64 in etcd) — delivered by two methods. Env vars are a snapshot (immutable; need a restart). Volume mounts are refreshed (~60s; hot reload). The reload timeline and injection values below are recomputed in JS from the identical model as configmap_secret.py.

① Two objects × two delivery methods

The objects hold config in etcd; the methods push it into the container.
object

ConfigMap

non-sensitive, plaintext in etcd
  • DB_HOST=10.0.0.5, app.yaml
  • created from literals or files
  • no encoding
object

Secret

sensitive, base64-ENCODED in etcd
  • types: Opaque, docker-registry, tls
  • decoded to plaintext inside the pod
  • volumes land on a tmpfs (never on disk)

METHOD 1 — env var

SNAPSHOT at start → IMMUTABLE
A config change needs a pod restart to be seen here.

METHOD 2 — volume mount

REFRESHED periodically → HOT RELOAD
Each key becomes a FILE; kubelet re-syncs every ~60s. No restart.
Same ConfigMap/Secret can be delivered BOTH ways. DB_HOST as env, app.yaml as a file — two paths, one object.

② Hot reload — env stays pinned, volume refreshes

What to do: set the ConfigMap update time and the kubelet refresh period. The app's env value never changes (it's a start-time snapshot); the file value flips at the next refresh tick. Drag and watch the cross-over move.
stale (old value) fresh (new value)
Why it matters: feature flags / throttles that you want to flip WITHOUT a restart MUST be read from a volume (and re-read by the app), never from an env var.
