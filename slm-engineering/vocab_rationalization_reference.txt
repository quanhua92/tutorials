vocab_rationalization_reference.txt
Web provenance log for vocab_rationalization.py / VOCAB_RATIONALIZATION.md.
Every formula, model config, and value claim below is traced to its URL and
verified in >=2 independent sources. This is the audit trail; VOCAB_RATIONALIZATION.md
"## Sources" is its reader-facing mirror.

--------------------------------------------------------------------------------

[1] https://huggingface.co/HuggingFaceTB/SmolLM2-1.7B/raw/main/config.json
    HuggingFaceTB/SmolLM2-1.7B official config.json (Hugging Face)
    Verifies: SmolLM2-1.7B architecture -> vocab_size=49152, hidden_size=2048,
    intermediate_size=8192, num_hidden_layers=24, num_attention_heads=32,
    num_key_value_heads=32, tie_word_embeddings=true. => embed = 49152*2048.

[2] https://huggingface.co/HuggingFaceTB/SmolLM2-135M/raw/main/config.json
    HuggingFaceTB/SmolLM2-135M official config.json (Hugging Face)
    Verifies: SmolLM2-135M -> vocab_size=49152, hidden_size=576,
    intermediate_size=1536, num_hidden_layers=30, num_attention_heads=9,
    num_key_value_heads=3, tie_word_embeddings=true.

[3] https://huggingface.co/blog/smollm
    Hugging Face blog, "SmolLM - blazingly fast and remarkably powerful" (official)
    Verifies: SmolLM family sizes 135M / 360M / 1.7B; "We used a tokenizer trained
    on the Smollm Corpus with a vocab size of 49152"; "For all three models we use
    embedding tying and a context length of 2048 tokens." Confirms V=49152 + tied.

[4] https://huggingface.co/Qwen/Qwen2.5-1.5B/raw/main/config.json
    Qwen/Qwen2.5-1.5B official config.json (Hugging Face)
    Verifies: Qwen2.5-1.5B -> vocab_size=151936, hidden_size=1536,
    intermediate_size=8960, num_hidden_layers=28, num_attention_heads=12,
    num_key_value_heads=2, head_dim=128, tie_word_embeddings=true.

[5] https://huggingface.co/docs/transformers/en/model_doc/qwen2
    Hugging Face Transformers docs - Qwen2 (Gemma2-style authority)
    Verifies: Qwen2 vocab_size default = 151936. (Second source for [4]'s vocab.)

[6] https://huggingface.co/unsloth/gemma-2-2b/raw/main/config.json
    unsloth/gemma-2-2b config.json (mirror of google/gemma-2-2b, ungated)
    Verifies: Gemma-2-2B -> vocab_size=256000, hidden_size=2304,
    intermediate_size=9216, num_hidden_layers=26, num_attention_heads=8,

[7] https://huggingface.co/docs/transformers/en/model_doc/gemma2
    Hugging Face Transformers docs - Gemma2Config (authority)
    Verifies: Gemma2Config defaults -> vocab_size=256000, hidden_size=2304,
    intermediate_size=9216, num_hidden_layers=26, num_attention_heads=8,
    num_key_value_heads=4, head_dim=256; tie_word_embeddings (bool, defaults True).
    Confirms Gemma-2-2B is TIED (the unsloth config omits the field => this default).

[8] https://arxiv.org/abs/2408.00118
    Gemma Team (2024), "Gemma 2: Improving Open Language Models at a Practical Size",
    arXiv:2408.00118 (official report)
    Verifies: Gemma 2 is a 2B / 9B / 27B family using grouped-query attention (GQA)
    and interleaved local/global attention; the 2B and 9B trained with knowledge
    distillation. Confirms the architecture family + the 2B size class.

[9] https://huggingface.co/unsloth/llama-3-8b/blob/main/config.json
    unsloth/llama-3-8b config.json (mirror of meta-llama/Meta-Llama-3-8B, ungated)
    Verifies: Llama-3-8B -> vocab_size=128256, tie_word_embeddings=false (UNTIED).
    Cross-confirmed by multiple config.json snapshots of Llama 3 (see [10]).

[10] https://apxml.com/models/llama-3-8b
     APX Machine Learning model card - Llama 3 8B (third-party spec aggregator)
     Verifies: Llama 3 8B -> hidden dimension 4096, 32 layers, vocabulary 128,256

[11] https://apxml.com/models/gemma-2-2b
     APX Machine Learning model card - Gemma 2 2B (third-party spec aggregator)
     Verifies: Gemma 2 2B -> marketed ~2B (actual ~2.6B) parameters, 26 layers,
     Grouped-Query Attention, RoPE, RMSNorm, 8192 context. (The page's hidden=2048
     is a rounding error; the authoritative hidden_size=2304 comes from [6]/[7].)

[12] https://huggingface.co/unsloth/llama-3-8b (and related config snapshots)
     Multiple config.json copies of Meta-Llama-3-8B (unsloth, AWS blog, cedricchee)
     Verifies: Llama 3 / 3.1 config.json repeatedly shows "tie_word_embeddings":
     false + vocab_size 128256 -> the conspicuous UNTIED big-model example.

[13] https://arxiv.org/abs/1608.05859
     Press, O. & Wolf, L. (2017), "Using the Output Embedding to Improve Language
     Models", EACL 2017, arXiv:1608.05859 (the weight-tying paper)
     Verifies: tying the output (lm_head) embedding to the input embedding improves
     language models and halves the parameter cost of the vocab tables (one V*H
     table instead of two). This is the lineage source for tied vs untied.

[14] https://arxiv.org/abs/2501.05465
     Survey of Small Language Models (arXiv:2501.05465)
     Verifies: "SmolLM2 ... tokenizer with a vocabulary size of 49,152 tokens" and
     the general SLM regime. (Second/third source for SmolLM2's 49152 vocab.)

[15] https://pub.towardsai.net/understanding-tokenization-in-large-language-models-25402f51461e
     Towards AI, "Understanding Tokenization in Large Language Models"
     Verifies: "SmolLM2 used a 49,152 token vocabulary ... This size has become a
     sweet spot for English-focused models" -> supports the ~32k-64k SLM sweet spot.

[16] https://huggingface.co/datasets/StentorLabs/Portimbria-150M-Vs.-SmolLM2-135M/blob/main/paper_final.md
     StentorLabs comparison report
     Verifies: "SmolLM2-135M uses the SmolLM tokenizer (49,152 tokens)" -> second
     source for the SmolLM2 family vocab size + 135M class.

[17] https://huggingface.co/docs/transformers/en/model_doc/llama
     Hugging Face Transformers docs - Llama (authority)
     Verifies: Llama architecture is decoder-only with separate (untied) lm_head by
     default; tie_word_embeddings defaults to False for Llama-class configs.

[18] ../llm/TOKENIZATION.md  (in-repo lineage source)
     Local tutorials/llm/TOKENIZATION.md
     Verifies: the tokenizer mechanics (BPE / SentencePiece) that PRODUCE the
     vocabulary whose size this bundle taxes; the vocab is the input to the V*H
     table. Quote: "vocab ~50k-150k in real LLMs"; Llama/Qwen use byte-level BPE.

NOTES ON HONESTY / UNVERIFIED RANGES:
* Every model config (V, H, I, L, heads, kv_heads, head_dim, tied) is taken
  DIRECTLY from each model's config.json ([1],[2],[4],[6],[9]) and cross-checked
  against the Hugging Face docs defaults ([5],[7],[17]) and/or third-party
  aggregators ([10],[11]). No marketing total is used as ground truth -- the
  computed totals in Section B reproduce each size class (135M/1.7B/2.6B/8B/1.5B)
  exactly, which cross-validates the param counter.
* Section D's compression curve tokens(V) = tokens(V0)*(V0/V)**alpha with
  alpha=0.40 is an ILLUSTRATIVE deterministic model (clearly labelled as such in
  the .py and .md), NOT a measured tokenizer fertility. Its [check]s validate
  internal consistency (sublinear compression, linear param cost), not external
  values. The qualitative claim (bigger vocab -> fewer tokens/text with
  diminishing returns; param cost grows linearly) is the well-established BPE
  behavior; the specific alpha constant is an assumption.
* Gemma-2-2B tie_word_embeddings=True is established from the Gemma2Config default
  ([7], defaults True) combined with the config.json omitting the field ([6]).
