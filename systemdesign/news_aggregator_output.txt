========================================================================
 Section A - RSS/Atom polling (fetch -> parse -> extract)
========================================================================
Periodically GET each feed, parse the XML items, extract fields. Use
conditional GET (If-None-Match ETag): if the stored ETag matches, the
server replies 304 Not Modified (~0.5 KB) and we SKIP parsing.

  GET feed_tech     stored ETag=etag-2024-10
      -> HTTP 200  (250.0 KB)
      parsed: OpenAI launches GPT-5
               url=https://wired.com/story/openai-gpt5
      parsed: Custom silicon for AI
               url=https://wired.com/story/ai-custom-silicon
  GET feed_blog     stored ETag=etag-2024-10
      -> HTTP 304  (0.5 KB)  [unchanged: skip parse]

  Bandwidth: naive full-fetch = 500 KB, conditional GET = 250.5 KB  (saved 249.5 KB, 50%)
  Extracted 2 article(s) from 1 changed feed(s).

  >>> Conditional GET skips unchanged feeds (304), so the polling
      cost is dominated by feeds that ACTUALLY changed - a big win
      when most feeds are quiet most of the time.

========================================================================
 Section B - Deduplication (URL canonicalization + SimHash)
========================================================================
Layer 1: URL canonicalization - mirror URLs collapse to one string.

  https://WWW.Wired.com/story/gpt5?ref=rss&ut=1    -> https://wired.com/story/gpt5?ref=rss&ut=1
  https://wired.com/story/gpt5/?ut=1&ref=rss#top   -> https://wired.com/story/gpt5?ref=rss&ut=1
  https://wired.com/story/gpt5?ref=rss&ut=1        -> https://wired.com/story/gpt5?ref=rss&ut=1
  http://wired.com:80/story/gpt5/?ref=rss&ut=1     -> https://wired.com/story/gpt5?ref=rss&ut=1

  4 variant URLs -> 1 canonical URL(s)  (dedup rate 75%)

Layer 2: SimHash (near-duplicate fingerprint, 64-bit).

  DUP_THRESHOLD = 3 bits (over 64) => near-duplicate
  body_exact fingerprint = 0x063549190461d056
  body_near  fingerprint = 0x06374b190461d256
  body_diff  fingerprint = 0x4ed9b81fc548d460

  hamming(exact, exact_copy) =  0  -> duplicate? YES
  hamming(exact, near_rewrite) =  3  -> duplicate? YES  (lightly rewritten copy, same story)
  hamming(exact, different_story) = 25  -> duplicate? no  (different vocabulary)

  >>> SimHash catches REWRITES that exact hashing misses: 2 changed
      words barely move the fingerprint (small hamming), while a
      totally different story flips ~half the bits.

Layer 3: LSH banding (scale dedup from O(n^2) to ~O(n)).

  Split 64-bit fp into 4 bands x 16 bits. Two articles
  are a CANDIDATE PAIR only if they share >=1 band (LSH property).

  bands shared (exact vs near_rewrite) = 1/4  -> candidate pair? YES
  bands shared (exact vs different_story) = 0/4  -> candidate pair? no

  >>> Banding means we only run full hamming on items sharing a
      bucket - O(n) candidate pairs instead of O(n^2) all-pairs.

HALF_LIFE = 6h. decay(age) = 0.5**(age/6).

 Section D - Adaptive crawl scheduling (poll interval optimization)
========================================================================
Poll each feed at an interval INVERSELY proportional to its update
rate, with min/max clamps. Poll ~2x the update rate to stay fresh.

  raw_interval = (1440 / updates_per_day) * 0.5,  clamped to [15, 1440] min

  feed               updates/day  interval  polls/day  % 304
  ----------------------------------------------------------
  wire (Reuters)              48       15m         96    50%
  tech (Wired)                 6      120m         12    50%
  blog (personal)              1      720m          2    50%

  Naive 'poll all every 15 min'     : 288 polls/day
  Adaptive (by update rate)         : 110 polls/day
  Reduction                         : 62%

  Bandwidth (3 feeds, 250 KB full / 0.5 KB 304):
    naive full-fetch         = 72.0 MB/day
    adaptive + conditional   = 13.8 MB/day  (saved 81%)

  Adaptive convergence (exponential rush-in / back-off):
    rule: new content -> interval/=2 (min 15); none -> interval*=2 (max 1440)
    trace A: fast wire (always new) from 60m:
      30m, 15m, 15m, 15m, 15m  -> converges to 15m
    trace B: dead blog (never new) from 60m:
      120m, 240m, 480m, 960m, 1440m  -> converges to 1440m

  >>> Adaptive scheduling makes polling proportional to how much a
      feed actually changes - fresh where it matters, quiet elsewhere.

========================================================================
 Section E - Personalised feed generation (same pool, two readers)
========================================================================
Merge candidates, dedup (already done), score against the reader's
interest profile, sort desc, return top-K. Same pool ranks differently
per reader because the RELEVANCE term differs.

  Reader A = tech enthusiast  (ai, openai, llm heavy)
    top-3: a1(0.808), a2(0.712), a6(0.703)
  Reader B = finance analyst  (fed, rates, economy heavy)
    top-3: a3(0.728), a1(0.658), a5(0.619)

  >>> Identical candidate pool, different top-3: a3/a5 (fed stories)
      surface for the finance reader but sink for the tech reader.
      Personalization is a READ-TIME re-scoring of the shared index.
