① The JAX compile pipeline — Python → Tracer → jaxpr → XLA → fused kernel

What you're looking at: @jax.jit does NOT run your function. On the first call it traces the fn with abstract Tracers (shapes only, no data), recording each op into a jaxpr IR. XLA then fuses the chain into one kernel. The traced jaxpr below is for f(x) = silu(x @ W1 + b1) @ W2 — the same 4 equations the .py prints.
Python fn→ @jax.jit · Tracer→ jaxpr (op list)→ XLA fuse→ fused kernel→ TPU MXU
The Tracer runs no real math — it only records ops. The compiled binary is cached by signature; later calls skip tracing. (No arXiv id for the JAX tracing paper — it is SysML 2018.)

② XLA fusion — drag the element-wise chain length, watch HBM writes collapse (GOLD: 4 → 1)

chain = [matmul] + E element-wise ops + [matmul] → eager (E+2) writes, fused 1
EAGER: every op materializes its output to HBM. FUSED: a straight-line single-user chain stays on-chip end to end — only the final output writes HBM. At E=2 (the gold silu(x@W1+b1)@W2: add + silu) eager = 4 writes → fused = 1.
eager: HBM write per op fused: only final output to HBM on-chip (no HBM round-trip)

③ TPU MXU systolic — weights sit STILL, activations stream, accumulate with no writeback

3×3 weight-stationary grid: C = W @ X. Step k adds the rank-1 update W[:,k] ⊗ X[k,:].
Weight-stationary: the weight column W[:,k] is held fixed in the grid; activation row X[k,:] streams in; the partial sum accumulates in the PEs and is never written back to HBM mid-compute. Slide k → 3 to drain the result; it must equal torch.matmul(W,X). For an N×N grid the wall-clock is fill+drain = 3N−1 cycles.
