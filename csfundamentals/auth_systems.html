OAuth 2.0 Authorization Code + PKCE — step through the flow

PKCE binds the authorization code to the originating client via code_challenge = base64url(SHA256(code_verifier)). A stolen code cannot be redeemed without the verifier. This is the only correct flow for web/SPA/mobile clients.

The demo token is HS256-signed with a fixed key. Real systems use RS256 (asymmetric): the auth service signs with a private key; every service verifies with the public key from /.well-known/jwks.json. Tampering any byte invalidates the signature — JWTs are integrity-protected, not encrypted.

Production answer is hybrid: JWT for cross-service identity propagation + a central session record for global logout. Every API request triggers an AuthZ check (billions/day); JWT verification is local (~1ms, no DB hit).

TOTP is strong against database breaches but weak against phishing (an attacker can relay it in real time). WebAuthn/passkeys are phishing-resistant by design (domain-bound keys) — the preferred MFA for 2024+. Verification uses a constant-time compare (compare_digest) to prevent timing oracles.
