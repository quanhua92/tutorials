# root_index_json — web provenance log
# Every URL has a "Verifies:" line stating exactly what it confirms.
# STEP 2 of HOW_TO_RESEARCH.md is non-skippable; ≥2 sources per concept.

https://en.wikipedia.org/wiki/1080p
  Verifies: the standard HD canvas. "1080p (1920 × 1080 progressively displayed pixels) ... characterized by 1,920 pixels displayed across the screen horizontally and 1,080 pixels down the screen vertically." Confirms root index.json `canvas: {width:1920, height:1080}`. Also confirms the fps range: "ATSC standards ... supported 1080p video, but only at the frame rates of 23.976, 24, 25, 29.97 and 30 frames per second" (later amended to add 50/59.94/60) — so `canvas.fps = 30` is a standard rate and 24/25/60 are valid alternatives.

https://json-schema.org/draft/2020-12
  Verifies: the JSON validation approach. JSON Schema Draft 2020-12 is the standard vocabulary for "structural validation of JSON" — describing structure, constraints, and data types for a JSON document. This is the industry-standard way to validate the root index.json schema (version number, canvas object, enum-constrained caption_style, slides string array, etc.) as an alternative to hand-rolled validation.

https://stonefly.com/blog/content-addressable-storage-enterprise-guide/
  Verifies: SHA-256 content addressing for the music asset ref. "That identifier is a cryptographic hash — typically SHA-256 — computed from the data itself. The hash becomes the object's permanent address within the system." Confirms `audio.music.asset = "sha256:..."` is a content-addressed reference and that dedup is automatic ("same bytes ⇒ same SHA ⇒ one stored blob").

https://t34ch.tech/articles/content-addressed-storage/
  Verifies: why content addressing gives free deduplication. "Deriving addresses from content rather than assigning them ... Deduplication becomes automatic." Independent second source for the SHA-256 asset-reference model used by `audio.music.asset` (and the assets/ directory in RFC §5.1).

(in-repo) docs/rfc-0001.md §5.2 "Root index.json (minimal, project-wide)"
  Verifies: the EXACT root index.json schema quoted verbatim in the runnable — version (number), canvas {width,height,fps}, theme {caption_style, colors{}, fonts{}}, audio {music {asset, volume, loop}, voiceover {asset, auto_generated}}, transition_default {type, duration}, slides (array of slide ids). Primary in-repo source.

(in-repo) docs/rfc-0001.md §5.1 + §10
  Verifies: §5.1 — why the root IS a unit folder (index.html + index.json), and the table splitting root index.json data (canvas/theme/audio refs/slide ordering/transition_default) from root index.html animation. §10 — that canvas dims/fps feed the HF render + ffmpeg encode export steps unchanged.

(in-repo) docs/AGENTS.md "Slot extras" + "Preview audio"
  Verifies: the `caption_style` enum is exactly {highlight, neon, editorial, eco-green} ("Controls caption CSS: highlight, neon, editorial, eco-green"). Also verifies the music-bed default volume: "ext-music: music-bed, looped, volume 0.08". The current templates/*/template.json (with id/name/duration/priority/mode/caption_style/layouts/default_slides) is the precursor the root index.json evolves from (RFC §15.1).
