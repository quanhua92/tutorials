# Type safety: inferred row/column types, `$inferSelect`, `$dynamic`

**Doc Source**: [Drizzle — Goodies (Type API)](https://orm.drizzle.team/docs/goodies) · [Select (result-type inference)](https://orm.drizzle.team/docs/select) · [Dynamic query building (`$dynamic`)](https://orm.drizzle.team/docs/dynamic-query-building) · [Get started (SQLite)](https://orm.drizzle.team/docs/get-started-sqlite)

## The Core Concept: Why This Example Exists

**The Problem:** In a dynamically-typed database layer (raw `better-sqlite3`, `pg`, `mysql2`), every row comes back as `any` or `Record<string, unknown>`. You write `user.emial` instead of `user.email` and find out at runtime — in production, from a user-facing error. The database *has* a strict schema; the application code just doesn't know it. Keeping a separate hand-written `interface User` in sync with the SQL `CREATE TABLE` is a losing game: the two drift, the compiler can't catch it, and the mismatch is again a runtime bug.

**The Solution:** Drizzle makes the **schema definition the single source of truth for types**. The same `pgTable('users', { id: serial().primaryKey(), name: text().notNull() })` object that `drizzle-kit generate` turns into `CREATE TABLE` SQL also *is* the TypeScript type — Drizzle derives the row type from it with `typeof users.$inferSelect`, the insert type with `typeof users.$inferInsert`, and the result type of *every* query flows from there through your `.select()`/`.where()`/`.columns()` projections. You write the schema once; selects, inserts, and results are all typed, with no manual interfaces to maintain.

This is the TypeScript analog of Rust's `sqlx` `query_as!` macro (🔗 [`../rust/sqlx/02-postgres-todos.md`](../rust/sqlx/02-postgres-todos.md)) — which derives a struct from a checked SQL query at compile time — and of Rust's `serde` derive (the row struct's `Serialize`/`Deserialize` is generated from the same `#[derive]` you write once). The shared promise: **one definition, many derived uses, compiler-enforced consistency.** When you rename a column in the schema, every query that selects it and every type that consumes it updates (or errors) at the same compile step.

> The schema-as-type-source is exactly what 🔗 [`DATABASE_DRIVERS`](../DATABASE_DRIVERS.md) Section C calls out ("drizzle-orm: schema + type-safe CRUD (inferred row types)"). This walkthrough unpacks the *mechanics*: `inferSelect`/`inferInsert`, projection-narrowed result types, and `$dynamic()` for queries built at runtime.

## Practical Walkthrough: Code Breakdown

### Deriving row types from the schema (`$inferSelect` / `$inferInsert`)

Drizzle exposes three equivalent spellings for pulling a type off a table object. The docs' canonical example (shown here in the SQLite dialect):

```ts
import { int, text, sqliteTable } from 'drizzle-orm/sqlite-core';
import { type InferSelectModel, type InferInsertModel } from 'drizzle-orm'

const users = sqliteTable('users', {
  id: int('id').primaryKey(),
  name: text('name').notNull(),
});

type SelectUser = typeof users.$inferSelect;
type InsertUser = typeof users.$inferInsert;
// or
type SelectUser = typeof users._.$inferSelect;
type InsertUser = typeof users._.$inferInsert;
// or
type SelectUser = InferSelectModel<typeof users>;
type InsertUser = InferInsertModel<typeof users>;
```

*Source: [orm.drizzle.team/docs/goodies](https://orm.drizzle.team/docs/goodies) (Type API)*

What you get, concretely, from the schema above:

- `SelectUser = { id: number | null; name: string }` — the shape of a row *as read* (note `id` is `number | null` because `int().primaryKey()` without `.notNull()` permits a nullable PK, e.g. for `autoincrement`).
- `InsertUser = { id?: number | null; name: string }` — the shape of a row *as written*. Required columns (`name`) are mandatory; columns with defaults or auto-generation (`id`) become optional.

Three things to notice:

1. **The types are derived, not declared.** There is no hand-written `interface User`. Rename `name` to `fullName` in the schema and both types update at the next typecheck.
2. **Nullability flows from the column builder.** `.notNull()` makes a column required in both select and insert; its absence leaves `| null`.
3. **Select vs insert differ on optionality.** A column generated by the DB (serial, default-now) is required on read but *optional* on write — Drizzle computes this from the column's `$defaultFn`/`default`/`generated` metadata. This is the same split Prisma models express as separate read/write types.

The `InferSelectModel<T>` / `InferInsertModel<T>` helper form is the one you use in **generic** positions where `typeof users.$inferSelect` won't fit — e.g. constraining a function parameter or building a mapped type (🔗 [`UTILITY_TYPES`](../UTILITY_TYPES.md), 🔗 [`MAPPED_CONDITIONAL_TYPES`](../MAPPED_CONDITIONAL_TYPES.md)).

### Result types follow your projection

The type of a query's result isn't just "the table type" — it's the table type **filtered through what you selected**. The core builder documents this directly:

```ts
// schema
export const users = pgTable('users', {
  id: serial('id').primaryKey(),
  name: text('name').notNull(),
  age: integer('age'),
});

// select everything → full row type, nullability preserved
const result = await db.select().from(users);
/*
  {
    id: number;
    name: string;
    age: number | null;
  }[]
*/
```

```sql
select "id", "name", "age" from "users";
```

*Source: [orm.drizzle.team/docs/select](https://orm.drizzle.team/docs/select) (Basic select)*

Note Drizzle **always emits an explicit column list** (never `select *`) precisely so the result type and the SQL agree field-for-field, in order. Now narrow the projection and the type narrows with it:

```ts
const result = await db.select({
  field1: users.id,
  field2: users.name,
}).from(users);

const { field1, field2 } = result[0];
// result[0] : { field1: number; field2: string }
```

*Source: [orm.drizzle.team/docs/select](https://orm.drizzle.team/docs/select) (Partial select)*

Arbitrary SQL expressions get a type via the `sql<T>` generic — you tell Drizzle the expected TS type, and it flows into the result:

```ts
const result = await db.select({
  id: users.id,
  lowerName: sql<string>`lower(${users.name})`,
}).from(users);
// result[0] : { id: number; lowerName: string }
```

*Source: [orm.drizzle.team/docs/select](https://orm.drizzle.team/docs/select)*

The docs are explicit about the contract: `sql<string>` is a *promise* to the compiler — Drizzle cannot cast at runtime, so if you lie (claim `sql<number>` for something returned as a string), the runtime value won't match. For real runtime transforms, use `.mapWith()`. The relational query builder's `columns: { id: true, content: true }` (🔗 [`07-rqb-query-builder.md`](./07-rqb-query-builder.md)) is the same mechanism in declarative form — the result type is the projection you wrote.

### `Equal<>` proofs — when inference must be exact

Drizzle's type-level correctness is often asserted with an `Equal<A, B>` utility (a conditional that resolves to `1` only when `A` and `B` are mutually assignable, i.e. identical). The pattern, used throughout Drizzle's own test suite and idiomatic in user code, is:

```ts
import { Equal, Expect } from 'drizzle-orm/[core]'; // or a local helper

// If these two types ever drift, this becomes a compile error.
type _Check = Expect<Equal<typeof users.$inferSelect, { id: number | null; name: string }>>;
```

The idea: assign the `Equal`/`Expect` result to a `_`-prefixed type alias that is never used as a value. If the inferred type ever drifts from the expected shape (because the schema changed), the alias stops being assignable and the file fails to typecheck — a *compile-time tripwire* that the "one schema → many types" invariant still holds. This is the same technique the curriculum's 🔗 [`GENERICS`](../GENERICS.md) and 🔗 [`MAPPED_CONDITIONAL_TYPES`](../MAPPED_CONDITIONAL_TYPES.md) bundles discuss for type-level assertions: types as tests. (Drizzle doesn't export a stable public `Equal`/`Expect` from the top level — grab them from `drizzle-orm/[core]` or copy the standard one-liner: `type Equal<A, B> = [A] extends [B] ? ([B] extends [A] ? 1 : 0) : 0`.)

### Narrowing a column's type with `$type<>`

Sometimes the DB column is `text` but your domain knows it's a closed set of string literals. `$type<>()` re-narrows the column's TS type without changing the SQL type:

```ts
import { serial, text, pgTable } from "drizzle-orm/pg-core";

export const user = pgTable("user", {
  id: serial("id").primaryKey(),
  name: text("name"),
  email: text("email"),
  password: text("password"),
  role: text("role").$type<"admin" | "customer">(),
});
```

*Source: [orm.drizzle.team/docs/goodies](https://orm.drizzle.team/docs/goodies) (getColumns example)*

Now `$inferSelect` yields `role: "admin" | "customer"` (not `string`), so assignments from the DB and comparisons in `where` clauses are checked against the union. This is the brand-string-literal pattern from 🔗 [`UNIONS_INTERSECTIONS`](../UNIONS_INTERSECTIONS.md) applied at the column level.

### `getColumns()` — typed column subsets

To select "all columns except a few" (the classic "never return `password`" case), `getColumns` returns a typed column map you can destructure:

```ts
import { getColumns } from "drizzle-orm";
import { user } from "./schema";

const { password, role, ...rest } = getColumns(user);

await db.select({ ...rest }).from(users);
```

*Source: [orm.drizzle.team/docs/goodies](https://orm.drizzle.team/docs/goodies) (Get typed columns)*

`rest` is already typed as the subset of `user`'s columns — `db.select({ ...rest })` then infers the result type from that subset, automatically excluding `password`. (`getColumns` is available from `drizzle-orm@1.0.0-beta.2`; on pre-1.0 versions the same job is done by `getTableColumns`.)

### `$dynamic()` — building queries at runtime, keeping the types

By default Drizzle's query builders enforce SQL's shape: a `SELECT` has one `WHERE`, so `.where()` can be called only once (calling it twice is a *type error*). That's great for one-shot queries but blocks a common pattern — a helper that takes a query and augments it. `$dynamic()` switches the builder into a mode that lifts the once-only restriction while keeping full type inference:

```ts
function withPagination<T extends PgSelect>(
  qb: T,
  page: number = 1,
  pageSize: number = 10,
) {
  return qb.limit(pageSize).offset((page - 1) * pageSize);
}

const query = db.select().from(users).where(eq(users.id, 1));
withPagination(query, 1); // ❌ Type error - the query builder is not in dynamic mode

const dynamicQuery = query.$dynamic();
withPagination(dynamicQuery, 1); // ✅ OK
```

*Source: [orm.drizzle.team/docs/dynamic-query-building](https://orm.drizzle.team/docs/dynamic-query-building)*

The generic constraint `<T extends PgSelect>` is the crucial part — it lets the helper refine the result type as it transforms the query (e.g. a `withFriends` helper that adds a `leftJoin` changes the row type, and `T` carries that change through):

```ts
function withFriends<T extends PgSelect>(qb: T) {
  return qb.leftJoin(friends, eq(friends.userId, users.id));
}

let query = db.select().from(users).where(eq(users.id, 1)).$dynamic();
query = withFriends(query);
```

*Source: [orm.drizzle.team/docs/dynamic-query-building](https://orm.drizzle.team/docs/dynamic-query-building)*

`PgSelect` (Postgres), `MySqlSelect`, `SQLiteSelect` (and their `Insert`/`Update`/`Delete` siblings, plus `…QueryBuilder` variants for the standalone builder) are the generic bounds the docs list. The reason these types *only* work in dynamic mode is that the static builder encodes "which clauses are already set" in its type parameters (so it can reject a second `.where()`); `$dynamic()` relaxes that tracking and hands you the underlying type that recomputes the result on each call. This is a direct application of the conditional/generic-type machinery in 🔗 [`GENERICS`](../GENERICS.md) and 🔗 [`UTILITY_TYPES`](../UTILITY_TYPES.md) — the builder's typestate is encoded in type parameters, exactly the pattern those bundles teach.

## Mental Model: One Schema, Many Types

```
                          ┌─ pgTable('users', { id, name, age }) ─┐
                          │   (the single source of truth)         │
                          └────────────────┬───────────────────────┘
                                           │
        ┌──────────────────┬───────────────┼────────────────┬───────────────────┐
        ▼                  ▼               ▼                ▼                   ▼
  drizzle-kit         $inferSelect     $inferInsert    db.select({...})    db.insert(...)
  generate → SQL      (read row)       (write row)     result type is       values type is
  (Section E)         { id, name,      { name,         the projection       checked against
                      age|null }       age? }          { field1, field2 }   $inferInsert
```

Five consumers, one definition. Change the schema and all five update together:

- **Migrations** (`generate`) — the SQL diff follows the schema (🔗 [`05-migrations.md`](./05-migrations.md)).
- **Reads** — `$inferSelect` is the row type; `.select()` narrows it by projection.
- **Writes** — `$inferInsert` is the input type; `db.insert().values(...)` checks against it.
- **Query results** — flow from `$inferSelect` through `.select`/`.where`/`.columns` (and through `with`/`columns` in the RQB).
- **Domain code** — `Equal<>` proofs assert the inferred types match your expectations.

### Drizzle inference ↔ Rust `sqlx` / `serde`

| Concern | Drizzle (TS) | Rust `sqlx` | Rust `serde` |
| --- | --- | --- | --- |
| Source of truth | `pgTable(...)` schema | `CREATE TABLE` SQL + `query_as!` | `#[derive(Serialize, Deserialize)] struct` |
| Row type derived from | `typeof users.$inferSelect` | macro-expanded struct fields | the derive |
| Nullability | column builder (`.notNull()`) | `Option<T>` in the struct | `Option<T>` field |
| Checked against DB at | runtime (and via `generate` diff) | **compile time** (macros query the DB) | compile time (struct defn) |
| Result narrows with projection | `.select({...})` / `columns` | `query_as!` against a specific `SELECT` | N/A |

The key cultural difference: `sqlx`'s `query!`/`query_as!` macros connect to a live database (or a prepared `sqlx-data.json`) at *compile time* and refuse to build if your SQL doesn't match the schema — compile-time verification. Drizzle can't do that (TS has no macro system that runs SQL), so it instead makes the *schema object* the compile-time truth and derives everything from it; the `generate`/`check` kit commands provide the schema↔DB consistency check as a separate step. Same end goal ("the compiler catches schema drift"), different leverage point.

### Why projection-narrowed types matter

Without them, every query returns the full row type and you defensively optional-chain everything. With them, `db.select({ id: users.id }).from(users)` returns `{ id: number }[]` — the compiler *knows* `name` isn't there, so `row.name` is a type error, not a runtime `undefined`. This is the same payoff as the literal-type narrowing in 🔗 [`TYPE_NARROWING`](../TYPE_NARROWING.md), applied to query results: the type describes exactly what the query returns, no more, no less.

## Pitfalls

- **`$inferSelect` vs `$inferInsert` differ deliberately.** A serial/autoincrement PK is `number` on read but `number | null`/optional on write. Don't assume they're the same type with different names.
- **`sql<T>` is a promise, not a cast.** Lie about `T` and the runtime value won't match the type. Use `.mapWith()` for real transforms.
- **`Equal`/`Expect` aren't top-level exports.** Pull them from `drizzle-orm/[core]` or copy the standard definition; don't assume `import { Equal } from 'drizzle-orm'` resolves in every version.
- **`$type<>()` changes only the TS type, not the SQL.** The DB column is still `text`; if you need a real `enum` type in Postgres, declare it with `pgEnum`, not `$type`.
- **`getColumns` vs `getTableColumns`.** `getColumns` ships from `drizzle-orm@1.0.0-beta.2`; pre-1.0 (e.g. `0.45.x`) the equivalent is `getTableColumns`. Mixing them up is a "module has no exported member" error.
- **`$dynamic()` is required to pass a builder to a helper.** The static builder's typestate rejects a second method call by design; you must opt into dynamic mode to compose. Forgetting `.$dynamic()` is the most common "why won't this compile" in dynamic query building.
- **Inference is only as good as the schema.** If a column is declared `integer('age')` without `.notNull()`, every select infers `age: number | null` — even if you "know" it's always set. The DB-side constraint and the TS-side optionality must agree.
- **Renaming a column in the schema doesn't rename it in the database.** Inference updates instantly, but the live DB still has the old column until you `generate` + `migrate` (🔗 [`05-migrations.md`](./05-migrations.md)). The type can be ahead of the database.
