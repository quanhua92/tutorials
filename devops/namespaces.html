Linux Namespaces — the isolation primitive behind containers NAMESPACES.md namespaces.py

A namespace is a pair of glasses a process wears — it filters ONE category of kernel resource (PIDs, network, mounts, …). A container = a process wearing all 6 at once. Toggle each pair of glasses off and the container starts leaking back into the host. The model below is recomputed live in JS from the identical logic as namespaces.py.  |  cgroups limit resources →

① The six pairs of glasses — toggle each namespace on/off, watch isolation

What to do: each button is one namespace (CLONE_NEW* flag). ON = container gets a fresh namespace (isolated). OFF = it inherits the host's (leaks). A real container ships with all six ON — flip any off and that axis of isolation is gone.
Isolation: 6 / 6 namespaces differ between host and container. Real container requires 6/6.

CONTAINER view (what PID 1 inside sees)

Same kernel, same disk — the only difference is which /proc/<pid>/ns/* inode each namespace points to. Equal inode = shared; different inode = isolated.

② PID namespace — same process, two numberings

What this shows: the host sees a flat PID list (1, 2, 3, 4, 5 …). Inside the container's PID namespace the kernel renumbers the first process to 1 — so the container's /bin/sh thinks it is the init of a fresh machine. The host keeps a hidden mapping table to translate signals.
Yellow lines = the hidden host→container PID mapping (host 3 ≡ container 1). The container can never see systemd or sshd — they live in a different PID namespace.
