① Statistics — what ANALYZE stores in pg_statistic (table t, column x)

What to do: N = 10,000 rows, column x. The bars are the most-common values (MCV) with their frequencies; the axis below is the histogram of equal-depth buckets over the remaining 6,000 rows. These four numbers (MCV, histogram, ndistinct, correlation) drive every estimate below.
stanullfrac = 0.0100  (100 nulls)  |  stadistinct = 35  |  Σ MCV freq = 0.39 (3,900 rows)  |  tail mass = 0.60 (6,000 rows)  |  correlation ρ = 0.00
most_common_vals / most_common_freqs (MCV) — hot values, exact equality
histogram_bounds — 10 equal-depth buckets over the non-MCV tail (range estimates)
bounds = [100, 103, 106, 109, 112, 115, 118, 121, 124, 127, 129] — each bucket ≈ 600 rows
In words: MCV covers the frequent values exactly; the histogram covers the long tail by row-count buckets (equal-depth), not equal value-ranges. nullfrac + Σmcv + tail = 1.0.

② Selectivity calculator — pick a predicate, see the estimate vs the truth

What to do: choose a predicate type and value. Equality on an MCV value uses its frequency (exact); equality on anything else uses (1−Σmcv−nullfrac)/(ndistinct−n_mcv) (the uniformity assumption — note it lies for values that don't exist!); ranges use the histogram with linear interpolation.
table t, N = 10,000
The gap between estimated and actual rows is the whole game: cardinality error is inherited by every downstream cost and can flip the chosen plan. (COST_ESTIMATION.md §2)

③ SeqScan vs Index Scan — drag R (matches) and ρ (correlation), find the crossover

What to do: the SeqScan line is flat (always 200 — it reads every page regardless of predicate). The Index Scan curve falls as R shrinks. Drag ρ from 0 (scattered, random I/O ×4) to 1 (clustered, sequential I/O) and watch the crossover move. The green check: OK confirms the JS matches cost_estimation.py (seqscan=200, index R=1500/ρ=0 → 422.5, R=1500/ρ=1 → 37.5).
seq_page_cost=1 · random_page_cost=4 · cpu_tuple=0.01 · cpu_index=0.005
SeqScan (fixed 200) Index Scan (this ρ) current R (cheaper plan wins) crossover
The "index wins below ~5%" rule of thumb assumes partial correlation. At ρ=0 the crossover is ~0.7%; at ρ=1 it is ~80%. random_page_cost on SSD (~1.1) shifts it further toward indexes. (COST_ESTIMATION.md §3)

④ Join order enumeration — brute force (3!) vs DP, for A⋈B⋈C

What this shows: A(10000)–B(1000)–C(100), chain join A-B-C (A-C has no predicate). Brute force tries all 3! orderings (two force a cartesian → 1,000,000 rows). DP keeps only the cheapest plan per subset, prunes the cartesian, and finds the same optimum in O(3ⁿ) instead of O(n!).
Optimal: join the two smallest first (B⋈C = 100 rows), then join A. PostgreSQL switches from DP to a genetic search (GEQO) above geqo_threshold = 12 tables. (COST_ESTIMATION.md §5)
