Think of image layers as printed acetate sheets and the running container as a blank sheet on top. To edit a printed note you first trace it onto your sheet (copy-on-write) — the printed original is never touched, so 100 containers share the same stack. Toggle writes below and watch the copy-up count. The numbers are recomputed in JS from the identical model as overlayfs.py.  |  → how those layers are built (Dockerfile)  |  namespaces

① The image as a stack of read-only layers + one writable top sheet

What this shows: image flask-app:1.0 = 3 read-only layers (bottom = ubuntu base, top = pip flask) stored once. A container adds a thin writable upperdir; the union view the container sees as '/' is merged.
Layer ids are content-addressed (sha256 of name + files) and pinned from overlayfs.py Section A. Same files ⇒ same id ⇒ shared on disk and across pulls (Section D).

② Copy-on-write — toggle the container's writes, watch the copy-up count

What to do: check the files the container writes. Files that live in a lower layer get copy-up'd into upper ONCE (the trace); brand-new files go straight to upper (new); re-editing an already-copied file is in-place (no new copy). The lower originals never change.
