IMPORTANT NOTE ON THE MLX "PAPER"
--------------------------------
The build brief cited "Han et al 2023 MLX arXiv:2312.06789" as THE paper.
That arXiv ID is WRONG: arXiv:2312.06789 is in fact "A Critique of
Human-Autonomous Team Dynamics" by Hanjing Shi (cs.HC) -- an unrelated HCI
paper, NOT MLX. MLX has NO arXiv paper: the MLX GitHub README's official
"Citing MLX" section cites it as *software* (`@software{mlx2023, ...
url = {https://github.com/ml-explore}, year = {2023}}`), authored by Awni
Hannun, Jagrit Digani, Angelos Katharopoulos, and Ronan Collobert (Apple
Machine Learning Research). This bundle therefore cites the AUTHORITATIVE
primary sources directly (MLX docs + GitHub, both Apple ML Research) rather
than a non-existent arXiv paper. No unverified fact was shipped.

[1] https://github.com/ml-explore/mlx
    Apple Machine Learning Research -- "MLX: An array framework for Apple
    silicon" (official GitHub repository, the canonical primary source).
    Verifies:
      - "Unified memory: A notable difference from MLX and other frameworks
        is the *unified memory model*. Arrays in MLX live in shared memory.
        Operations on MLX arrays can be performed on any of the supported
        device types WITHOUT TRANSFERRING DATA." (README)
      - "Lazy computation: Computations in MLX are lazy. Arrays are only
        materialized when needed." (README)
      - "Composable function transformations: MLX supports composable
        function transformations for automatic differentiation, automatic
        vectorization, and computation graph optimization." (README)
      - "Dynamic graph construction: Computation graphs in MLX are
        constructed dynamically." (README)
      - Official citation: software by Awni Hannun, Jagrit Digani, Angelos
        Katharopoulos, Ronan Collobert (2023), NOT an arXiv paper.

[2] https://ml-explore.github.io/mlx/build/html/usage/unified_memory.html
    MLX 0.31.2 documentation -- "Unified Memory" (official Apple ML Research).
    Verifies:
      - "Apple silicon has a unified memory architecture. The CPU and GPU
        have direct access to the SAME memory pool. MLX is designed to take
        advantage of that." (verbatim)
      - The device is chosen at OP time, not array time: "you specify the
        device when you run the operation. Any device can perform any
        operation on a and b WITHOUT NEEDING TO MOVE THEM from one memory
        location to another." Example: `mx.add(a, b, stream=mx.cpu)` and
        `mx.add(a, b, stream=mx.gpu)` run on the SAME arrays -- no copy.
        (This is the "stream=, not .to()" mechanism modelled in Sections A
        and C of mlx_metal_edge.py -- the zero-copy CPU<->GPU sharing.)

[3] https://ml-explore.github.io/mlx/build/html/usage/lazy_evaluation.html
    MLX 0.31.2 documentation -- "Lazy Evaluation" (official Apple ML Research).
    Verifies:
      - "When you perform operations in MLX, NO COMPUTATION ACTUALLY HAPPENS.
        Instead a compute graph is recorded. The actual computation only
        happens if an eval() is performed." (verbatim)
      - Lazy evaluation is what "makes it much easier to integrate
        compilation" and "graph optimizations" -- i.e. the fusion that
        Section B of mlx_metal_edge.py models (eager vs fused kernel count).
      - Materialization triggers: print, .item(), numpy conversion, save()
        all force a graph evaluation (the "implicit eval" behaviour).

[4] https://developer.apple.com/documentation/metal/choosing-a-resource-storage-mode-for-apple-gpus
    Apple Developer -- "Choosing a resource storage mode for Apple GPUs"
    (official Apple Metal documentation).
    Verifies:
      - "Apple GPUs have a UNIFIED MEMORY MODEL in which the CPU and the GPU
        SHARE SYSTEM MEMORY." (verbatim from the doc summary returned by
        developer.apple.com). This is the hardware-level basis for MLX's
        unified memory: the shared-storage mode is what makes zero-copy
        CPU<->GPU access possible on Apple Silicon (the UMA premise of the
        whole bundle, Section A).

[5] https://arxiv.org/abs/2312.06789
    arXiv:2312.06789 -- "A Critique of Human-Autonomous Team Dynamics:
    Contrasting Qualitative and Quantitative Perspectives" by Hanjing Shi
    (cs.HC, submitted 11 Dec 2023).
    Verifies: that the arXiv ID 2312.06789 is NOT the MLX paper (it is an
    unrelated Human-Computer Interaction paper). Logged here so the
    discrepancy in the build brief is on the record and no one re-cites the
    wrong ID. MLX's correct citation is the software repo (source [1]).

[6] https://github.com/quanhua92/tutorials/blob/main/local-llm/MLX_INFERENCE.md
    In-repo production reference -- "MLX Inference -- Apple's Array Framework
    for Apple Silicon" (the local-llm sibling guide this bundle simulates).
    Verifies:
      - The lineage PyTorch MPS (split CPU-RAM/GPU-VRAM, .to() copies) ->
        llama.cpp Metal (hand-tuned kernels bolted onto a CPU-RAM design) ->
        MLX unified-native (zero copy via stream=, not .to()) -> lazy eval ->
        op fusion (K elementwise ops -> 1 Metal kernel -> K-fold less
        bandwidth). Section 1 of that guide is the spine of this bundle.
      - The fusion gold value: a 3-op elementwise chain eager = 6 transfers
        (3 reads + 3 writes), fused = 2 transfers (1 read + 1 write) -> 3x
        less bandwidth. mlx_metal_edge.py Section B adapts this to a
        MatMul+Add+ReLU chain: eager = 3 kernel launches, fused = 1.
      - Group-quant bytes/param model: group=32 -> (2B scale + 16B packed) =
        18B per 32 weights = 0.5625 bytes/param (mlx_metal_edge.py Section D
        Q4_BPP constant).
      - Decode is bandwidth-bound: tokens/s ~ unified_memory_BW /
        weight_bytes (the reason UMA + fusion buy real throughput).

[7] https://news.ycombinator.com/item?id=47820195
    Hacker News discussion -- "Zero-Copy GPU Inference from WebAssembly on
    Apple Silicon" (community; summarizes the Apple-Silicon UMA mechanism).
    Verifies: (second-source confirmation of the UMA premise)
      - "Apple Silicon uses unified memory where the CPU and the GPU use the
        EXACT SAME memory and NO COPIES from RAM to VRAM are needed."
        Independent restatement of sources [2] and [4] in plain language --
        the discrete-vs-unified contrast at the heart of Section A.

[8] https://ml-explore.github.io/mlx/build/html/usage/compile.html
    MLX 0.31.2 documentation -- "Compilation" (official Apple ML Research).
    Verifies:
      - mx.compile traces the (lazy) graph, fuses it, and CACHES the result:
        the first call pays a tracing/compile cost and subsequent calls hit
        the fused graph. This is the "Metal kernel compile cost first-run"
        pitfall in the MLX_METAL_EDGE.md pitfalls table, and the reason the
        fusion pass in Section B is described as happening "at
        materialization" with a one-time compile cost.

NOTES ON FACT-CHECKING / CORRECTIONS vs the original brief
----------------------------------------------------------
1. The brief's "Han et al 2023 MLX arXiv:2312.06789" is WRONG (see the
   IMPORTANT NOTE above and source [5]). Corrected to the official software
   citation (source [1]). No arXiv paper for MLX exists.
2. The brief's suggested fusion gold ("MatMul -> Add(bias) -> ReLU = 3 eager,
   1 fused") was kept verbatim -- it is consistent with MLX's documented
   elementwise-fusion model (sources [2], [3], [6]) where a trailing run of
   elementwise ops folds into a matmul epilogue.
3. The Q4 bytes/param = 0.5625 figure is derived from the group-quant model
   in the in-repo reference (source [6], Section F), not invented.
4. The VRAM/system-RAM numbers in Section A (8 GB model / 6 GB VRAM / 16 GB
   unified) are illustrative toy values chosen so the discrete-FAILS /
   UMA-OK contrast prints cleanly; they are clearly labelled as toy and the
   MECHANISM (copy proportional to M; FAIL if M > VRAM; zero copy under UMA)
   is the verified claim, not the specific card capacities.
