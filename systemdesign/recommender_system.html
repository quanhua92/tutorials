① Recommendation explorer — CF vs content vs hybrid

What to do: pick a user, drag the hybrid weight w. CF = “users who liked X also liked Y” (item-item cosine, ≥2 co-raters); content = feature-cosine; hybrid = w·CF + (1−w)·content. A “−” means CF has no signal there (sparsity hole) — the case content-based rescues (recommender_system.py Sections 1–3).

What to do: toggle the source. CF cosine is computed over co-rating users (blank = <2 co-raters, unreliable); content cosine uses the 4-dim genre vectors. Bar height = cosine magnitude. Notice CF and content largely agree, but CF is blind across genres while content is not.

③ Cold start — new user (popularity) / new item (content-based)

What to do: CF needs history, so brand-new users/items need fallbacks. New user → popularity (most-rated items). New item → content-based: find near-duplicate items and inherit their audience until clicks accumulate (recommender_system.py Section 4).

④ Embedding retrieval — two-tower brute kNN vs LSH ANN

What to do: the query embedding (user tower) is scored against all 8 item embeddings by cosine. Brute kNN is exact O(N); LSH ANN hashes into buckets via 2 random-projection planes and probes only the query bucket — O(1) but approximate. Here recall = 1.0 yet the bucket also returns a rank-4 false positive — exactly why retrieval over-fetches and rankers filter (recommender_system.py Section 5).
