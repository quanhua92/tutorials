========================================================================
  SECTION A: static vs continuous batching timeline
========================================================================
Two requests: R0 needs 3 tokens, R1 needs 6 tokens.

STATIC (request-level) batching: run the WHOLE batch to completion.
| step | R0 (3 tok) | R1 (6 tok) | GPU useful? |
|------|------------|------------|-------------|
| 1    | gen        | gen        | yes         |
| 2    | gen        | gen        | yes         |
| 3    | gen        | gen        | yes         |
| 4    | IDLE (pad) | gen        | NO - R0 slot padded |
| 5    | IDLE (pad) | gen        | NO - R0 slot padded |
| 6    | IDLE (pad) | gen        | NO - R0 slot padded |

Static: 9/12 token-steps useful (75%); 3 are padded idle.

CONTINUOUS (iteration-level) batching: reclaim a finished slot NEXT step.
Suppose R2 (2 tokens) arrives right after R0 frees up.
| step | R0 (3) | R1 (6) | R2 (2) | note              |
|------|--------|--------|--------|-------------------|
| 1    | gen    | gen    | -      |                   |
| 2    | gen    | gen    | -      |                   |
| 3    | gen    | gen    | -      |                   |
| 4    | -      | gen    | gen    | R2 slots into R0's freed slot |
| 5    | -      | gen    | gen    |                   |
| 6    | -      | gen    | -      |                   |

Continuous: 11/12 token-steps useful (92%). R2 finished 1 step after R0.

Orca (OSDI'22) calls this 'iteration-level scheduling': schedule at
the granularity of ONE token, not one request.

========================================================================
  SECTION B: Sequence 3-state lifecycle + preempt
========================================================================
Every request is a Sequence in exactly one of three states:

  WAITING  - prefill not started, or PREEMPTED back here
  RUNNING  - blocks allocated, actively prefilled / decoding
  FINISHED - EOS or max_tokens reached

Transitions (who fires them):
  WAITING  --schedule()+allocate()--> RUNNING
  RUNNING  --EOS / max_tokens-------> FINISHED   (postprocess)
  RUNNING  --OOM in decode----------> WAITING    (preempt; re-queue FRONT)

preempt(seq): status=WAITING; is_prefill=True; deallocate blocks;
              waiting.appendleft(seq).  This is vLLM's RECOMPUTE mode:
              the seq re-prefills from num_cached_tokens (or from 0 if
              its old blocks got reused). Eviction picks the NEWEST
              running seq (running.pop()) ~= LRU under FCFS admission.

Preempt demo: 2 blocks total, both seqs prefilled (each holds 1 block).
  after prefill: running=[0, 1], free=[]
  step2 decode: scheduled=[0], is_prefill=False
  running now=[0], waiting=[1]
  s1 preempted (RUNNING->WAITING, is_prefill=True)? True

[check] preempt moves newest RUNNING->WAITING at front: True -> OK

========================================================================
  SECTION E: decode may_append + preempt-on-OOM (evict newest)

Decode s0: len grows 2->3, crosses a page boundary (3%2==1) -> needs a

  running=[0]  waiting(front first)=[1]

========================================================================
  SECTION F: postprocess (hash + append + EOS / max_tokens)
========================================================================
postprocess(seqs, token_ids, is_prefill) does, per seq:
  1. block_manager.hash_blocks(seq)  - register newly FILLED pages
  2. num_cached_tokens += num_scheduled_tokens
  3. if chunked prefill still incomplete: CONTINUE (no token yet)
  4. append_token(generated)
  5. if EOS or num_completion_tokens==max_tokens: FINISHED + deallocate

Scenario: prompt len 2, max_tokens=3 -> prefill then 2 decodes.

  step 1 (prefill, 2 tok): cached 0 -> 2, len=3, completion=1, status=RUNNING, blocks=[0]
  step 2 (decode, 1 tok): cached 2 -> 3, len=4, completion=2, status=RUNNING, blocks=[0, 1]
  step 3 (decode, 1 tok): cached 3 -> 4, len=5, completion=3, status=FINISHED, blocks=freed (deallocate)

[check] FINISHED at max_tokens=3, blocks freed: True -> OK
