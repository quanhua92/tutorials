Two-Phase Locking (2PL) — interactive 📖 TWO_PHASE_LOCKING.md 📄 two_phase_locking.py

📖 Read the full guide — architecture diagrams, decision tables, killer gotchas, cheat sheet. This page is the interactive companion.
Every transaction lives in two phases: a growing phase (only acquire) and a shrinking phase (only release). The first release is one-way — after it, no more acquires. That single rule is what guarantees serializability. Step through a transaction, watch its lock footprint grow then shrink, and see why strict 2PL holds X-locks to commit. ① The two phases — step through T1 and watch the lock footprint grow then shrink

What to do: click step ▶. Steps 1–2 are the growing phase (only acquires, footprint grows). Step 3 is the first release — T1 flips to shrinking. Step 4 tries to acquire C and is REJECTED (the 2PL rule). Step 5 commits. The footprint must grow then shrink — never the reverse.

This is two_phase_locking.py Section A. At step 4 the lock manager refuses C because T1 already entered the shrinking phase — acquiring after releasing is the one move 2PL forbids.

② Lock compatibility matrix — only co-readers (S-S) mix

What this is: the lock manager's entire granting rule. A request is granted iff it is compatible with every OTHER holder. Click a cell to run the live test (T1 holds, T2 requests) on the right.
live test (T1 holds first, T2 asks)
A writer (X) is mutually exclusive with everyone — including other writers. Two readers (S) coexist. That is the readers-writers problem in one 2×2 grid.

③ Strict vs basic 2PL — does T2 read T1's uncommitted write?

What to do: toggle the variant. Basic 2PL releases X(B) mid-txn → T2 reads dirty data → if T1 aborts, T2 must abort too (cascading abort). Strict 2PL holds X(B) to COMMIT → T2 waits → never sees uncommitted data.
schedule: T1 writes B; T2 reads B
Strict/rigorous 2PL is what production systems use on the write path precisely to kill cascading aborts. (two_phase_locking.py Section B)

④ 2PL vs MVCC throughput — pessimistic (block) vs optimistic (version)

What to do: drag contention. 2PL blocks on any conflict (throughput ~ 100·(1−c)); MVCC only blocks on write-write (throughput ~ 100·(1−c²)). MVCC's curve stays strictly above 2PL's for all 0<c<1.
c = fraction of txns touching the same hot rows
At c=0 both hit 100. At c=0.5: 2PL = 50.0, MVCC = 75.0. That gap is why PostgreSQL/Oracle/InnoDB default to MVCC for reads. 🔗 how MVCC works →

⑤ The 2PL theorem — precedence graph acyclic ⟺ conflict-serializable

What this shows: toggle between a 2PL schedule (graph acyclic → serializable) and a non-2PL schedule (graph has a cycle → NOT serializable). The cycle in the second case comes precisely from a transaction reacquiring a lock after releasing it.
Kung & Papadimitriou (1979): if every txn follows 2PL, the precedence graph is acyclic, hence the schedule is conflict-serializable. (two_phase_locking.py GOLD)
