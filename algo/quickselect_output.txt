========================================================================
  SECTION A: the algorithm -- quickselect, find the median (k=4)
========================================================================
Worked array: [3, 8, 2, 5, 1, 9, 4, 7, 6]   (n = 9)
Goal: find the k=4-th smallest (0-based) = the median.

sorted(WORKED) = [1, 2, 3, 4, 5, 6, 7, 8, 9] -> index 4 = 5 (ground truth; quickselect does NOT sort)

The whole algorithm is quicksort that only walks ONE branch:

    partition arr[lo..hi] around a pivot -> pivot lands at index p.
    if k == p: DONE, arr[p] is the answer.
    if k <  p: recurse LEFT  only  (arr[lo..p-1]).
    if k >  p: recurse RIGHT only  (arr[p+1..hi]).

Quicksort recurses BOTH halves -> O(n log n); quickselect recurses
ONE half -> O(n) average. That single change is the whole algorithm.

Level 0: search region arr[0..8] (size 9), pivot = arr[8] = 6, 8 comparisons
  after partition: [3, 2, 5, 1, 4, 6, 8, 7, 9], pivot at index 5
  k=4 < p=5 -> discard the other half, recurse LEFT.

Level 1: search region arr[0..4] (size 5), pivot = arr[4] = 4, 4 comparisons
  after partition: [3, 2, 1, 4, 5, 6, 8, 7, 9], pivot at index 3
  k=4 > p=3 -> discard the other half, recurse RIGHT.

Level 2: search region arr[4..4] (size 1), pivot = arr[4] = 5, 0 comparisons
  after partition: [3, 2, 1, 4, 5, 6, 8, 7, 9], pivot at index 4
  k=4 == p=4 -> DONE. The k-th smallest = 5.

Result: the 4-th smallest of [3, 8, 2, 5, 1, 9, 4, 7, 6] = 5 (== sorted median).
Total comparisons = 12. (Full sort would take ~
n log2 n = 29; quickselect did 12 by walking
one branch per level.)

========================================================================
  SECTION B: complexity -- O(n) average, O(n^2) worst (same pivot trap as quicksort)
========================================================================
AVERAGE case (good / random pivot splits ~in half):
    Each level does O(n) partition work, then recurses into ONE side
    of size ~n/2, then n/4, ... The work telescopes:
        T(n) = n + n/2 + n/4 + ... = 2n = Theta(n).
    (Quicksort recurses BOTH sides -> n + 2*(n/2) + 4*(n/4) + ... =
    n log n. Walking one branch is what halts the doubling.)

WORST case (pivot is always the smallest/largest):
    Each level peels off ONE element, recursing into size n-1:
        T(n) = n + (n-1) + (n-2) + ... = n(n+1)/2 = Theta(n^2).
    Same trap as quicksort: 'last'/'first' pivot on sorted input.

Empirical scaling -- comparisons vs n (mean over 200 random inputs,
random pivot). Compare to n and to n log2 n (full sort):

       n   quickselect (avg)       / n   sort (n log2 n)    ratio qs/sort
      64               190.9      2.98               384         49.72%
     256               824.7      3.22              2048         40.27%
    1024              3450.3      3.37             10240         33.69%
    4096             14268.4      3.48             49152         29.03%

Read it: quickselect comparisons grow ~ LINEARLY in n (the / n
column is roughly constant ~ a few). A full sort is n log2 n, so
quickselect is ~1/log2(n) of the sort cost -- e.g. ~1/6 at n=64,
~1/12 at n=4096. That is the payoff of walking one branch.

Worst case, concretely -- 'last' pivot on a SORTED array, find the MIN
(k=0). Each partition peels the max and recurses into [0..hi-1], so it
walks the whole shrinking array: n + (n-1) + ... + 1.

  n = 256
  last pivot,  k=0 (min) : 32640 comparisons  (== n(n-1)/2 = 32640, O(n^2))
  random pivot, k=median : 933 comparisons   (linear, dodges the trap)

========================================================================
  SECTION C: median-of-medians (BFPRT 1973) -- the O(n) WORST-CASE guarantee
========================================================================
The O(n^2) worst case is real (Section B). Median-of-medians kills it
by spending extra work to pick a pivot GUARANTEED to be good.

THE RULE (Blum, Floyd, Pratt, Rivest, Tarjan 1973):
  1. Split arr[lo..hi] into groups of 5 (last group may be smaller).
  2. Sort each group (<=5 elements, O(1)) and take its median.
  3. Recursively find the MEDIAN of those medians -> use as pivot.

WHY IT IS LINEAR: half of the ~n/5 group-medians are <= the pivot,
and each of those is >= 2 elements in its own group. So at least
3 elements out of every 10 are guaranteed <= pivot (and 3/10 >= it).
The pivot is in the 30%-70% range, so each partition discards >= 3n/10:
    T(n) <= T(n/5) [find medians] + T(7n/10) [recurse one side] + O(n)
        = Theta(n).     (1/5 + 7/10 = 0.9 < 1 -> the recursion sums linearly.)

THE TRADE: the constant is ~5-10x quickselect's, so in practice people
use INTROSELECT (quickselect + median-of-medians fallback past a depth
limit). That is exactly std::nth_element: quickselect's speed, linear
guarantee. See QUICKSELECT.md for the parallel with introsort.

Comparisons to find the median (n=256), by strategy x input:

  input           last   median3    random   mm (BFPRT)   full sort
  ----------------------------------------------------------------------
  random          1103       597      1023         1405        1738
  sorted         24512       496       933         1490        1024
  reversed       32640       255       846         2454        1024

Read the 'sorted'/'reversed' rows: 'last' pivot blows up to ~n(n-1)/2
(O(n^2)); median-of-medians stays linear (~ a few x n) on EVERY input,
including the hostile ones. That guaranteed linearity is what BFPRT
buys you -- at a constant-factor cost (compare mm vs random on 'random').

========================================================================
  SECTION D: applications -- when to SELECT instead of SORT
========================================================================
USE quickselect when you need ONE order statistic (or a few), not the
whole sorted order:

  - MEDIAN of a dataset: the canonical use. O(n) vs O(n log n) to sort.
  - PERCENTILES / quantiles: P95 latency, box-plot whiskers, the k-th
    largest sale. Select index k = p*n for the p-th percentile.
  - top-k / bottom-k: select the k-th element, then everything on one
    side is your set (O(n + k log k) to finish-sort the top-k).
  - outlier / anomaly thresholds: 'is this value above the 99th pct?'
    select once, then compare.

Where it lives in the wild:
  - C++  std::nth_element  -- introselect (quickselect + mm fallback).
  - numpy.partition / np.median (small arrays) -- introselect-based.
  - quickselect is the subroutine behind 'median-of-medians' pivot
    selection in some quicksort variants.

WHEN NOT to use it:
  - you need the WHOLE array sorted -> just sort (O(n log n)).
  - you need MANY order statistics from the same array -> sort once,
    then every k-th query is O(1). Quickselect per query would be
    O(n) each -> worse than one O(n log n) sort after ~log n queries.
  - the array is tiny -> the constant factors of partition/recursion
    dominate; a small sort is simpler and as fast.
  - you need a WORST-CASE linear guarantee on adversarial input and
    can't afford introselect's fallback machinery -> median-of-medians
